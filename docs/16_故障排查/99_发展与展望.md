# 故障排查发展与展望

> 故障排查正在从"经验驱动"走向"数据驱动 + AI 辅助"，但**底层原理永远是排查的根**。

## 一、可观测性三大支柱演进

```
2010s:  Metrics (Prometheus) + Logs (ELK) + Traces (Jaeger)
         三套独立系统、SDK 各异
              ↓
2020s:  OpenTelemetry 统一采集
         一份 SDK → 任意后端
              ↓
2024+:  增加第四支柱 → Profiles（持续性能分析）
         Parca / Pyroscope / Polar Signals
```

| 支柱 | 工具 | 价值 |
|:---|:---|:---|
| Metrics | Prometheus / Mimir / VM | 量化 |
| Logs | Loki / ClickHouse / ES | 细节 |
| Traces | Tempo / Jaeger | 链路 |
| Profiles | Parca / Pyroscope | 性能根因 |

## 二、eBPF 重塑排查

```
传统:  应用打点 + 系统命令 + sidecar 采集
   ↓
eBPF:  无需改代码 → 内核级钩子 → 全栈观测 → 性能开销 < 1%
```

| 工具 | 用途 |
|:---|:---|
| **Pixie** | 零侵入应用可观测 |
| **Cilium Hubble** | 网络流量可视化 |
| **Parca / Pyroscope** | 持续 CPU / Memory Profiling |
| **Coroot** | eBPF 全栈观测 |
| **bcc-tools / bpftrace** | 命令行 eBPF |
| **Tetragon** | 安全 + 可观测 |

## 三、LLM 辅助排查（颠覆性）

### 3.1 工具

| 工具 | 厂商 | 能力 |
|:---|:---|:---|
| **HolmesGPT** | Robusta | K8s 故障 LLM Agent |
| **k8sgpt** | 社区 | K8s 资源诊断 |
| **Datadog Bits AI** | Datadog | 全栈助手 |
| **Grafana LLM** | Grafana | Grafana 内置助手 |
| **PagerDuty Copilot** | PagerDuty | 告警上下文 |

### 3.2 典型用法

```
传统:  
  SRE 看告警 → kubectl describe → 查日志 → 查指标 → 推断 → 修复
  
LLM 辅助:  
  告警触发 → LLM 自动调用 kubectl/PromQL/LogQL
        → 综合分析 → 输出根因 + 修复建议
        → SRE 一键执行
```

### 3.3 风险与边界

| 风险 | 应对 |
|:---|:---|
| 幻觉（错误结论） | 必须人工验证 |
| 上下文不全 | 提供完整 prompts |
| 越权访问数据 | 严格 RBAC |
| 错误执行命令 | 写操作必须人工确认 |
| 隐私泄露 | 敏感数据脱敏 |

### 6.2 常见故障图谱

```
Pod 异常:
  Pending → 调度问题、资源不足
  CrashLoopBackOff → 应用问题、依赖缺失
  ImagePullBackOff → 镜像/认证
  Init:Error → InitContainer 失败

网络:
  Service 不通 → kube-proxy/CNI/NetworkPolicy
  DNS 不解析 → CoreDNS/NodeLocal DNS
  外部访问失败 → Ingress/Gateway/DNS

存储:
  PVC Pending → StorageClass/Provisioner
  Mount Failed → CSI/权限
  IO 慢 → 后端存储性能

控制平面:
  API 慢 → etcd 性能、负载
  kubelet 不工作 → certificates、节点资源
```

## 十、学习路径

1. **基础**：Linux 工具六字诀
2. **进阶**：eBPF + Brendan Gregg 性能分析
3. **可观测**：OpenTelemetry + Prometheus + Loki + Tempo
4. **K8s 专项**：常见故障图谱
5. **中间件**：每个一个真实案例
6. **AI Infra**：GPU/NCCL/vLLM 排查
7. **方法论**：USE/RED + 复盘文化 + Runbook

> 💡 故障排查的最高境界：**还没发生你就预防了**。退而求其次：**发生后 5 分钟内定位**。再退一步：**故障复盘后写出 Runbook，让下一次更快**。
