# 09_中间件 · 概述

> 中间件是业务和基础设施之间的"胶水层"，是绝大多数线上故障的发生地。

## 一、中间件分类全景

```
┌──────────── 数据存储 ────────────┐
│ 关系数据库:  MySQL, PostgreSQL    │
│ 缓存:        Redis, Memcached     │
│ 文档:        MongoDB, Couchbase   │
│ KV/列族:     Cassandra, HBase     │
│ 搜索:        Elasticsearch, OpenS │
│ 时序:        InfluxDB, Prometheus │
│ 列式分析:    ClickHouse, Doris    │
└─────────────────────────────────┘

┌──────────── 消息中间件 ──────────┐
│ Kafka, RabbitMQ, Pulsar, RocketMQ│
│ NATS, Redis Streams              │
└─────────────────────────────────┘

┌──────────── 微服务支撑 ─────────┐
│ 注册中心:  Nacos, Consul, etcd   │
│ 配置中心:  Apollo, Nacos          │
│ 网关:      Nginx, APISIX, Kong   │
│ 服务网格:  Istio, Linkerd, Cilium │
└─────────────────────────────────┘

┌──────────── 任务调度 ───────────┐
│ XXL-Job, Airflow, Argo Workflow │
└─────────────────────────────────┘
```

## 二、中间件选型方法论

| 维度 | 关键问题 |
|:---|:---|
| **业务模型** | 读多写少？写多？强一致还是最终？ |
| **数据规模** | GB / TB / PB？日均增长？ |
| **延迟要求** | < 1ms / 10ms / 100ms？ |
| **运维成本** | 团队能力？有原厂支持？开源活跃？ |
| **生态适配** | 现有技术栈、云厂商支持 |
| **总成本** | 硬件 + License + 人力 + 故障成本 |

> 详见 `00_选型方法论.md`

## 四、统一运维思路

所有中间件运维都有共同框架：

```
┌─────────────────────────────────┐
│  部署      高可用集群 + 监控告警  │
│  配置      性能参数 + 安全加固    │
│  备份      定时全备 + 持续增量    │
│  监控      关键指标 + 慢日志      │
│  容量      预估 + 扩容预案        │
│  应急      故障演练 + Runbook    │
│  升级      Canary + 回滚预案     │
└─────────────────────────────────┘
```

## 五、学习路径

1. **基础**：MySQL + Redis（业务必用）
2. **进阶**：Kafka + ES（数据流必备）
3. **专家**：选 1-2 个深度（如 PG + CH）
4. **统筹**：选型方法论、监控告警体系

> 📖 中间件没有"最好"，只有"最适合"。
