# 中间件发展与展望

> 中间件领域正在两极化：通用产品继续深耕、垂直产品（AI/向量/流计算）爆发。

## 一、数据库领域演进

### 1.1 HTAP 融合

```
传统:  OLTP (MySQL)   +   OLAP (Hive/CH)
              ↑ ETL 同步（小时级延迟）
HTAP:  一份数据，同时支持 OLTP + OLAP
              ↑ 实时（秒级）
```

| 产品 | 厂商 | 模式 |
|:---|:---|:---|
| **TiDB** | PingCAP | TiKV (行) + TiFlash (列) |
| **OceanBase** | 蚂蚁 | 行存 + 列存合一 |
| **GaussDB** | 华为 | 行列混合 |
| **PolarDB** | 阿里 | 共享存储 |

### 1.2 向量数据库

为 RAG / 语义搜索 / 推荐而生：

| 产品 | 特点 |
|:---|:---|
| **Milvus** | CNCF、生产级 |
| **Qdrant** | Rust 写、性能强 |
| **Weaviate** | 内置 ML |
| **PostgreSQL + pgvector** | 老牌 DB 加扩展 |
| **Elasticsearch 8+** | 集成 KNN |

**趋势**：通用数据库纷纷集成向量能力（PG/MySQL/Redis 都有），独立向量库专精方向（极致性能）。

### 1.3 Serverless 数据库

| 产品 | 厂商 |
|:---|:---|
| Aurora Serverless | AWS |
| PlanetScale | MySQL |
| Neon | PostgreSQL |
| TiDB Cloud Serverless | PingCAP |
| Supabase | 开源全栈 |

**特征**：按使用付费、自动 scale-to-zero、分钟级冷启动。

## 二、流处理与消息

### 2.1 Kafka 演进

| 版本 | 重大变化 |
|:---|:---|
| 2.8 | KRaft（去除 ZooKeeper） |
| 3.0+ | KRaft GA |
| 3.7+ | 分层存储（Tiered Storage） |
| 4.0 | 完全移除 ZooKeeper |

**分层存储**：热数据本地 SSD，冷数据自动归档 S3，存储成本下降 80%。

### 2.2 Kafka 替代品

| 产品 | 卖点 |
|:---|:---|
| **Apache Pulsar** | 计算存储分离、多租户 |
| **Redpanda** | C++ 重写、单二进制、零依赖 |
| **WarpStream** | S3 原生、Kafka 协议兼容 |
| **NATS JetStream** | 轻量、内置 KV/Object |

### 2.3 Flink 与流计算

实时数据处理的事实标准：

| 趋势 | 状态 |
|:---|:---|
| Flink SQL | 主流开发方式 |
| 流批一体 | 一份 SQL 跑两种 |
| Flink CDC | 入湖入仓主力 |
| Paimon | 流式数据湖（Flink 主推） |

## 七、未来 3-5 年展望

| 方向 | 预测 |
|:---|:---|

## 八、对运维的影响

| 变化 | 运维新要求 |
|:---|:---|
| HTAP | 同时懂 OLTP + OLAP 调优 |
| 向量库 | 索引参数（HNSW/IVF）调优 |
| Kafka KRaft | 抛弃 ZooKeeper 运维知识 |
| eBPF Mesh | Cilium 调试 |
| AI 中间件 | LLM 网关流量管理 |
| 国产 DB | TiDB/OceanBase 运维专业知识 |

## 九、学习路径

1. **基础**：MySQL + Redis + Kafka 三大件精通
2. **进阶**：PostgreSQL + ES + ClickHouse 数据三剑客
3. **专项**：选一个深度（TiDB 或 ClickHouse 或 Flink）
4. **AI 时代**：向量库 + 模型网关

> 💡 中间件知识是运维的"硬功夫"，它决定了你能 hold 住多复杂的系统。
