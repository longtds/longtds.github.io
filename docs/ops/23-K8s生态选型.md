# Kubernetes 生态选型

> K8s 生态那么多组件, 到底选哪个? Ingress 用 Nginx 还是 Traefik? CNI 用 Calico 还是 Cilium? Service Mesh 上不上 Istio? 本文覆盖 K8s 生态各层选型对比 + 选型决策矩阵。

### 1.2 选型基本原则

```
选型 5 大原则:

  1. 团队能力: 选运维得起的, 不选最新最炫的
  2. 场景匹配: 不要过度设计, 小规模不上 Istio
  3. 社区活跃: CNCF Graduated > Incubating > Sandbox
  4. 云厂商锁定: 混合云优先选云中立
  5. 长期成本: License + 硬件 + 运维人力

避免选型陷阱:
  ❌ 追新: 生产上 CNCF Sandbox 项目
  ❌ 大而全: 一个 Istio 解决所有问题 (可能 90% 特性用不上)
  ❌ 过度隔离: 每个服务一个独立 Namespace + NetworkPolicy
  ❌ 云原生洁癖: 明明 NFS 够用非要上 Ceph
  ❌ 单点决策: 没和研发/安全/网络 沟通
```

## 二、CRI 容器运行时

### 2.1 主流方案

| 运行时 | 状态 | K8s 兼容 | 性能 | 特点 | 推荐 |
|:---|:---|:---|:---|:---|:---|
| **containerd** | CNCF Graduated | 1.24+ 默认 | ⭐ 高 | 轻量, Docker 底层, 生产首选 | ⭐⭐⭐ |
| **CRI-O** | CNCF Incubating | 良好 | ⭐ 高 | OCI 原生, RedHat 主推, OpenShift 默认 | ⭐⭐ |
| **Docker Engine** | - | ❌ 1.24+ 移除 | 中 | 已废弃 (需 cri-dockerd 适配) | ❌ |
| **Kata Containers** | OpenInfra | 良好 | 中 | 轻量 VM 隔离, 强安全 | 特殊场景 |
| **gVisor** | Google | 良好 | 低 | 用户态内核, 强隔离 | 多租户 |

### 2.2 选型建议

```
containerd (推荐 90% 场景):
  ✅ K8s 1.24+ 默认, 社区主流
  ✅ 性能好, 资源占用低
  ✅ 生态完善, Docker 命令兼容 (nerdctl)
  ✅ 阿里云 ACK、腾讯云 TKE、AWS EKS 默认

CRI-O (RedHat 生态):
  ✅ 更轻量 (专为 K8s 设计, 无多余功能)
  ✅ OpenShift、Rocky/RHEL 默认
  ❌ 生态相对小, 社区活跃度低

Kata (强隔离场景):
  ✅ 多租户 (SaaS 平台/公共云)
  ✅ 运行不可信代码 (CI/CD 沙盒)
  ❌ 启动慢 (2-5s vs 100ms)
  ❌ 内存占用高 (每 Pod 多几十 MB)

决策:
  通用 → containerd
  RHEL 系 → CRI-O
  多租户/沙盒 → Kata + containerd
```

## 三、CNI 网络插件

### 3.1 主流方案对比

| CNI | 数据面 | 性能 | 网络策略 | eBPF | 适用规模 | 推荐 |
|:---|:---|:---|:---|:---|:---|:---|
| **Calico** | BGP/VXLAN/eBPF | ⭐ 高 | ⭐ 强 | 可选 | 中大规模 | ⭐⭐⭐ |
| **Cilium** | eBPF | ⭐⭐ 极高 | ⭐⭐ 强 (L7) | ⭐ 原生 | 大规模 | ⭐⭐⭐ |
| **Flannel** | VXLAN/host-gw | 中 | ❌ | ❌ | 小规模 | ⭐ |
| **Weave** | 加密网络 | 中 | 弱 | ❌ | 小规模 | 已停维 |
| **Antrea** | OVS | 高 | 强 | ❌ | 中大规模 | ⭐⭐ |
| **kube-router** | BGP/IPVS | 高 | 强 | ❌ | 中规模 | ⭐ |
| **Multus** | 多网络 | - | - | - | 特殊场景 | 电信 NFV |

### 3.2 详细特性对比

```
Calico:
  数据面: BGP (推荐) / VXLAN / IPIP / eBPF (新版)
  优势:
    - BGP 模式无封装开销, 性能接近裸机
    - NetworkPolicy 完善 (社区标准实现)
    - 支持 IPv4/IPv6/双栈
    - 稳定, 生产验证多
  劣势:
    - 大规模 BGP 需网络配合 (交换机学 BGP)
    - Overlay 模式性能有损耗

Cilium:
  数据面: eBPF (无 iptables, 内核态转发)
  优势:
    - eBPF 高性能 (10Gbps 线速)
    - L7 网络策略 (HTTP/gRPC/Kafka)
    - 内置服务网格能力 (Cilium Service Mesh)
    - Hubble 可观测性 (流量拓扑/指标)
    - 替代 kube-proxy (更快 Service 转发)
  劣势:
    - 内核要求 ≥ 5.4 (功能全需 5.10+)
    - 学习曲线陡, 调试难 (需懂 eBPF)
    - 资源占用较高

Flannel:
  数据面: VXLAN (推荐) / host-gw / UDP
  优势:
    - 极简, 5 分钟部署
    - 稳定, 少 bug
  劣势:
    - 不支持 NetworkPolicy (需搭配 Calico)
    - 功能少, 无 BGP/eBPF

Antrea:
  数据面: OVS (Open vSwitch)
  优势:
    - VMware 主推, VMware 生态用户友好
    - 支持 Windows Node (K8s 混合集群)
    - NetworkPolicy 完善
  劣势:
    - 生态较小
```

### 3.3 选型决策矩阵

```
场景 → 推荐 CNI:

  ┌─────────────────────────────┬────────────────┐
  │ 场景                        │ 推荐           │
  ├─────────────────────────────┼────────────────┤
  │ 学习/测试 (< 20 节点)       │ Flannel        │
  │ 中小生产 (20-100 节点)      │ Calico VXLAN   │
  │ 大规模 (100+ 节点, 有网络)  │ Calico BGP     │
  │ 需要 L7 策略                │ Cilium         │
  │ 高性能计算 (10G/25G/100G)   │ Cilium eBPF    │
  │ Service Mesh 平替           │ Cilium         │
  │ 混合 Windows/Linux 节点     │ Antrea/Calico  │
  │ 电信 NFV/多网卡             │ Multus         │
  │ 内核 ≥ 5.10, 追新           │ Cilium         │
  │ 内核 < 5.4                  │ Calico         │
  │ 多集群互通                  │ Calico/Cilium  │
  │ IPv6/双栈                    │ Calico/Cilium │
  └─────────────────────────────┴────────────────┘

保守选择: Calico
先进选择: Cilium
最简单:  Flannel + Calico NetworkPolicy
```

## 四、CSI 存储

### 4.1 主流方案对比

| CSI | 类型 | 部署位置 | 性能 | 复杂度 | 适用规模 | 推荐 |
|:---|:---|:---|:---|:---|:---|:---|
| **Rook-Ceph** | 分布式块/文件/对象 | 集群内 | 中 | ⭐⭐⭐ 高 | 大规模 | ⭐⭐⭐ |
| **Ceph-CSI** | 分布式块/文件 | 外部 Ceph | ⭐ 高 | ⭐⭐ 中 | 大规模 | ⭐⭐⭐ |
| **Longhorn** | 分布式块 | 集群内 | 中 | ⭐ 低 | 中小规模 | ⭐⭐⭐ |
| **OpenEBS** | 多引擎 (Jiva/cStor/Mayastor) | 集群内 | 中-高 | ⭐⭐ 中 | 中规模 | ⭐⭐ |
| **NFS-CSI** | 文件共享 | 外部 NFS | 低 | ⭐ 低 | 小规模 | ⭐ |
| **local-path** | 本地磁盘 | Node | ⭐⭐ 极高 | ⭐ 极低 | 测试/边缘 | ⭐⭐ |
| **云盘 CSI** | EBS/OSS/COS | 云 | ⭐ 高 | ⭐ 低 | 公有云 | ⭐⭐⭐ |

### 4.2 选型建议

```
需求分析:

  1. RWO (单节点读写, ReadWriteOnce):
     数据库、Elasticsearch、Kafka 等有状态服务
     推荐: 块存储 (Ceph RBD / Longhorn / 云盘)

  2. RWX (多节点读写, ReadWriteMany):
     共享数据 (媒体/日志/共享配置)
     推荐: 文件系统 (CephFS / NFS)

  3. RWOP (单 Pod 只写, K8s 1.29+):
     强隔离场景
     推荐: 块存储

场景 → 推荐 CSI:

  ┌─────────────────────────────┬────────────────────┐
  │ 场景                        │ 推荐               │
  ├─────────────────────────────┼────────────────────┤
  │ 公有云 (AWS/阿里云/腾讯云)  │ 云盘 CSI (EBS/OSS) │
  │ 私有云, 大规模 (500+ TB)    │ 外部 Ceph + CSI    │
  │ 私有云, 中规模 (10-500 TB)  │ Rook-Ceph          │
  │ 私有云, 小规模 (< 10 TB)    │ Longhorn           │
  │ 简单需求 (共享文件)         │ NFS-CSI            │
  │ 边缘/测试                    │ local-path         │
  │ 高性能 (NVMe/RDMA)          │ Mayastor           │
  │ HCI 超融合                   │ Rook-Ceph          │
  └─────────────────────────────┴────────────────────┘
```

## 五、Ingress 与 Gateway API

### 5.1 Ingress Controller 对比

| Controller | 底层 | 性能 | 特性 | 学习曲线 | 推荐 |
|:---|:---|:---|:---|:---|:---|
| **ingress-nginx** | Nginx | ⭐ 高 | 全面, annotation 多 | 低 | ⭐⭐⭐ |
| **NGINX Plus IC** | Nginx Plus | ⭐ 高 | 商业, CRD 完善 | 中 | 商业 |
| **Traefik** | Traefik | 中 | 简洁, 自动发现 | 低 | ⭐⭐ |
| **HAProxy** | HAProxy | ⭐ 高 | 稳定, TCP 强 | 中 | ⭐⭐ |
| **Envoy** | Envoy | ⭐ 高 | xDS, 云原生 | ⭐⭐ 高 | ⭐⭐ |
| **APISIX Ingress** | APISIX | ⭐ 高 | 插件丰富, 国产 | 中 | ⭐⭐ |
| **Kong** | Kong | 高 | API 网关 + Ingress | 中 | API 网关 |
| **Cilium Gateway** | Cilium | ⭐⭐ 极高 | eBPF, L7 | 高 | Cilium 用户 |

### 5.2 Gateway API vs Ingress

```
Ingress (老):
  - 单一 HTTPRoute 模型
  - 依赖 annotation 扩展 (每个 Controller 语法不同)
  - HTTP 优先, TCP/UDP 弱
  - 简单, 但生态碎片化

Gateway API (新, K8s 1.29+ GA):
  - 分层模型: GatewayClass → Gateway → Route
  - 多协议: HTTPRoute / TCPRoute / UDPRoute / GRPCRoute / TLSRoute
  - 角色分离: 基础设施 (Gateway) vs 应用 (Route)
  - 标准化, 跨 Controller 迁移容易
  - 命名空间隔离友好

选型建议:
  新集群: Gateway API (面向未来)
  存量集群: Ingress (兼容性好)
  Envoy/Cilium/Istio 用户: 优先 Gateway API
```

### 5.3 场景推荐

```
场景 → 推荐:

  ┌─────────────────────────────┬────────────────────┐
  │ 场景                        │ 推荐               │
  ├─────────────────────────────┼────────────────────┤
  │ 简单 Web/API 暴露           │ ingress-nginx      │
  │ 需要 API 网关 (限流/认证)   │ APISIX / Kong      │
  │ 微服务多路由/自动化         │ Traefik            │
  │ 高性能 TCP/UDP              │ HAProxy / Envoy    │
  │ Service Mesh 场景           │ Istio Gateway      │
  │ Cilium 用户                 │ Cilium Gateway API │
  │ 混合协议 (gRPC/WebSocket)  │ Envoy Gateway      │
  │ 国产化                       │ APISIX             │
  └─────────────────────────────┴────────────────────┘
```

## 六、Service Mesh

### 6.1 主流方案

| Mesh | 数据面 | 控制面 | 性能损耗 | 特性 | 复杂度 | 推荐 |
|:---|:---|:---|:---|:---|:---|:---|
| **Istio** | Envoy sidecar | Istiod | 5-15% | ⭐⭐ 完整 | ⭐⭐⭐ 高 | ⭐⭐ |
| **Istio Ambient** | ztunnel/waypoint | Istiod | 3-8% | ⭐⭐ 完整 | ⭐⭐ 中 | ⭐⭐⭐ |
| **Linkerd** | Rust proxy | 简单 | 2-5% | 中 | ⭐ 低 | ⭐⭐⭐ |
| **Cilium SM** | eBPF | Cilium | 1-3% | 中 | ⭐⭐ 中 | ⭐⭐⭐ |
| **Consul Connect** | Envoy | Consul | 5-15% | 强 (多集群) | 中 | ⭐⭐ |
| **AWS App Mesh** | Envoy | AWS | 5-10% | 中 | 低 (托管) | AWS 用户 |
| **Kuma** | Envoy | Kuma CP | 5-10% | 中 | 中 | ⭐ |

### 6.2 上不上 Service Mesh?

```
上 Mesh 判断:

  必要:
    - 服务数量 > 30 个
    - 跨集群/跨云
    - 严格零信任 (mTLS 全站)
    - 复杂流量策略 (金丝雀/A-B 测试)
    - 详细服务级可观测

  不必要:
    - 服务 < 20 个
    - 单集群
    - 团队没经验
    - 只需要简单负载均衡

  中间态 (轻量方案):
    - Cilium Service Mesh (借助 CNI)
    - Linkerd (轻量)
```

### 6.3 选型建议

```
场景 → 推荐:

  ┌─────────────────────────────┬────────────────────┐
  │ 场景                        │ 推荐               │
  ├─────────────────────────────┼────────────────────┤
  │ 学习/试点 (小规模)          │ Linkerd            │
  │ 大厂/大规模 (需完整能力)    │ Istio Ambient      │
  │ CNI 已选 Cilium             │ Cilium SM          │
  │ 多云/混合云                  │ Istio / Consul     │
  │ AWS 全家桶                   │ AWS App Mesh       │
  │ 极致性能 (延迟敏感)         │ Cilium SM          │
  │ 简单 mTLS + 观测            │ Linkerd            │
  │ 服务数量 < 20                │ 不用 Mesh          │
  └─────────────────────────────┴────────────────────┘
```

## 七、可观测性

### 7.1 三大支柱选型

```
Metrics (指标):
  Prometheus:       ⭐⭐⭐ 事实标准, PromQL 丰富
  Prometheus HA:    Thanos / Cortex / Mimir / VictoriaMetrics
  云托管:           AMP (AWS) / Managed Prometheus (GCP)

  长期存储选型:
    Thanos:        对象存储, 全球查询, 复杂度中
    Cortex:        水平扩展, 复杂
    Mimir:         Cortex 后继, Grafana 主推
    VictoriaMetrics: 高性能, 单二进制, 推荐 ⭐

Logs (日志):
  Loki:            ⭐⭐⭐ 类 Prometheus, 只索引 label
  ELK/EFK:         Elasticsearch + Fluentd/Filebeat + Kibana, 全文检索强
  ClickHouse:      SQL, 超大规模低成本
  OpenSearch:      ES 分叉, AWS 主推

Traces (追踪):
  Jaeger:          ⭐⭐ CNCF, 兼容 OpenTracing
  Tempo:           ⭐⭐⭐ Grafana Labs, 对象存储便宜
  Zipkin:          老牌, 简单
  Skywalking:      ⭐⭐ 国产, APM + 追踪一体
  Datadog/NewRelic: 商业, 全家桶

Profiling (新):
  Pyroscope:       ⭐⭐ Grafana Labs, 持续 profiling
  Parca:           CNCF, eBPF 持续 profiling
```

### 7.2 组合选型

```
可观测栈组合推荐:

  1. Grafana 全家桶 (推荐):
     Metrics: Prometheus + Mimir
     Logs:    Loki
     Traces:  Tempo
     Profile: Pyroscope
     UI:      Grafana
     采集:    Alloy (原 Grafana Agent) / OpenTelemetry Collector

  2. Elastic 全家桶:
     Metrics: Metricbeat
     Logs:    Filebeat + Elasticsearch + Kibana
     Traces:  APM Server
     统一 UI: Kibana
     优势:    全文检索强, 商业支持
     劣势:    资源占用大

  3. 混合 (务实):
     Metrics: Prometheus + VictoriaMetrics
     Logs:    Loki (日志量小) 或 ClickHouse (日志量大)
     Traces:  Jaeger 或 Tempo
     UI:      Grafana

  4. 商业:
     Datadog / New Relic / Dynatrace / 阿里云 ARMS
     省心, 但贵 (按 Host/GB 计费)

场景 → 推荐:

  ┌─────────────────────────────┬────────────────────┐
  │ 场景                        │ 推荐               │
  ├─────────────────────────────┼────────────────────┤
  │ 中小规模 (< 100 节点)       │ Prom + Loki + Tempo│
  │ 大规模 (100+ 节点)          │ VictoriaMetrics + Loki │
  │ 全文检索强需求               │ ELK/EFK            │
  │ 日志量极大 (TB/day)         │ ClickHouse         │
  │ 国产化                       │ Skywalking + Loki  │
  │ 云托管                       │ AWS AMP + CloudWatch│
  │ 有钱, 要省心                 │ Datadog            │
  └─────────────────────────────┴────────────────────┘
```

## 八、GitOps 与 CD

### 8.1 主流方案

| 工具 | CNCF | 复杂度 | 特点 | 推荐 |
|:---|:---|:---|:---|:---|
| **Argo CD** | Graduated | 中 | UI 强, 多集群, 主流 | ⭐⭐⭐ |
| **Flux CD** | Graduated | 低 | 轻量, 无 UI, K8s 原生 | ⭐⭐⭐ |
| **Rancher Fleet** | - | 低 | Rancher 用户 | ⭐⭐ |
| **Jenkins X** | - | 高 | 全流水线 | ⭐ |
| **Spinnaker** | - | ⭐⭐ 高 | Netflix 出品, 重 | ⭐ |

### 8.2 选型建议

```
Argo CD (推荐):
  ✅ UI 直观, 团队协作友好
  ✅ 多集群支持强
  ✅ ApplicationSet 批量管理
  ✅ 社区活跃, 生态好
  ❌ 组件多 (5-6 个 Pod)

Flux CD (轻量选择):
  ✅ 极简, GitOps 原教旨
  ✅ 资源占用少
  ✅ Kustomize/Helm 原生
  ❌ 无 UI (可搭配 Weave GitOps UI)
  ❌ 多集群略麻烦

Rancher Fleet:
  ✅ Rancher 用户开箱即用
  ✅ 大规模多集群 (千级)
  ❌ 生态限于 Rancher

场景 → 推荐:

  单集群/团队大    → Argo CD
  多集群/CI 强大   → Flux CD
  Rancher 用户     → Fleet
  必须 UI          → Argo CD
```

### 8.3 部署工具

```
Kustomize (推荐):
  K8s 原生, kubectl 内置
  YAML 覆盖模式, 无模板
  适合环境差异 (dev/staging/prod)

Helm (主流):
  模板 + Values
  Chart 生态最丰富
  版本管理, rollback
  ⚠️ 复杂 Values 难维护

Helmfile / Kluctl (进阶):
  Helm 之上的编排
  多 Chart 依赖管理

Operator (SDK / Kubebuilder):
  声明式 CRD
  自定义业务逻辑
  重, 学习曲线陡

选型:
  简单应用      → Kustomize
  开源组件安装  → Helm
  多组件编排    → Kustomize + Helmfile
  复杂状态管理  → Operator
```

## 九、安全组件

### 9.1 主流方案

```
证书管理:
  cert-manager: ⭐⭐⭐ 事实标准
  ACME (Let's Encrypt) / 自建 CA / Vault

策略引擎:
  OPA/Gatekeeper: ⭐⭐ Rego 语言
  Kyverno:        ⭐⭐⭐ YAML 语法, 简单 (推荐)

准入控制:
  Kyverno / OPA Gatekeeper / native ValidatingAdmissionPolicy (K8s 1.30+)

运行时安全:
  Falco:          ⭐⭐⭐ eBPF 系统调用监控
  Tracee:         Aqua 出品, eBPF
  Tetragon:       Cilium 出品, eBPF (更快)

镜像扫描:
  Trivy:          ⭐⭐⭐ Aqua 出品, 简单强大
  Grype:          Anchore, 支持 SBOM
  Clair:          RedHat, Quay 集成
  Snyk / Sysdig:  商业

密钥管理:
  Sealed Secrets: 加密提交到 Git
  External Secrets: 从 Vault/AWS SM/Azure KV 同步
  SOPS + KMS:     GitOps 友好

零信任:
  SPIFFE/SPIRE:   身份联邦
  Istio mTLS:     服务间认证
  Cilium mTLS:    eBPF 加密

镜像签名:
  Cosign:         ⭐⭐⭐ Sigstore 项目
  Notation:       CNCF, OCI 原生
```

### 9.2 推荐组合

```
最小安全栈:
  - Kyverno (策略)
  - cert-manager (TLS)
  - Trivy (镜像扫描)
  - Falco (运行时)

完整安全栈:
  - + External Secrets (密钥)
  - + Cosign (镜像签名)
  - + Tetragon (eBPF 检测)
  - + OPA (复杂策略)

金融/政企:
  - + SPIRE (零信任身份)
  - + Vault (密钥管理)
  - + 商业 (Aqua/Sysdig)
```

## 十、监控/日志/追踪采集

### 10.1 采集器对比

| 采集器 | 场景 | 优势 | 劣势 |
|:---|:---|:---|:---|
| **Fluentd** | 日志 | 插件多, 老牌 | Ruby, 慢, 内存高 |
| **Fluent Bit** | 日志 | ⭐ C 编写, 轻量 | 插件少于 Fluentd |
| **Vector** | 日志/指标 | Rust, 超快 | 生态较新 |
| **Filebeat** | 日志 | Elastic 生态 | 只对接 Elastic |
| **Promtail** | 日志 | Loki 专用, label 友好 | 只对接 Loki |
| **OTel Collector** | ⭐⭐⭐ 统一 | 三大支柱统一, 供应商中立 | 配置略复杂 |
| **Alloy** | ⭐⭐⭐ 统一 | Grafana Labs, OTel 兼容 | Grafana 生态 |

### 10.2 推荐架构

```
现代可观测采集架构:

  应用 → OpenTelemetry SDK → OTel Collector → 后端 (Prom/Loki/Tempo)
       ↑                    ↓
    埋点         (agent 模式部署到每个节点)
                            ↓
                     (Gateway 模式集中聚合)

  优势:
    - 统一采集 (指标+日志+追踪)
    - 供应商中立 (可切换后端)
    - 语义规范化 (OTel Semantic Conventions)

  过渡方案:
    Prometheus → Prom scrape / OTel Prom Receiver
    日志       → Fluent Bit / Alloy / OTel
    追踪       → OpenTelemetry SDK
```

  Backstage Software Catalog: 服务清单/所有权
  ServiceNow CMDB: 企业 (重)

流水线:
  Tekton:         ⭐⭐ CNCF, K8s 原生 CI/CD
  Argo Workflows: CNCF, DAG 工作流
  Jenkins:        传统, K8s Agent
  GitHub Actions: SaaS, 简单
  GitLab CI:      SaaS/自建, 一体化

### 11.2 平台工程建议

```
从小到大演进:

  阶段 1: 会用 K8s (< 5 服务)
    K8s + Helm + Prometheus + kubectl 手动 → 够用

  阶段 2: 团队协作 (5-20 服务)
    + Argo CD (GitOps)
    + Ingress + cert-manager
    + Grafana + Alertmanager

  阶段 3: 平台化 (20-100 服务)
    + Backstage (服务目录)
    + Tekton/GitHub Actions (标准流水线)
    + Kyverno (策略)
    + Trivy (安全扫描)

  阶段 4: 内部云 (100+ 服务/多团队)
    + Crossplane (自服务基础设施)
    + Service Mesh
    + 多集群 (Karmada/Rancher/Anthos)
    + 成本管理 (KubeCost/OpenCost)
```

## 十二、综合选型矩阵

### 12.1 按规模推荐

```
小规模 (< 20 节点, 单集群):
  CRI:      containerd
  CNI:      Flannel + Calico NP (或纯 Calico)
  CSI:      Longhorn / NFS-CSI / local-path
  Ingress:  ingress-nginx
  Mesh:     不用
  监控:     Prometheus + Loki (kube-prometheus-stack)
  CD:       Argo CD
  安全:     cert-manager + Kyverno + Trivy

中等规模 (20-100 节点, 单/多集群):
  CRI:      containerd
  CNI:      Calico VXLAN
  CSI:      Rook-Ceph / 云盘
  Ingress:  ingress-nginx / APISIX
  Mesh:     Linkerd (可选)
  监控:     Prometheus + Loki + Tempo (Grafana 栈)
  CD:       Argo CD
  安全:     cert-manager + Kyverno + Trivy + Falco

大规模 (100+ 节点, 多集群/多云):
  CRI:      containerd
  CNI:      Cilium eBPF / Calico BGP
  CSI:      外部 Ceph + Ceph-CSI / 云盘
  Ingress:  Gateway API (Envoy/Cilium)
  Mesh:     Istio Ambient / Cilium SM
  监控:     VictoriaMetrics + Loki + Tempo
  CD:       Argo CD (ApplicationSet)
  安全:     Kyverno + External Secrets + Cosign + Tetragon
  平台:     Backstage + Crossplane
  多集群:   Karmada / Fleet / Rancher
```

### 12.2 按团队能力推荐

```
初级 (刚入门 K8s):
  containerd + Flannel + Longhorn + ingress-nginx +
  kube-prometheus-stack + Helm + Argo CD

  哲学: 选文档最全的、社区最大的

中级 (会调优):
  containerd + Calico + Rook-Ceph + APISIX +
  Grafana 全家桶 + Argo CD + Kyverno + Trivy

  哲学: 平衡功能和维护成本

高级 (深度定制):
  containerd + Cilium + 外部 Ceph + Cilium Gateway +
  Istio Ambient + VictoriaMetrics 栈 +
  Argo CD + Crossplane + Backstage

  哲学: 追求性能和平台化
```

### 12.3 云上 vs 自建

```
公有云 (托管 K8s):

  AWS EKS:  优选托管服务
    - CNI: AWS VPC CNI (原生) 或 Cilium
    - CSI: EBS / EFS / S3
    - Ingress: AWS Load Balancer Controller / ALB
    - 日志: CloudWatch Logs
    - 监控: AMP (Managed Prometheus) / CloudWatch

  阿里云 ACK:
    - CNI: Terway / Flannel
    - CSI: 云盘 / NAS / OSS
    - Ingress: ALB Ingress Controller / MSE
    - 监控: SLS / ARMS

  腾讯云 TKE:
    - CNI: GlobalRouter / VPC-CNI
    - CSI: CBS / CFS / COS

自建 (私有云/裸金属):
  - 所有组件自己选/维护
  - CNI: Calico/Cilium
  - CSI: Ceph
  - Ingress: ingress-nginx / APISIX
  - 监控: 全套自己搭

原则:
  能用云托管就别自建 (省人力)
  除非: 合规要求、成本敏感、混合部署需求
```

## 十三、避坑指南

### 13.1 常见坑

```
坑 1: 一次选完整套云原生
  症状: 团队 5 人, 上了 Istio + Ceph + Cilium + Prometheus HA + Argo CD
  结果: 运维不过来, 组件之间互相踩坑
  正确: 循序渐进, 每次引入 1 个新组件

坑 2: 追新 CNCF Sandbox 项目
  症状: 生产用 Sandbox 项目, 遇 bug 无人修
  结果: 定制补丁, 陷入维护地狱
  正确: 生产用 Graduated / Incubating, Sandbox 只测试

坑 3: Service Mesh 早上
  症状: 20 个服务上 Istio, 学习成本 3 个月
  结果: 团队学不会, 反而降低稳定性
  正确: 服务 > 30 才考虑, 优先 Linkerd/Cilium SM

坑 4: 忽略网络策略默认放通
  症状: 装完 K8s Pod 间随意通信
  结果: 一个 Pod 被攻破, 横向感染整个集群
  正确: 默认 default-deny, 显式放通

坑 5: 存储没备份
  症状: 上了 Ceph 觉得就是 "存储", 没备份
  结果: 一次意外 (磁盘损坏/误删) 数据全丢
  正确: Velero/Kasten K10 定期备份 PVC

坑 6: 监控 alert 疲劳
  症状: 装 kube-prometheus-stack 默认 100+ 告警规则
  结果: 每天几百条告警, 逐渐屏蔽全部
  正确: 只留 critical, 建立告警审查机制

坑 7: Ingress + Service Mesh 冲突
  症状: 同时用 ingress-nginx + Istio Sidecar
  结果: mTLS 冲突/双重代理/延迟高
  正确: Ingress 换成 Istio Gateway 或独立处理

坑 8: 大规模 iptables 性能
  症状: kube-proxy iptables 模式, Service 数量 > 1000
  结果: 规则匹配延迟高
  正确: 切 IPVS 或用 Cilium 替代 kube-proxy
```

### 13.2 决策清单

```
每次引入新组件, 问自己:

  □ 我们真的需要吗? (不是跟风)
  □ 团队有能力维护吗?
  □ CNCF 成熟度? (Graduated > Incubating > Sandbox)
  □ 社区活跃度? (GitHub stars/commits/issues 响应)
  □ 有备选方案吗? (锁定风险)
  □ 与现有组件冲突吗?
  □ 升级路径清晰吗? (2 年后怎么办?)
  □ 生产故障能自查吗? (可观测/文档)
  □ 有没有商业支持? (关键场景)
  □ 长期 TCO 算过吗? (License + 硬件 + 人力)
```
