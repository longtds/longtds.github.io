### 2.2 CPU 选型要点

```
CPU 选型决策矩阵:

  Web / API / 通用计算:
    - 中主频 + 中核心 (32-64C)
    - Intel Xeon Gold 或 EPYC 9354
    - 关注: 单核性能 + IPC

  数据库 (MySQL/PG):
    - 高主频 + 大 L3 缓存
    - Intel Xeon Gold 8480+ 或 EPYC 9384X (3D V-Cache)
    - 关注: 单核性能 + 内存带宽

  虚拟化 (KVM/VMware/OpenStack):
    - 多核 + 大内存支持
    - EPYC 9654 (96C) 或 Xeon 8592+
    - 关注: 核心数 + 内存通道

  容器 / K8s:
    - 多核 (支持多 Pod)
    - Bergamo (128C 云原生) / Sierra Forest
    - 关注: 核心密度 + 功耗比

  大数据 (Spark/Flink):
    - 多核 + 高内存带宽
    - EPYC Turin (12 通道 DDR5)
    - 关注: 内存带宽 + PCIe (存储/网络)

  AI 训练/推理:
    - CPU 是配角, 主要看 GPU

  HPC:
    - 高核心 + AVX512/AMX + 高频

## 三、内存详解

### 3.1 内存类型与规格

```
DDR5 vs DDR4:
  ┌─────────────┬─────────────┬─────────────┐
  │             │ DDR4        │ DDR5        │
  ├─────────────┼─────────────┼─────────────┤
  │ 数据率       │ 1600-3200MT│ 4800-8400MT │
  │ 每 DIMM 通道 │ 1           │ 2 (独立)     │
  │ 单模块容量   │ 4-256GB     │ 8-256GB     │
  │ 电压         │ 1.2V       │ 1.1V         │
  │ 引脚         │ 288        │ 288 (不兼容)  │
  │ On-DIMM PMIC│ ❌          │ ✅ (电压调节)│
  │ ECC          │ Side-band  │ On-die + SB  │
  │ Bank        │ 16          │ 32           │
  │ 主流用途     │ 老平台      │ 4th/5th Gen+│
  └─────────────┴─────────────┴─────────────┘

内存模块类型:
  UDIMM   Unbuffered, 桌面用, 服务器不用
  RDIMM   Registered, 服务器主流, 每 CPU 最多 8 条
  LRDIMM  Load-Reduced, 大容量 (256GB+), 双路可达 8TB
  RDIMM 3DS 3D Stacking, 单条 128GB+, 高密度
  MRDIMM  Multi-Ranked, DDR5-8800, 未来
  CXL     计算快速链接, 通过 PCIe 扩展内存

命名规则 (JEDEC):
  Kingston KSM48R40BD8-32MB
           ──── ─── ─── ────
           速度 类型 rank 容量

  48:   4800 MT/s
  R:    RDIMM
  40:   电压 CL (延迟)
  BD8:  组织 (die)
  32:   32GB
  MB:   Micron 芯片
```

### 3.2 内存配置原则

```
黄金规则:

  1. 每 CPU 每通道容量必须一致
     Intel: 8 通道, 16 DIMM/CPU → 每通道 2 DIMM
     AMD Genoa: 12 通道, 24 DIMM/CPU

  2. 填满所有通道 (性能最优)
     ❌ 单 CPU 4 条内存 (只用 4 通道)
     ✅ 单 CPU 8 或 16 条内存 (全通道)

  3. 单 CPU 8 通道最佳带宽
     8 * 4800 MT/s * 8 bytes = 307 GB/s

  4. 混插风险
     - 容量/速度不一致 → 降频到最低
     - Rank 不一致 → 可能训练失败
     - 品牌型号不一致 → 兼容性风险

  5. 双路服务器 NUMA
     每 CPU 独立内存, 跨 CPU 访问慢 2-3 倍
     应用应绑定 NUMA 节点 (numactl)

  6. 容量建议
     ┌──────────────┬──────────┬─────────────┐
     │ 场景         │ 每核内存  │ 双路总容量   │
     ├──────────────┼──────────┼─────────────┤
     │ Web/API      │ 2-4 GB   │ 128-256 GB  │
     │ 数据库       │ 4-8 GB   │ 256-512 GB  │
     │ 虚拟化       │ 4-8 GB   │ 512-1024 GB │
     │ 大数据       │ 4-8 GB   │ 512-1024 GB │
     │ AI 推理      │ 8-16 GB  │ 1-2 TB      │
     │ 内存计算     │ 16-32 GB │ 2-4 TB      │
     └──────────────┴──────────┴─────────────┘
```

# rasdaemon (推荐, 长期收集)

# +----------+---------------------+
# | Label    | Total Errors        |

# | DIMM_A1  | 5 Corrected         |
# | DIMM_B1  | 0                   |
# +----------+---------------------+

# === 内存测试工具 ===

# memtest86+ (启动前测试, 最权威)

# BMC 内嵌诊断 (最推荐)
# iDRAC / iLO / XCC 的 "Memory Diagnostic"

# === 内存故障判定 ===
# 1. 单条 DIMM ce_count 持续增长 → 该 DIMM 老化
# 2. ue_count > 0 → 立即隔离/更换该 DIMM
# 3. 多条同时故障 → 检查主板/内存插槽/CPU IMC
# 4. 系统 kernel panic + MCE → 硬件问题

```

  ┌────────┬──────────┬────────┬────────┬──────┬────────────┐
  │ RAID   │ 最小盘数 │ 容量%  │ 冗余    │ 性能  │ 场景        │
  ├────────┼──────────┼────────┼────────┼──────┼────────────┤
  │ RAID 0 │ 2        │ 100%   │ 无     │ 最快 │ 缓存/临时   │
  │ RAID 1 │ 2        │ 50%    │ 单盘   │ 读快 │ OS 盘       │
  │ RAID 5 │ 3        │ N-1    │ 单盘   │ 中   │ 传统 (慎用)│
  │ RAID 6 │ 4        │ N-2    │ 双盘   │ 中   │ 大容量 HDD  │
  │ RAID 10│ 4        │ 50%    │ 单组   │ 快   │ DB/OLTP    │
  │ RAID 50│ 6        │ N-2    │ 单组   │ 快   │ 高性能大容量│
  │ RAID 60│ 8        │ N-4    │ 双组   │ 中快 │ 归档 + 性能 │

  RAID 选型:
    OS 盘:      RAID 1 (2 * SSD)
    数据库:     RAID 10 (最快 + 冗余, 4-8 * NVMe)

    大数据/HDFS: RAID 0 (软件冗余) 或 JBOD
    存储池:    JBOD + Ceph / ZFS (软件定义)

  RAID 5 慎用原因:
    - 单盘故障后重建时间长 (12TB HDD 可能 24h+)
    - 重建期间又坏一块 → 数据全丢

  └────────────────────┴──────────┴──────────┘

    - Cache (2/4/8 GB): 写回缓存, 加速

  │ 34 ▓▓▓ 空 (散热)       │
  │ 33 [服务器 2U]         │
  │ 32                    │
  │ 31 [服务器 2U]         │
  │ 30                    │
  │ 29 [服务器 2U]         │
  │ ...                   │
  │ 03 [服务器 2U]         │
  │ 02                    │
  │ 01 [PDU/理线]         │
  └───────────────────────┘

  规划原则:
    1. 交换机在顶部 (ToR):
       - 光纤/网线短, 减少信号衰减
       - 便于运维
    2. 重设备靠底部 (存储/大功耗):
       - 重心低, 稳定
    3. 每机柜功耗:
       - 传统机房: 3-5 kW
       - 现代机房: 8-15 kW
       - 高密度/GPU: 20-50 kW (需液冷或冷通道)
    4. 冷热通道:
       - 服务器前吸风 (冷通道 22-25°C)
       - 服务器后排风 (热通道 35-40°C)
       - 冷热隔离, 不混流
    5. 冗余电源:
       - 每台服务器双电源 → 分接两条 PDU (A/B 电源)
       - PDU 分别来自两路市电或 UPS

  机柜尺寸:
    U 高度:   42U (最常见) / 48U / 52U
    宽度:     600mm (标准) / 800mm (高密度走线)
    深度:     1000mm / 1100mm / 1200mm
    承重:     1000-1500kg
    材质:     冷轧钢, 静电喷涂
```

### 6.2 PDU (机柜配电)

```
PDU 类型:

  基础 (Basic PDU):
    - 只提供电源分配
    - 无监控
    - 便宜, 中小机房用

  监控 PDU (Metered):
    - 显示电流/电压/功率
    - LCD 面板 + SNMP
    - 中型机房主流

  智能 PDU (Switched/Managed):
    - 每插口独立监控 + 可远程控制 (通电/断电/重启)
    - Web/SNMP/API
    - 云机房必备

  典型规格:
    输入:   32A/380V 三相 (10-20 kW/柜)
            或 32A/220V 单相 (7 kW)
    插座:   IEC C13 (10A) / C19 (16A)
            国标 GB1002 (10A) / GB1002 (16A)
    输出插座数: 24-48
    形态:  0U 垂直 (侧面) / 1U 水平

  主流品牌:
    - APC (Schneider): AP7XXX 系列, 全球第一
    - Vertiv (Liebert): MPX 系列
    - Raritan:         PX 系列
    - Server Tech:     Sentry 系列
    - 国产: 迈斯克 / 图腾 / 华胜天成

PDU 供电规划:
  ┌──────────────────────────────────┐
  │  A 路市电/UPS  → PDU-A → 服务器 PSU1│
  │  B 路市电/UPS  → PDU-B → 服务器 PSU2│
  └──────────────────────────────────┘

  每机柜双 PDU:
    单机柜总功耗 = PDU-A 负载 + PDU-B 负载
    单条 PDU 负载 <= 单条容量 * 80% (预留冗余)

  预防 "单点满载":
    A/B 电源应交替分配, 避免全部服务器 PSU1 都在 A 路
```

### 6.3 UPS 与配电

```
UPS 类型:

  在线式 (Online, 双转换):
    - 市电 → 整流 → 电池 → 逆变 → 输出
    - 断电切换时间: 0ms (从未断)
    - 效率: 92-96%
    - 关键场景必用

  在线互动 (Line-Interactive):
    - 平时市电直供 + AVR 稳压
    - 断电切换到电池 (~4ms)
    - 效率: 97-98%
    - SMB 场景

  后备式 (Standby):
    - 平时市电直供, 断电才切电池
    - 家用/小办公

  规格计算:
    单机柜: 5kW / 8kW / 15kW
    UPS 容量 (kVA) = 总负载 (kW) / 功率因数 (0.9-0.95)
    冗余: N+X (X 通常 1-2, 例如 4+1 = 5 台)

    电池续航:
      - 5-15 分钟 (等柴油机启动)
      - 30 分钟 (小型机房)
      - 4 小时+ (无发电机的场景, 罕见)

  典型部署:
    ┌──────────────────────────────────────────┐
    │ 市电 A 路 → UPS 组 A → 精密配电柜 → 机柜 A 路│
    │ 市电 B 路 → UPS 组 B → 精密配电柜 → 机柜 B 路│
    │ ATS (自动转换开关) → 柴油发电机 (断电启动)  │
    └──────────────────────────────────────────┘

  主流品牌:
    - APC / Schneider Symmetra / Galaxy
    - Vertiv Liebert (原 Emerson)
    - Eaton 93E/PW
    - 华为 UPS5000
    - 中恒 / 山特 (国产)
```

### 6.4 制冷与散热

```
数据中心制冷方案:

  1. 房间级 (最基础, 老机房):
     - 精密空调 (CRAC/CRAH), 20-100kW
     - 冷通道封闭
     - PUE: 1.8-2.5

  2. 行级 (Row-based):
     - 空调放在机柜行之间
     - 缩短送风距离
     - PUE: 1.4-1.6

  3. 机柜级 (Rack-based):
     - 每机柜后门有热交换器
     - 适合 20 kW+ 高密度
     - PUE: 1.2-1.4

  4. 液冷 (2026 主流):
     ─ 冷板 (Direct-to-Chip):
       液体流经 CPU/GPU 冷板
       液气分离, 相对温和
       PUE: 1.05-1.15
     ─ 浸没式 (Immersion):
       整台服务器泡在绝缘液体
       PUE: 1.03-1.10
       AI 训练/HPC 主流方向

  5. 自然冷源:
     - 新风直供 (冬天, 北方)
     - 蒸发冷却 (干燥地区)
     - 海水/湖水冷却

温度湿度标准:
  推荐:
    冷通道: 22-25°C
    湿度: 40-60% (防静电 + 防结露)
  允许:
    冷通道: 18-27°C (ASHRAE A1)
    宽松: 5-45°C (A4, 少量场景)

热管理:
  - 冷通道封闭 (Cold Aisle Containment)
  - 盲板 (Blanking Panel) 填空 U 位
  - 前后通路无阻挡 (无线缆堵)
  - 温度传感器: 每机柜前/中/后, 3-9 个点
```

### 7.1 BMC 芯片与协议

```

### 7.2 IPMI 实操

```bash
# === 安装 ipmitool ===
dnf install -y ipmitool          # Rocky/RHEL
apt install -y ipmitool          # Ubuntu

# === 本地 IPMI (通过 KCS 接口, 无需网络) ===
ipmitool sensor list             # 传感器
ipmitool sel list                # 事件日志
ipmitool sel clear               # 清事件日志
ipmitool sdr                     # 传感器数据仓库
ipmitool mc info                 # BMC 版本/型号

# 电源
ipmitool power status
ipmitool power on
ipmitool power off               # 硬关机
ipmitool power soft              # 软关机 (发 ACPI)
ipmitool power cycle             # 重启
ipmitool power reset             # 复位

# 引导设备
ipmitool chassis bootdev pxe     # 下次 PXE
ipmitool chassis bootdev disk    # 下次硬盘
ipmitool chassis bootdev cdrom   # 下次光盘

# === 远程 IPMI (LAN) ===
# 首先启用 BMC 网络
ipmitool lan print 1
ipmitool lan set 1 ipsrc static
ipmitool lan set 1 ipaddr 10.0.100.10
ipmitool lan set 1 netmask 255.255.255.0
ipmitool lan set 1 defgw ipaddr 10.0.100.1
ipmitool lan set 1 access on

# 用户管理
ipmitool user list 1
ipmitool user set name 2 admin
ipmitool user set password 2 Pass@2026
ipmitool user priv 2 4 1         # 用户 2, 权限 4=Administrator, 通道 1
ipmitool user enable 2

# 远程操作
export IPMI_ARGS="-I lanplus -H 10.0.100.10 -U admin -P Pass@2026"
ipmitool $IPMI_ARGS power status
ipmitool $IPMI_ARGS sensor list
ipmitool $IPMI_ARGS sel list

# 温度/风扇
ipmitool $IPMI_ARGS sdr type temperature
ipmitool $IPMI_ARGS sdr type fan

# === Serial over LAN (SoL, 串口重定向) ===
# BIOS 里启用 SoL Console Redirection

ipmitool $IPMI_ARGS sol activate
# 按 ~. 退出 (回车+波浪号+点)

# 用途: OS 启动过程 / 内核崩溃 / 网卡失效时也能远程操作!

# === 批量操作脚本 ===
cat > /opt/scripts/ipmi-batch.sh << 'EOF'
#!/bin/bash
# 批量收集集群硬件信息

HOSTS="10.0.100.10 10.0.100.11 10.0.100.12"
USER=admin
PASS='Pass@2026'

for h in ${HOSTS}; do
  echo "=== ${h} ==="
  ipmitool -I lanplus -H ${h} -U ${USER} -P ${PASS} \
    mc info | grep -E "Manufacturer|Product Name|Firmware"
  ipmitool -I lanplus -H ${h} -U ${USER} -P ${PASS} \
    sensor list | awk -F '|' '/CPU|Temp|Fan/ {print $1, $2}'
done
EOF
```

### 7.3 Redfish (现代管理)

```bash
# === Redfish 是 RESTful, 用 curl / Python 即可 ===

# 系统信息
curl -k -u admin:Pass@2026 https://10.0.100.10/redfish/v1/Systems/System.Embedded.1 \
  | jq '.'

# 电源
curl -k -u admin:Pass@2026 -X POST \
  -H "Content-Type: application/json" \
  -d '{"ResetType": "ForceRestart"}' \
  https://10.0.100.10/redfish/v1/Systems/System.Embedded.1/Actions/ComputerSystem.Reset

# 存储列表
curl -k -u admin:Pass@2026 https://10.0.100.10/redfish/v1/Systems/System.Embedded.1/Storage \
  | jq '.Members'

# 传感器
curl -k -u admin:Pass@2026 https://10.0.100.10/redfish/v1/Chassis/System.Embedded.1/Thermal \
  | jq '.Temperatures[] | {Name, ReadingCelsius}'

# 事件订阅 (SubscribedEvents)
curl -k -u admin:Pass@2026 -X POST \
  -H "Content-Type: application/json" \
  -d '{
    "Destination": "https://alerts.example.com/webhook",
    "EventTypes": ["Alert", "StatusChange"],
    "Context": "prod-datacenter",
    "Protocol": "Redfish"
  }' \
  https://10.0.100.10/redfish/v1/EventService/Subscriptions

# === Python 库 ===
pip install redfish

python3 << 'EOF'
import redfish

conn = redfish.redfish_client(
    base_url='https://10.0.100.10',
    username='admin', password='Pass@2026',
    default_prefix='/redfish/v1'
)
conn.login()

# 系统信息
resp = conn.get('/redfish/v1/Systems/System.Embedded.1')
print(resp.dict['Model'], resp.dict['SerialNumber'])

# 内存
resp = conn.get('/redfish/v1/Systems/System.Embedded.1/Memory')
for m in resp.dict['Members']:
    dimm = conn.get(m['@odata.id']).dict
    print(dimm['Name'], dimm.get('CapacityMiB'), 'MiB')

conn.logout()
EOF
```

## 十、故障处理

### 10.1 硬件故障速查表

| 故障 | 现象 | 排查 | 处置 |
|:---|:---|:---|:---|
| **服务器不加电** | 按开机无反应 | 检查电源灯 / PDU / 电源线 | 换电源模块 / PSU 冗余测试 |
| **开机自检失败** | POST 卡住 / 蜂鸣 | BMC 日志 / SEL / 蜂鸣代码 | 拔插内存/CPU / 换主板 |
| **反复重启** | 启动到内核就重启 | MCE 日志 / dmesg / syslog | 查 CPU/内存 ECC, 换硬件 |
| **单核异常** | 某 CPU 温度飙升 | `sensors` / `turbostat` | 换 CPU / 检查散热 |
| **内存条故障** | ECC 错误 / kpanic | `edac-util --report=full` | 换 DIMM |
| **磁盘故障** | I/O error / SMART 警告 | `smartctl -a` | 换盘, RAID 自动重建 |
| **RAID 降级** | RAID 卡告警 | `storcli /c0 show` | 换故障盘, 观察重建 |
| **RAID 卡故障** | 所有盘不可见 | RAID 卡指示灯/BMC | 换 RAID 卡, 保 config |
| **网卡失联** | ping 不通 | `ethtool` / `dmesg` | 换网线/模块/网卡 |
| **光模块故障** | 端口 Down | `ethtool -m` | 换光模块 (先清洁试试) |
| **风扇故障** | 温度升高 / BMC 告警 | `ipmitool sensor list` | 换风扇, 冗余可撑 |
| **电源故障** | PSU 灯红 / BMC 告警 | `sensor` / iLO/iDRAC | 热插拔换 PSU |
| **GPU 掉线** | nvidia-smi 找不到 | dmesg Xid / 供电 / PCIe 链路 | 重启 / 换 GPU |
| **系统夯死** | 无响应, ping 通 | Kdump / SoL 抓崩溃 | 通过 BMC 强制重启 |

### 10.3 硬件更换实操

```bash
# === 硬盘热插拔更换 (RAID 环境) ===

# 1. 确认故障盘物理位置
storcli64 /c0/eall/sall show
# 找到 State = Failed 的盘: 例如 e252/s3

# 2. 定位灯闪烁 (方便运维找)
storcli64 /c0/eall/s3 start locate

# 3. 物理拔出故障盘 (机箱前面板 SAS/NVMe 支持热插拔)

# 4. 插入新盘, 等待自动检测
watch -n 1 'storcli64 /c0/eall/sall show'
# 新盘应显示为 Unconfigured Good (UGood)

# 5. 加入 RAID (自动重建)
# 若热备盘 (hotspare) 已配置, 会自动开始重建
# 手动加入:
storcli64 /c0/eall/s3 add hotsparedrive dgs=0

# 或替换指定盘:
storcli64 /c0/vall replacemissing pd=252:3 array=0 row=3

# 查看重建进度
storcli64 /c0/eall/s3 show rebuild
# rebuild 时间: 12TB HDD 约 24-48 小时

# 6. 完成后关闭定位灯
storcli64 /c0/eall/s3 stop locate

# === 内存条更换 ===
# 1. 关机 (内存不支持热插拔, 除非特殊 Persistent Memory)
poweroff

# 2. 断电, 打开机箱, 找到故障 DIMM
# 通过 SEL 或 edac 定位: DIMM_A1 / CPU0_A1 / P0-DIMMA0 (品牌不同)
# BMC UI 一般有物理示意图

# 3. 更换后加电
# 首次开机 BIOS 会重新训练内存, 时间较长 (数分钟)

# 4. 验证
ipmitool sel elist | tail -20
dmidecode -t memory | grep -A 15 "DIMM_A1"
edac-util --report=full

# === CPU 更换 ===
# 1. 关机 + 断电
# 2. 卸下散热器 (小心弯针脚)
# 3. 换 CPU + 重涂硅脂 (推荐: TG-K7 Pro / Cooler Master IceFusion)
# 4. 装回散热器
# 5. 开机 BIOS 中检测新 CPU
# 6. 更新 BIOS 微码 (如果新 CPU 需要)

# === PCIe 卡 (GPU/网卡/RAID) 更换 ===
# 1. 关机断电 (少数支持热插, 需 hot-plug PCIe slot)
# 2. 卸载驱动 (可选, 避免遗留状态)
#    rmmod nvidia         # GPU
#    rmmod ice           # 网卡
# 3. 物理更换
# 4. 开机, 重新加载驱动
#    modprobe nvidia
# 5. 验证 lspci / nvidia-smi / ethtool

# === 电源模块 (支持热插) ===
# 1. 观察冗余状态 (需 1+1 或 N+1)
ipmitool sensor list | grep -i power
# 2. 定位红灯 PSU
# 3. 直接拔出 (机箱后, PSU 会有拉手)
# 4. 插入新 PSU, 观察指示灯变绿
# 5. 确认 BMC 告警消失
```
