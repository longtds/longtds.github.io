> ELK、Loki、Graylog 怎么选？日志采集、传输、存储、查询全链路怎么搭？本文覆盖架构原理、部署配置和生产实践。

## 一、日志系统选型

### 1.1 日志系统的核心挑战

```
日志处理的完整链路:

  采集 → 传输 → 处理 → 存储 → 查询 → 告警

  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
  │ 采集  │→│ 传输  │→│ 处理  │→│ 存储  │→│ 查询  │→│ 告警  │
  │      │  │      │  │      │  │      │  │      │  │      │
  │File  │  │Kafka │  │解析   │  │ES    │  │Kibana│  │告警   │
  │Beat  │  │Queue │  │过滤   │  │Loki  │  │Grafana│ │规则   │
  │Agent │  │      │  │富化   │  │ClickH│  │      │  │通知   │
  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘

挑战:
  1. 数据量大 (日均 TB 级)
  2. 多源异构 (应用日志/Nginx/MySQL/系统日志/容器日志)
  3. 实时性要求 (秒级可查)
  4. 查询性能 (全文搜索 + 聚合分析)
  5. 成本控制 (存储成本随数据量线性增长)
  6. 可靠性 (日志不能丢)
```

### 1.2 主流方案对比

| 维度 | ELK Stack | Grafana Loki | Graylog | ClickHouse 栈 |
|:---|:---|:---|:---|:---|
| 存储 | Elasticsearch | 对象存储/FS | Elasticsearch | ClickHouse |
| 采集 | Filebeat/Logstash | Promtail/Alloy | Sidecar/Filebeat | Vector |
| 查询 | Lucene (DSL) | LogQL | Lucene (GEL) | SQL |
| 索引 | 全文索引 (倒排) | ⭐ 仅索引标签 | 全文索引 (倒排) | 列存 (无全文) |
| 压缩比 | 1x (基准) | ⭐ 5-10x | 1x | ⭐ 5-10x |
| 资源占用 | 高 (JVM) | ⭐ 低 (Go) | 高 (JVM) | ⭐ 低 (C++) |
| 存储成本 | 高 | ⭐ 极低 | 高 | ⭐ 低 |
| 全文搜索 | ⭐ 最强 | 弱 (仅标签) | 强 | 弱 (LIKE) |
| 聚合分析 | 强 | 中 | 强 | ⭐ 最强 |
| 实时性 | 近实时 (1s) | 实时 | 近实时 | 实时 |
| 告警 | Kibana Alerting | Grafana Alerting | 内置告警 | 第三方 |
| K8s 原生 | ⭐ 强 | ⭐ 强 | 中 | 中 |
| 运维复杂度 | 高 | ⭐ 低 | 中 | 中 |
| 许可证 | SSPL (ES) | AGPLv3 | GPL v3 | Apache 2.0 |
| 适用场景 | 全文搜索/日志分析 | 大规模低成本日志 | 企业日志管理 | 高吞吐日志分析 |

### 1.3 选型建议

| 场景 | 推荐 | 原因 |
|:---|:---|:---|
| 全文搜索 + 日志分析 | **ELK** | 搜索能力最强，Kibana 生态丰富 |
| 大规模低成本日志 | **Loki** | 仅索引标签，存储成本极低 |
| 企业级日志管理 | **Graylog** | 内置告警/流解析/权限管理 |
| 超高吞吐 + 聚合分析 | **ClickHouse 栈** | 写入极快，列存聚合最强 |
| K8s 容器日志 | **Loki + Promtail** | 与 Grafana/Prom 无缝集成 |
| 已有 ES 基础 | **ELK / Graylog** | 复用 ES 集群 |
| 资源有限 | **Loki** | Go 编写，无 JVM 开销 |

## 二、ELK Stack

### 2.1 架构原理

```
经典 ELK 架构:

  ┌──────────┐
  │ 日志文件   │  Nginx/MySQL/App 日志
  └────┬─────┘
       │
  ┌────▼─────┐
  │ Filebeat │  轻量采集器 (Go, 资源占用低)
  │ (Agent)  │  读文件 → 多行合并 → 缓冲 → 发送
  └────┬─────┘
       │
  ┌────▼─────┐
  │  Kafka   │  缓冲层 (削峰, 解耦, 多消费)
  │ (可选)   │  高峰期日志先入队
  └────┬─────┘
       │
  ┌────▼─────┐
  │ Logstash │  处理引擎 (JVM, 资源占用高)
  │          │  解析 (Grok) → 过滤 → 富化 (GeoIP) → 输出
  └────┬─────┘
       │
  ┌────▼─────┐
  │Elasticsea│  存储引擎 (倒排索引)
  │ rch      │  索引 → 分片 → 副本
  └────┬─────┘
       │
  ┌────▼─────┐
  │ Kibana   │  可视化 (仪表盘/搜索/告警)
  └──────────┘

Filebeat vs Logstash:
  Filebeat: 轻量采集 (Go), 只做读取/发送, 不做复杂处理
  Logstash: 重量处理 (JVM), 解析/过滤/富化, 资源占用高

  生产推荐: Filebeat → Kafka → Logstash → ES
  轻量方案: Filebeat → ES (Filebeat 内置简单处理)
```

### 2.2 Elasticsearch 配置（日志场景）

```bash
      "hot": {
        "actions": {

        "actions": {
          "delete": {}
        }
      }

# 初始别名
```

### 2.3 Filebeat 部署

```bash
# === 安装 ===
# Rocky / RHEL
rpm --import https://packages.elastic.co/GPG-KEY-elasticsearch
cat > /etc/yum.repos.d/elastic.repo << 'EOF'
[elastic-8.x]
name=Elastic repository for 8.x packages
baseurl=https://artifacts.elastic.co/packages/8.x/yum
gpgcheck=1
gpgkey=https://artifacts.elastic.co/GPG-KEY-elasticsearch
enabled=1
type=rpm-md
EOF
dnf install -y filebeat

# Ubuntu
curl -fsSL https://artifacts.elastic.co/GPG-KEY-elasticsearch | gpg --dearmor -o /usr/share/keyrings/elastic-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/elastic-keyring.gpg] https://artifacts.elastic.co/packages/8.x/apt stable main" > /etc/apt/sources.list.d/elastic-8.x.list
apt update && apt install -y filebeat

# === 配置: /etc/filebeat/filebeat.yml ===
cat > /etc/filebeat/filebeat.yml << 'EOF'
# ================== Filebeat 输入 ==================

# Nginx 访问日志
filebeat.inputs:
- type: filestream
  id: nginx-access
  enabled: true
  paths:
    - /var/log/nginx/access.log
  parsers:
    - multiline:
        type: count
        count_lines: 1
  fields:
    service: nginx
    log_type: access
  fields_under_root: true

# Nginx 错误日志
- type: filestream
  id: nginx-error
  enabled: true
  paths:
    - /var/log/nginx/error.log
  fields:
    service: nginx
    log_type: error
  fields_under_root: true

# 应用日志 (Java 堆栈多行合并)
- type: filestream
  id: app-log
  enabled: true
  paths:
    - /opt/app/logs/*.log
  parsers:
    - multiline:
        type: pattern
        pattern: '^\d{4}-\d{2}-\d{2}'           # 以日期开头的行
        negate: true
        match: after                              # 不匹配的行合并到前一行
  fields:
    service: myapp
    log_type: application
  fields_under_root: true

# MySQL 慢查询日志
- type: filestream
  id: mysql-slow
  enabled: true
  paths:
    - /var/log/mysql/slow.log
  parsers:
    - multiline:
        type: pattern
        pattern: '^# Time:|^# User@Host|^SET|^USE|^UPDATE|^INSERT|^DELETE|^SELECT|^COMMIT'
        negate: true
        match: after
        flush_pattern: '^\# Time:'
  fields:
    service: mysql
    log_type: slow_query
  fields_under_root: true

# 系统日志
- type: filestream
  id: syslog
  enabled: true
  paths:
    - /var/log/messages
    - /var/log/secure
  fields:
    service: system
    log_type: syslog
  fields_under_root: true

# ================== 处理器 ==================
processors:
  # 添加主机信息
  - add_host_metadata:
      when.not.contains.tags: forwarded
  # 添加云信息
  - add_cloud_metadata: ~
  # 添加 Docker 容器信息
  - add_docker_metadata: ~
  # 解析 Nginx 日志
  - decode_json_fields:
      fields: ['message']
      process_array: false
      max_depth: 1
      target: ''
      overwrite_keys: false
  # 删除不需要的字段 (减少存储)
  - drop_fields:
      fields:
        - agent.ephemeral_id
        - agent.id
        - agent.type
        - agent.version
        - ecs.version
        - input.type
        - log.offset
      ignore_missing: true

# ================== 输出 ==================

# 方式一: 直接输出到 ES (简单方案)
output.elasticsearch:
  hosts: ["https://192.168.1.10:9200"]
  username: "filebeat"
  password: "FilebeatPass2026!"
  ssl.verification_mode: "certificate"
  # 索引名 (匹配 ILM 模板)
  indices:
    - index: "logs-nginx-access-%{+yyyy.MM.dd}"
      when.equals:
        service: "nginx"
    - index: "logs-app-%{+yyyy.MM.dd}"
      when.equals:
        service: "myapp"
    - index: "logs-mysql-slow-%{+yyyy.MM.dd}"
      when.equals:
        service: "mysql"

# 方式二: 输出到 Kafka (推荐, 高吞吐场景)
# output.kafka:
#   hosts: ["192.168.1.10:9092", "192.168.1.11:9092"]
#   topic: "logs"
#   partition.round_robin:
#     reachable_only: true
#   required_acks: 1
#   compression: lz4
#   max_message_bytes: 1000000

# 方式三: 输出到 Logstash (复杂处理)
# output.logstash:
#   hosts: ["192.168.1.20:5044"]
#   loadbalance: true
#   worker: 4
#   bulk_max_size: 2048

# ================== 全局 ==================
queue.mem:
  events: 4096
  flush.min_events: 2048
  flush.timeout: 1s

# 日志
logging.level: info
logging.to_files: true
logging.files:
  path: /var/log/filebeat
  name: filebeat
  keepfiles: 7
  permissions: 0640
EOF

systemctl enable --now filebeat

# 验证
filebeat test config
filebeat test output
```

### 2.4 Logstash 配置

```bash
# 安装 (与 Filebeat 同源)
dnf install -y logstash

# /etc/logstash/conf.d/logs.conf
cat > /etc/logstash/conf.d/logs.conf << 'EOF'
# === 输入 ===
input {
  # 从 Kafka 消费
  kafka {
    bootstrap_servers => "192.168.1.10:9092,192.168.1.11:9092"
    topics => ["logs"]
    group_id => "logstash-consumer"
    consumer_threads => 4
    decorate_events => true
    auto_offset_reset => "earliest"
    # 解码为 JSON
    codec => json
  }

  # 从 Filebeat 接收
  # beats {
  #   port => 5044
  #   ssl => false
  # }
}

# === 过滤处理 ===
filter {
  # Nginx 访问日志解析
  if [service] == "nginx" and [log_type] == "access" {
    grok {
      match => {
        "message" => [
          # Nginx combined 格式
          '%{IPORHOST:client_ip} - %{DATA:user} \[%{HTTPDATE:timestamp}\] "%{WORD:method} %{URIPATHPARAM:request} HTTP/%{NUMBER:http_version}" %{NUMBER:status:int} %{NUMBER:bytes:int} "%{DATA:referer}" "%{DATA:user_agent}" rt=%{NUMBER:request_time:float} uct=%{NUMBER:upstream_connect_time:float} uht=%{NUMBER:upstream_header_time:float} urt=%{NUMBER:upstream_response_time:float}'
        ]
      }
      overwrite => ["message"]
      tag_on_failure => ["_grokparsefailure"]
    }

    # GeoIP 富化
    geoip {
      source => "client_ip"
      target => "geoip"
      fields => ["country_name", "region_name", "city_name"]
    }

    # UA 解析
    useragent {
      source => "user_agent"
      target => "ua"
    }
  }

  # Java 应用日志解析
  if [service] == "myapp" {
    grok {
      match => {
        "message" => "%{TIMESTAMP_ISO8601:log_time} \[%{DATA:thread}\] %{LOGLEVEL:level} %{DATA:logger} - %{GREEDYDATA:message}"
      }
      overwrite => ["message"]
    }
  }

  # MySQL 慢查询解析
  if [service] == "mysql" and [log_type] == "slow_query" {
    grok {
      match => {
        "message" => [
          "# Time: %{TIMESTAMP_ISO8601:query_time}",
          "# User@Host: %{DATA:user}\[%{DATA:user}\] @ %{DATA:host} \[%{IP:host_ip}\]",
          "# Query_time: %{NUMBER:query_time:float}  Lock_time: %{NUMBER:lock_time:float}",
          "SET timestamp=%{NUMBER:timestamp};",
          "%{GREEDYDATA:sql_query}"
        ]
        break_on_match => false
      }
    }
  }

  # 通用处理
  date {
    match => ["timestamp", "dd/MMM/yyyy:HH:mm:ss Z", "ISO8601"]
    target => "@timestamp"
  }

  # 删除不需要的字段
  mutate {
    remove_field => ["@version", "host", "agent", "ecs", "input", "log"]
  }
}

# === 输出 ===
output {
  # 写入 ES
  elasticsearch {
    hosts => ["https://192.168.1.10:9200"]
    user => "logstash"
    password => "LogstashPass2026!"
    ssl => true
    ssl_certificate_verification => false

    # 按服务分索引
    index => "%{[service]}-%{+yyyy.MM.dd}"

    # ILM
    ilm_enabled => true
    ilm_rollover_alias => "logs-%{[service]}"
    ilm_pattern => "000001"
    ilm_policy => "logs-ilm"
  }
}

# 性能调优
# /etc/logstash/logstash.yml
# pipeline.workers: 4          # 工作线程 (= CPU 核数)
# pipeline.batch.size: 1000     # 批处理大小
# pipeline.batch.delay: 50      # 批处理延迟 (ms)
# queue.type: persisted         # 持久化队列 (防丢数据)
# path.queue: /var/lib/logstash/queue
# dead_letter_queue.enable: true
EOF

systemctl enable --now logstash
```

## 三、Grafana Loki

### 3.1 架构原理

```
┌─────────────────────────────────────────────────────────┐
│                    Loki 架构                             │
│                                                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ Promtail │  │ Alloy    │  │ Fluent   │              │
│  │ (采集)   │  │ (采集)   │  │ Bit      │              │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘              │
│       │              │             │                      │
│       │ Push (日志+标签)            │                      │
│       ▼              ▼             ▼                      │
│  ┌──────────────────────────────────────────┐           │
│  │              Distributor                  │           │
│  │  接收日志 → 验证 → 分割 chunk → 分发       │           │
│  └──────────────────┬───────────────────────┘           │
│                     │                                    │
│  ┌──────────────────▼───────────────────────┐           │
│  │              Ingester                     │           │
│  │  缓存日志 → 压缩 → 切块                   │           │
│  │  内存 → 磁盘 → 对象存储                    │           │
│  └──────────────────┬───────────────────────┘           │
│                     │                                    │
│  ┌──────────────────▼───────────────────────┐           │
│  │          对象存储 (S3/MinIO/FS)            │           │
│  │                                          │           │
│  │  chunks/   ← 日志数据 (压缩, 仅标签索引)   │           │
│  │  index/    ← 标签索引 (BoltDB)             │           │
│  └──────────────────┬───────────────────────┘           │
│                     │                                    │
│  ┌──────────────────▼───────────────────────┐           │
│  │              Querier                      │           │
│  │  接收 LogQL → 查索引 → 拉 chunk → 解压     │           │
│  └──────────────────┬───────────────────────┘           │
│                     │                                    │
│  ┌──────────────────▼───────────────────────┐           │
│  │              Grafana                      │           │
│  │  日志查询 + 仪表盘 + 告警                  │           │
│  └──────────────────────────────────────────┘           │
│                                                         │
│  Loki vs ELK 核心差异:                                   │
│    ELK:  对每条日志建倒排索引 (全文搜索, 存储成本高)      │
│    Loki: 仅对标签建索引, 日志数据压缩存储 (存储成本低)     │
│                                                         │
│    ELK:  适合全文搜索 (搜关键词)                         │
│    Loki: 适合标签过滤 (按服务/级别/主机筛选)              │
└─────────────────────────────────────────────────────────┘

Loki 数据模型:
  日志流 (Stream) = 标签集合的唯一组合

  {service="nginx", level="error", host="web-01"} → 日志流 1
  {service="nginx", level="access", host="web-01"} → 日志流 2
  {service="myapp", level="info", host="app-01"} → 日志流 3

  每个日志流 = 多个 Chunk (压缩的日志块)
  Chunk = 多条日志行 (按时间排序)

  查询流程:
  1. LogQL 查询 → 解析标签选择器 {service="nginx"}
  2. 查索引 → 找到匹配的日志流
  3. 拉取 Chunk → 解压 → 按时间范围过滤
  4. 可选: 正则过滤/JSON 解析/聚合
```

### 3.5 LogQL 查询

```logql
# === 日志查询 (Log Query) ===

# 基本查询 (标签选择器)
{job="nginx"}                                     # 所有 nginx 日志
{job="nginx", level="error"}                      # nginx 错误日志
{job="myapp", level=~"ERROR|WARN"}                # ERROR 或 WARN
{namespace="production"}                           # K8s 命名空间

# 过滤日志内容
{job="nginx"} |= "timeout"                        # 包含 timeout
{job="nginx"} != "200"                            # 不包含 200
{job="nginx"} |~ "error|exception"                # 正则匹配
{job="nginx"} !~ "debug|trace"                    # 正则排除

# JSON 解析
{job="myapp"} | json                              # 解析 JSON 日志
{job="myapp"} | json | level="ERROR"              # 解析后过滤字段
{job="myapp"} | json | status >= 500              # 数字比较

# 日志格式化
{job="nginx"} | logfmt                            # 解析 logfmt 格式
{job="nginx"} | pattern "<ip> - - <ts> \"<method>"  # 模式匹配

# 正则提取
{job="nginx"} | regexp "duration=(?P<duration>\\d+)" | duration > 5

# === 指标查询 (Metric Query) ===

# 日志计数 (每分钟)
rate({job="nginx"}[1m])

# 按标签分组计数
sum by (level) (rate({job="myapp"}[5m]))

# 错误率
sum(rate({job="myapp", level="ERROR"}[5m])) by (host) /
sum(rate({job="myapp"}[5m])) by (host) * 100

# P99 延迟 (从日志中提取)
quantile_over_time(0.99,
  {job="nginx"} | json | unwrap request_time [5m]
) by (host)

# TopK (错误最多的主机)
topk(10, sum by (host) (rate({job="myapp", level="ERROR"}[1h])))

# === 告警规则 (Ruler) ===
# /var/lib/loki/rules/rules.yaml
groups:
  - name: log-alerts
    rules:
      - alert: HighErrorRate
        expr: |
          sum by (service) (rate({job="myapp", level="ERROR"}[5m])) > 10
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "日志错误率过高 ({{ $labels.service }})"

      - alert: Nginx5xxErrors
        expr: |
          sum by (host) (rate({job="nginx", level="error"}[5m])) > 5
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "Nginx 5xx 错误 ({{ $labels.host }})"
```

## 五、Vector（高性能日志管道）

### 5.1 特点

```
Vector — Rust 编写的可观测数据管道

核心优势:
  1. Rust 编写, 无 JVM/GC 开销
  2. 吞吐量比 Logstash 高 10x+
  3. 内存占用低 (Logstash 的 1/10)
  4. 支持指标/日志/追踪 三种数据类型
  5. 内置 200+ 组件 (source/transform/sink)
  6. 支持 WASM 自定义处理
  7. 配置式 (TOML/YAML), 无需写代码

  - 替代 Logstash (更高性能, 更低资源)
  - 替代 Filebeat + Logstash (一步到位)
  - 日志/指标统一采集
  - ClickHouse 日志栈的采集层
```

### 5.2 安装与配置

```bash
# 安装
curl --proto '=https' --tlsv1.2 -sSf https://sh.vector.dev | sh
# 或
dnf install -y vector

# 配置: /etc/vector/vector.toml
cat > /etc/vector/vector.toml << 'EOF'
# === 数据源 (Source) ===
[sources.app_logs]
type = "file"
include = ["/opt/app/logs/*.log"]
read_from = "end"
multiline.start_pattern = '^\d{4}-\d{2}-\d{2}'
multiline.mode = "halt_before"
multiline.condition_pattern = '^\d{4}-\d{2}-\d{2}'
multiline.timeout_ms = 1000

[sources.nginx_access]
type = "file"
include = ["/var/log/nginx/access.log"]

[sources.syslog]
type = "journald"          # systemd journal

# === 处理 (Transform) ===
# 解析 Nginx 日志
[transforms.parse_nginx]
type = "remap"
inputs = ["nginx_access"]
source = '''
# 解析为结构化
.parsed = parse_regex!(.message, r'^(?P<ip>\S+) - (?P<user>\S+) \[(?P<time>[^\]]+)\] "(?P<method>\S+) (?P<path>\S+) (?P<proto>[^"]+)" (?P<status>\d+) (?P<bytes>\d+) "(?P<referer>[^"]*)" "(?P<ua>[^"]*)" rt=(?P<rt>[\d.]+)$')

# 展开字段
.ip = .parsed.ip
.method = .parsed.method
.status = to_int(.parsed.status)
.bytes = to_int(.parsed.bytes)
.request_time = to_float(.parsed.rt)

# 添加日志级别
.level = if .status >= 500 { "error" } else if .status >= 400 { "warn" } else { "info" }

# 删除临时字段
del(.parsed)
'''

# 解析 JSON 应用日志
[transforms.parse_app]
type = "remap"
inputs = ["app_logs"]
source = '''
. = parse_json!(.message)
.timestamp = parse_timestamp!(.timestamp, "%Y-%m-%d %H:%M:%S%.3f")
'''

# 添加主机信息
[transforms.add_host]
type = "add_fields"
inputs = ["parse_nginx", "parse_app"]
fields.hostname = "${HOSTNAME}"
fields.env = "production"

# === 输出 (Sink) ===
# 写入 ClickHouse
[sinks.clickhouse]
type = "clickhouse"
inputs = ["add_host"]
endpoint = "http://192.168.1.10:8123"
database = "logs"
table = "app_logs"
compression = "lz4"
skip_unknown_fields = true

# 写入 Loki
[sinks.loki]
type = "loki"
inputs = ["add_host"]
endpoint = "http://192.168.1.10:3100"
labels.service = "{{ service }}"
labels.level = "{{ level }}"
labels.host = "{{ hostname }}"

# 写入 Kafka
[sinks.kafka]
type = "kafka"
inputs = ["add_host"]
bootstrap_servers = "192.168.1.10:9092,192.168.1.11:9092"
topic = "logs"
compression = "lz4"
```

## 六、ClickHouse 日志栈

### 6.1 架构

```
ClickHouse 日志分析栈:

  ┌──────────┐
  │ 应用日志  │
  └────┬─────┘
       │
  ┌────▼─────┐
  │ Vector   │  采集 + 解析
  └────┬─────┘
       │
  ┌────▼─────┐         ┌──────────┐
  │  Kafka   │◄────────│ 缓冲削峰  │
  └────┬─────┘         └──────────┘
       │
  ┌────▼─────┐
  │ClickHouse│  存储 + 查询 (列存, 高压缩, 极速聚合)
  │ Kafka引擎 │  直接消费 Kafka
  └────┬─────┘
       │
  ┌────▼─────┐
  │ Grafana  │  可视化 (SQL 数据源)
  └──────────┘

优势:
  - 写入吞吐极高 (百万行/秒)
  - 压缩比 5-10x (存储成本低)
  - 聚合查询极快 (列存 + SIMD)
  - SQL 接口 (比 DSL 更灵活)

劣势:
  - 无全文索引 (只能 LIKE 搜索)
  - 不适合频繁更新
  - 需要预定义 Schema
```

### 6.2 建表与消费

```sql
-- ClickHouse 日志表 (参考 12-列式数据库.md)
CREATE TABLE logs.app_log
(
    timestamp     DateTime64(3),
    level         LowCardinality(String),
    service       LowCardinality(String),
    host          LowCardinality(String),
    env           LowCardinality(String),
    trace_id      String,
    message       String,
    method        LowCardinality(String),
    status_code   UInt16,
    request_time  Float32,
    client_ip     IPv4,
    properties    String                     -- JSON 扩展字段
)
ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(timestamp)
ORDER BY (service, level, timestamp)
TTL timestamp + INTERVAL 30 DAY
SETTINGS compression = 'lz4';

-- Kafka 消费表
CREATE TABLE logs.app_log_kafka
(
    timestamp     DateTime64(3),
    level         String,
    service       String,
    host          String,
    env           String,
    trace_id      String,
    message       String,
    method        String,
    status_code   UInt16,
    request_time  Float32,
    client_ip     String,
    properties    String
)
ENGINE = Kafka()
SETTINGS
    kafka_broker_list = '192.168.1.10:9092,192.168.1.11:9092',
    kafka_topic_list = 'app-logs',
    kafka_group_name = 'clickhouse-logs',
    kafka_format = 'JSONEachRow',
    kafka_num_consumers = 3;

-- 物化视图 (Kafka → MergeTree)
CREATE MATERIALIZED VIEW logs.app_log_mv TO logs.app_log
AS
SELECT
    timestamp,
    level,
    service,
    host,
    env,
    trace_id,
    message,
    method,
    status_code,
    request_time,
    toIPv4(client_ip) AS client_ip,
    properties
FROM logs.app_log_kafka;

-- 查询 (SQL)
-- 按级别统计
SELECT level, count() AS cnt
FROM logs.app_log
WHERE timestamp >= today() - 1
GROUP BY level
ORDER BY cnt DESC;

-- 错误日志 TOP 10
SELECT timestamp, service, host, message
FROM logs.app_log
WHERE level = 'ERROR' AND timestamp >= now() - INTERVAL 1 HOUR
ORDER BY timestamp DESC
LIMIT 10;

-- P99 响应时间 (按服务)
SELECT
    service,
    quantile(0.99)(request_time) AS p99,
    quantile(0.95)(request_time) AS p95,
    avg(request_time) AS avg_rt,
    count() AS total
FROM logs.app_log
WHERE timestamp >= today() - 1 AND level = 'access'
GROUP BY service
ORDER BY p99 DESC;

-- 模糊搜索 (LIKE, 性能不如 ES 全文索引)
SELECT timestamp, message
FROM logs.app_log
WHERE message LIKE '%timeout%'
  AND timestamp >= now() - INTERVAL 10 MINUTE
LIMIT 50;

-- Grafana 数据源: ClickHouse
-- 使用 SQL 查询日志, 配合 Grafana Logs 面板
```

## 七、对比总结

### 7.1 方案对比

| 维度 | ELK | Loki | Graylog | ClickHouse 栈 |
|:---|:---|:---|:---|:---|
| 全文搜索 | ⭐ 最强 | 弱 | 强 | 弱 (LIKE) |
| 聚合分析 | 强 | 中 | 强 | ⭐ 最强 |
| 写入吞吐 | 中 | 高 | 中 | ⭐ 最高 |
| 压缩比 | 1x | ⭐ 5-10x | 1x | ⭐ 5-10x |
| 存储成本 | 高 | ⭐ 极低 | 高 | ⭐ 低 |
| 资源占用 | 高 (JVM) | ⭐ 低 (Go) | 高 (JVM) | ⭐ 低 (C++) |
| 查询语言 | KQL/DSL | LogQL | GEL | SQL |
| 告警 | Kibana | Grafana | ⭐ 内置 | 第三方 |
| K8s 集成 | 强 | ⭐ 最强 | 中 | 中 |
| 多租户 | 中 | ⭐ 强 | 强 | 中 |
| 运维难度 | 高 | ⭐ 低 | 中 | 中 |

### 7.2 选型决策树

```
需要日志系统?
├── 需要全文搜索 (搜关键词)
│   └── ELK (ES 倒排索引最强)
│
├── 需要低成本大规模存储
│   └── Loki (仅索引标签, 压缩比 5-10x)
│
├── 需要企业级管理 (权限/告警/流路由)
│   └── Graylog (内置全套功能)
│
├── 需要超高吞吐 + 聚合分析
│   └── ClickHouse 栈 (Vector + Kafka + CH)
│
├── 已有 Grafana + Prom
│   └── Loki (无缝集成, 统一可观测)
│
└── 已有 ES 集群
    └── ELK / Graylog (复用存储)
```
