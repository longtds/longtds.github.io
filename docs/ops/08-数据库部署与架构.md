# 数据库部署配置与架构原理

> MySQL 和 PostgreSQL 怎么选？Redis 为什么要持久化？本文覆盖主流数据库的部署配置、内部架构原理和生产实践。

## 一、数据库选型

### 1.1 关系型数据库对比

| 维度 | MySQL | PostgreSQL | MariaDB | SQLite |
|:---|:---|:---|:---|:---|
| 定位 | 通用 OLTP | 功能丰富的 OLTP | MySQL 分支 | 嵌入式 |
| 存储引擎 | InnoDB / MyISAM | 堆表 + WAL | XtraDB / Aria | 单文件 |
| MVCC 实现 | Undo Log 回滚段 | 多版本行 tuple | 同 MySQL | WAL |
| 索引类型 | B+Tree / Hash / Fulltext | B-Tree / Hash / GiST / GIN / BRIN | 同 MySQL | B-Tree |
| JSON 支持 | JSON / JSONB (有限) | JSONB (强大, 可索引) | 同 MySQL | JSON1 扩展 |
| 复制 | 异步/半同步 (binlog) | 物理(流) + 逻辑 | 同 MySQL | 无 |
| 扩展性 | 插件有限 | 丰富 (扩展、自定义类型/函数) | 插件 | 无 |
| GIS | 空间索引 (有限) | PostGIS (最强) | 空间索引 | 无 |
| 并发模型 | 线程 (每连接一线程) | 进程 (每连接一进程) | 线程 | 进程内 |
| 协议 | MySQL 协议 | PostgreSQL 协议 | MySQL 协议 | 库调用 |
| 许可证 | GPL / 商业 | PostgreSQL (类 BSD) | GPL | 公有领域 |
| 适用场景 | Web/电商/传统业务 | 复杂查询/GIS/数据仓库 | MySQL 替代 | 嵌入式/测试 |

### 1.2 选型建议

| 场景 | 推荐 | 原因 |
|:---|:---|:---|
| Web 应用 / 电商 | **MySQL** | 生态成熟，运维资源多，性能够用 |
| 复杂查询 / 数据分析 | **PostgreSQL** | 优化器强，窗口函数，CTE，并行查询 |
| GIS / 空间数据 | **PostgreSQL** | PostGIS 无可替代 |
| 高并发简单事务 | **MySQL** | 线程模型开销小 |
| JSON 文档存储 | **PostgreSQL** | JSONB 可索引，查询灵活 |
| 微服务 / 中小项目 | **MySQL / PostgreSQL** | 均可，看团队熟悉度 |
| 嵌入式 / 边缘计算 | **SQLite** | 零配置，单文件 |

### 1.3 NoSQL 选型

| 数据库 | 类型 | 适用场景 |
|:---|:---|:---|
| **Redis** | KV / 内存 | 缓存、会话、排行榜、消息队列 |
| **MongoDB** | 文档 | 内容管理、日志、灵活 Schema |
| **ClickHouse** | 列存 OLAP | 日志分析、BI 报表、时序数据 |
| **Elasticsearch** | 搜索/文档 | 全文搜索、日志检索 (ELK) |

### 2.2 InnoDB 存储引擎原理

```
InnoDB 内存架构:
┌──────────────────────────────────────────┐
│              Buffer Pool                  │
│  ┌──────────────────────────────────┐    │
│  │ 数据页 (16KB)                     │    │
│  │ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐│    │
│  │ │页 1  │ │页 2  │ │页 3  │ │ ...  ││    │
│  │ └─────┘ └─────┘ └─────┘ └─────┘│    │
│  │                                    │    │
│  │ LRU 链表 (最近最少使用淘汰)         │    │
│  │ ┌──────────────────────────┐      │    │
│  │ │  New 区 (5/8)  │ Old 区 (3/8)│      │    │
│  │ └──────────────────────────┘      │    │
│  └──────────────────────────────────┘    │
│                                          │
│  ┌──────────┐  ┌──────────┐             │
│  │Change    │  │Adaptive  │             │
│  │Buffer    │  │Hash Index│             │
│  │(异步写)  │  │(自动索引) │             │
│  └──────────┘  └──────────┘             │
└──────────────────────────────────────────┘

InnoDB 磁盘架构:
┌──────────────────────────────────────────┐
│  系统表空间 (ibdata1) 或 独立表空间       │
│  ┌────────────┐  ┌────────────────────┐ │
│  │数据字典     │  │ .ibd 文件 (每表一个)│ │
│  │undo log    │  │  ┌──────────────┐ │ │
│  │doublewrite │  │  │聚簇索引 (B+Tree)│ │ │
│  │buffer      │  │  │  叶子=数据行   │ │ │
│  └────────────┘  │  └──────────────┘ │ │
│                   │  ┌──────────────┐ │ │
│                   │  │二级索引 (B+Tree)│ │ │
│                   │  │  叶子=主键值  │ │ │
│                   │  └──────────────┘ │ │
│                   └────────────────────┘ │
│                                          │
│  ┌────────────┐  ┌────────────┐         │
│  │ Redo Log   │  │ Undo Log   │         │
│  │ (ib_logfile│  │ (ibdata 或 │         │
│  │  或 redo/) │  │  undo/)   │         │
│  └────────────┘  └────────────┘         │
└──────────────────────────────────────────┘
```

**InnoDB 关键机制：**

| 机制 | 说明 |
|:---|:---|
| **聚簇索引** | 数据按主键的 B+Tree 存储，叶子节点就是行数据。按主键查询最快。 |
| **二级索引** | 索引的叶子存主键值，需要回表查数据。索引太多影响写入。 |
| **MVCC** | 通过 undo log 实现多版本并发控制，读不加锁，写不阻塞读。 |
| **Redo Log** | WAL (Write-Ahead Logging)，先写日志再写数据页，保证崩溃恢复。 |
| **Undo Log** | 保存旧版本数据，用于事务回滚和 MVCC 快照读。 |
| **Buffer Pool** | 数据页缓存在内存，避免频繁磁盘 IO。生产建议设为物理内存 50-70%。 |
| **Change Buffer** | 对二级索引的修改先缓冲，合并到磁盘，减少随机 IO。 |
| **Doublewrite Buffer** | 写入前先写两遍，防止页撕裂 (partial page write) 导致数据损坏。 |
| **自适应哈希索引** | InnoDB 自动对热点数据建哈希索引，等值查询变 O(1)。 |

### 2.3 一条 SQL 的执行流程

```
SELECT * FROM users WHERE id = 1;

1. 客户端发送 SQL → 连接线程接收
   │
2. 查询缓存 (MySQL 8.0 已移除)
   │
3. 解析器: 词法分析 → 语法分析 → 生成解析树
   │
4. 预处理器: 检查表/列是否存在, 权限校验
   │
5. 优化器: 生成执行计划
   │  ├── 成本估算 (IO 成本 + CPU 成本)
   │  ├── 选择索引 (走哪个索引? 全表扫描?)
   │  ├── 选择连接顺序 (多表 JOIN)
   │  └── 生成执行计划
   │
6. 执行器: 调用存储引擎 API
   │  ├── 调用 InnoDB 的 index_read → 读 Buffer Pool
   │  ├── Buffer Pool 命中 → 直接返回
   │  └── Buffer Pool 未命中 → 从磁盘读页到内存
   │
7. 返回结果集给客户端

# 查看执行计划
EXPLAIN SELECT * FROM users WHERE id = 1;
EXPLAIN ANALYZE SELECT * FROM users WHERE id = 1;  -- MySQL 8.0+
```

### 2.6 主从复制

```
主从复制原理:

  Master                          Slave
  ┌────────┐    binlog      ┌─────────┐
  │ 客户端  │               │         │
  └───┬────┘               │  应用    │
      │ 写入                 │  读取    │
  ┌───▼────┐               └─────────┘
  │ InnoDB │                    ▲
  │ 数据页  │                    │ relay log
  └───┬────┘               ┌─────┴─────┐
      │                     │ SQL 线程   │ ← 回放 SQL
  ┌───▼────┐               │ (apply)   │
  │Redo Log│               └─────▲─────┘
  └───┬────┘                     │
      │                          │
  ┌───▼────┐  IO 线程      ┌─────┴─────┐
  │Binlog  │─────────────►│ IO 线程    │ ← 拉取 binlog
  │        │  TCP          │ (fetch)   │
  └────────┘               └───────────┘

  1. Master 写入 → 记录 binlog
  2. Slave IO 线程连接 Master, 请求 binlog
  3. Master 的 dump 线程发送 binlog
  4. Slave IO 线程写入 relay log
  5. Slave SQL 线程读取 relay log, 回放 SQL
```

```bash
# /etc/my.cnf.d/mysql-server.cnf 已配置:
#   server_id=1, log_bin=mysql-bin, gtid_mode=ON

EOF

# 查看主库状态
mysql -u root -p -e "SHOW MASTER STATUS;"
# +------------------+----------+--------------+------------------+------------------------------------------+
# | File             | Position | Binlog_Do_DB | Binlog_Ignore_DB | Executed_Gtid_Set                        |
# +------------------+----------+--------------+------------------+------------------------------------------+
# | mysql-bin.000003 |     1234 |              |                  | 3E4FA5A1-...:1-5                         |
# +------------------+----------+--------------+------------------+------------------------------------------+

# === 从库配置 ===
# /etc/my.cnf.d/mysql-server.cnf
#   server_id=2
#   log_bin=mysql-bin
#   gtid_mode=ON
#   enforce_gtid_consistency=ON
#   relay_log=relay-bin
#   read_only=ON              # 从库只读
#   super_read_only=ON

systemctl restart mysqld

# 配置复制 (GTID 方式, 推荐)
mysql -u root -p << 'EOF'
CHANGE REPLICATION SOURCE TO
    SOURCE_HOST='192.168.1.10',
    SOURCE_PORT=3306,
    SOURCE_USER='repl',
    SOURCE_PASSWORD='ReplPass2026!',

# 查看复制状态
mysql -u root -p -e "SHOW REPLICA STATUS\G"

#   Replica_IO_Running: Yes
#   Replica_SQL_Running: Yes
#   Seconds_Behind_Master: 0          ← 延迟秒数

#   Last_SQL_Error: (空)

# === 半同步复制 (降低数据丢失风险) ===
# 主库
INSTALL PLUGIN rpl_semi_sync_source SONAME 'semisync_source.so';
SET GLOBAL rpl_semi_sync_source_enabled = 1;
SET GLOBAL rpl_semi_sync_source_timeout = 3000;  # 3秒超时

# 从库
INSTALL PLUGIN rpl_semi_sync_replica SONAME 'semisync_replica.so';
SET GLOBAL rpl_semi_sync_replica_enabled = 1;
STOP REPLICA; START REPLICA;
```

### 2.7 备份与恢复

```bash
# === 逻辑备份 (mysqldump) ===
# 全量备份
mysqldump -u root -p --single-transaction --routines --triggers --events \
    --all-databases > /backup/full_$(date +%Y%m%d).sql

# 单库备份
mysqldump -u root -p --single-transaction mydb > /backup/mydb_$(date +%Y%m%d).sql

# 仅表结构
mysqldump -u root -p --no-data mydb > /backup/mydb_schema.sql

# 仅数据
mysqldump -u root -p --no-create-info mydb > /backup/mydb_data.sql

# 关键参数:
#   --single-transaction: InnoDB 一致性快照 (不锁表)
#   --routines: 包含存储过程/函数
#   --triggers: 包含触发器
#   --events: 包含事件
#   --master-data=2: 记录 binlog 位置 (注释中, 用于搭建从库)
#   --set-gtid-purged=ON: GTID 环境

# 恢复
mysql -u root -p < /backup/full_20260709.sql
mysql -u root -p mydb < /backup/mydb_20260709.sql

# === 物理备份 (Percona XtraBackup) ===
dnf install -y https://repo.percona.com/yum/percona-release-latest.noarch.rpm
percona-release setup ps80
dnf install -y percona-xtrabackup-80

# 全量备份
xtrabackup --backup --target-dir=/backup/xtrabackup/full \
    --user=root --password=xxx

# 准备 (apply log, 使备份一致)
xtrabackup --prepare --target-dir=/backup/xtrabackup/full

# 恢复 (需要先停止 MySQL)
systemctl stop mysqld
xtrabackup --copy-back --target-dir=/backup/xtrabackup/full
chown -R mysql:mysql /var/lib/mysql
systemctl start mysqld

# 增量备份
xtrabackup --backup --target-dir=/backup/inc1 \
    --incremental-basedir=/backup/xtrabackup/full \
    --user=root --password=xxx

# === binlog 恢复 (点恢复) ===
# 查看已用的 binlog
cat /backup/xtrabackup/full/xtrabackup_binlog_info
# mysql-bin.000003 1234

# 从指定位置重放 binlog
mysqlbinlog --start-position=1234 \
    --stop-datetime="2026-07-09 15:00:00" \
    /var/lib/mysql/mysql-bin.000003 \
    /var/lib/mysql/mysql-bin.000004 \
    | mysql -u root -p

# === 自动备份脚本 ===
cat > /opt/mysql-backup.sh << 'SCRIPT'
#!/bin/bash
set -euo pipefail
BACKUP_DIR=/backup/mysql
DATE=$(date +%Y%m%d_%H%M%S)
RETAIN_DAYS=7

mkdir -p "$BACKUP_DIR"

# 全量备份
mysqldump -u root -p"${MYSQL_ROOT_PASS}" --single-transaction \
    --routines --triggers --events --all-databases \
    --master-data=2 \
    | gzip > "$BACKUP_DIR/full_${DATE}.sql.gz"

# 清理旧备份
find "$BACKUP_DIR" -name "*.sql.gz" -mtime +${RETAIN_DAYS} -delete

echo "[$(date)] 备份完成: full_${DATE}.sql.gz"
SCRIPT
chmod +x /opt/mysql-backup.sh

# crontab
# 0 2 * * * /opt/mysql-backup.sh >> /var/log/mysql-backup.log 2>&1
```

### 2.8 性能调优

**MySQL vs PostgreSQL 架构差异：**

| 维度 | MySQL (InnoDB) | PostgreSQL |
|:---|:---|:---|
| 并发模型 | 每连接一线程 | 每连接一进程 (fork) |
| 数据存储 | 聚簇索引 (数据在索引叶子) | 堆表 (数据独立, 索引指向行号) |
| 页大小 | 16KB | 8KB |
| WAL | Redo Log (固定大小, 循环写) | WAL (不断增长, 归档) |
| MVCC | Undo Log 回滚段 | 行内多版本 (xmin/xmax) |
| 死数据清理 | Purge 线程自动清理 | Autovacuum 清理死 tuple |
| 索引 | B+Tree | B-Tree / Hash / GIN / GiST / BRIN |
| 复制 | binlog (逻辑/混合) | 流复制 (物理) + 逻辑复制 |

### 3.2 MVCC 与 VACUUM 原理

```
PostgreSQL MVCC (多版本并发控制):

  原始行:
  ┌──────────────────────────────────┐
  │ id=1  name='Alice'  age=25      │
  │ xmin=100 (创建事务ID)             │
  │ xmax=空 (未删除)                  │
  └──────────────────────────────────┘

  UPDATE name='Bob' 后:
  ┌──────────────────────────────────┐
  │ id=1  name='Alice'  age=25      │ ← 旧版本 (死行)
  │ xmin=100  xmax=200 (被事务200删除)│   VACUUM 清理
  └──────────────────────────────────┘
  ┌──────────────────────────────────┐
  │ id=1  name='Bob'    age=25      │ ← 新版本
  │ xmin=200 (创建事务ID)             │
  │ xmax=空                           │
  └──────────────────────────────────┘

  特点:
  - UPDATE = DELETE 旧行 + INSERT 新行
  - 旧行不会立即删除, 留给其他事务的快照读
  - 旧行成为 "死行" (dead tuple)
  - VACUUM 负责回收死行空间

  VACUUM:
  - VACUUM: 标记死行空间为可重用 (不归还 OS)
  - VACUUM FULL: 重组表, 归还空间给 OS (锁表!)
  - Autovacuum: 自动 VACUUM (生产必须开启)

  MySQL 的 MVCC 不同:
  - InnoDB 在 Undo Log 中保存旧版本
  - 旧版本不占表空间, Purge 线程自动清理
  - 不需要 VACUUM
```

### 3.5 流复制（主从）

```
PostgreSQL 流复制原理:

  Primary                           Standby
  ┌────────┐                   ┌─────────┐
  │ 客户端  │                   │  只读    │
  └───┬────┘                   │  查询    │
      │ 写入                     └────▲────┘
  ┌───▼────┐                        │
  │ 数据页  │                   ┌────┴────┐
  └───┬────┘                   │ 恢复进程 │ ← 应用 WAL
      │                         └────▲────┘
  ┌───▼────┐  WAL stream        ┌────┴────┐
  │ WAL    │───────────────────►│WAL接收  │ ← 接收 WAL
  │        │   TCP              │进程     │
  └────────┘                   └─────────┘

  特点:
  - 物理复制: 块级别, 完全一致
  - 异步/同步: 可配置同步级别
  - 级联复制: 从库可以从从库复制
  - 热备: 从库可读 (Hot Standby)
```

```bash
# === 主库配置 ===
# postgresql.conf 已配置:
#   wal_level=replica, max_wal_senders=10, wal_keep_size=1GB

# 创建复制用户
sudo -u postgres psql << 'EOF'
CREATE ROLE repl WITH REPLICATION LOGIN PASSWORD 'ReplPass2026!';
EOF

# pg_hba.conf 添加:
# host replication repl 192.168.1.0/24 scram-sha-256

systemctl restart postgresql-16

# === 从库配置 ===
# 停止从库 PostgreSQL
systemctl stop postgresql-16

# 清空从库数据目录
mv /var/lib/pgsql/16/data /var/lib/pgsql/16/data.bak
mkdir /var/lib/pgsql/16/data
chown postgres:postgres /var/lib/pgsql/16/data
chmod 700 /var/lib/pgsql/16/data

# 使用 pg_basebackup 从主库同步
sudo -u postgres /usr/pgsql-16/bin/pg_basebackup \
    -h 192.168.1.10 \
    -U repl \
    -D /var/lib/pgsql/16/data \
    -Fp -Xs -P -R
# -Fp: plain 格式
# -Xs: 流式传输 WAL
# -P: 显示进度
# -R: 自动创建 standby.signal + 配置 primary_conninfo

# 配置从库参数
cat >> /var/lib/pgsql/16/data/postgresql.conf << 'EOF'

# === Standby 设置 ===
hot_standby = on                     # 热备 (可读)
hot_standby_feedback = on            # 向主库反馈 (防从库清理过早)
EOF

# 启动从库
systemctl start postgresql-16

# === 验证 ===
# 主库查看复制状态
sudo -u postgres psql << 'EOF'
SELECT application_name, client_addr, state, sync_state,
       sent_lsn, write_lsn, flush_lsn, replay_lsn,

EOF

# 从库查看状态
sudo -u postgres psql << 'EOF'
SELECT pg_is_in_recovery();          -- true = 从库
SELECT now() - pg_last_xact_replay_timestamp() AS replication_lag;
EOF

# === 同步复制 (可选, 零数据丢失) ===
# 主库 postgresql.conf:
# synchronous_standby_names = 'FIRST 1 (standby1)'
# 或: synchronous_standby_names = 'ANY 1 (*)'
```

### 3.7 性能调优

```sql
-- 查看数据库大小
SELECT pg_size_pretty(pg_database_size('mydb'));

-- 查看表大小 (含索引)
SELECT schemaname, relname,
       pg_size_pretty(pg_total_relation_size(relid)) AS total_size,
       pg_size_pretty(pg_relation_size(relid)) AS table_size,
       n_live_tup AS row_count
FROM pg_stat_user_tables
ORDER BY pg_total_relation_size(relid) DESC
LIMIT 20;

-- 查看索引大小和使用率
SELECT schemaname, relname, indexrelname,
       pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
       idx_scan AS scan_count,
       idx_tup_read, idx_tup_fetch
FROM pg_stat_user_indexes
ORDER BY pg_relation_size(indexrelid) DESC;

-- 查看死行 (需要 VACUUM 的表)
SELECT relname, n_live_tup, n_dead_tup,
       ROUND(n_dead_tup::numeric / NULLIF(n_live_tup, 0) * 100, 2) AS dead_pct,
       last_autovacuum, last_autoanalyze
FROM pg_stat_user_tables
WHERE n_dead_tup > 0
ORDER BY n_dead_tup DESC;

-- 手动 VACUUM (不锁表)
VACUUM (VERBOSE, ANALYZE) mytable;

-- VACUUM FULL (锁表, 归还空间)
VACUUM FULL mytable;

-- 查看执行计划
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'test@example.com';
-- 关键:
--   Seq Scan = 全表扫描 (慢)
--   Index Scan = 索引扫描
--   Index Only Scan = 覆盖索引 (最快)
--   Bitmap Index Scan = 位图索引 (多条件)

-- 查看当前连接
SELECT pid, usename, datname, client_addr, state,
       query_start, state_change, query
FROM pg_stat_activity
WHERE state != 'idle';

-- 终止长时间查询
SELECT pg_terminate_backend(pid) FROM pg_stat_activity
WHERE state = 'active' AND query_start < now() - interval '10 minutes';

-- 查看锁等待
SELECT pid, mode, granted, query
FROM pg_locks l
JOIN pg_stat_activity a USING (pid)
WHERE NOT granted;
```

```
│  │  └────────┘ └────────┘ └────────┘    │  │
│  └───────────────────────────────────────┘  │
│                        │                    │
│  ┌─────────────────────▼─────────────────┐  │
│  │           持久化                        │  │
│  │  ┌──────────┐    ┌──────────────┐    │  │
│  │  │RDB       │    │AOF           │    │  │
│  │  │(快照)     │    │(追加日志)     │    │  │
│  │  │fork+bgsave│   │appendfsync   │    │  │
│  │  └──────────┘    └──────────────┘    │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘

Redis 为什么快:
1. 纯内存操作 (无磁盘 IO, 除非持久化)
2. 单线程 (无锁竞争, 无上下文切换)
3. epoll 事件驱动 (单线程处理万级连接)
4. 高效数据结构 (SDS, skiplist, ziplist, intset)
5. 6.0+ IO 多线程 (仅网络读写并行, 命令仍单线程)
```

### 4.3 生产配置

```bash
# /etc/redis/redis.conf

# === 网络 ===
bind 0.0.0.0                          # 监听地址
protected-mode yes                    # 保护模式 (无密码时只允许本地)
port 6379
tcp-backlog 511                       # TCP 队列
tcp-keepalive 300                     # TCP 心跳
timeout 0                             # 客户端空闲超时 (0=不超时)

# === 通用 ===
daemonize no                          # systemd 管理时用 no
supervised systemd
pidfile /var/run/redis/redis-server.pid
loglevel notice                       # debug | verbose | notice | warning
logfile /var/log/redis/redis-server.log
databases 16                          # 逻辑库数量

# === 内存 ===
maxmemory 4gb                         # 最大内存
maxmemory-policy allkeys-lru          # 淘汰策略:
  # noeviction:        满了拒绝写入
  # allkeys-lru:       全部键 LRU 淘汰 (缓存推荐)
  # allkeys-lfu:       全部键 LFU 淘弃
  # volatile-lru:      仅设了过期的键 LRU
  # volatile-ttl:      优先淘汰快过期的
maxmemory-samples 5                   # 淘汰采样数 (越大越精确)

# === RDB 持久化 (快照) ===
save 900 1                            # 900秒内 1 次修改 → 快照
save 300 10                           # 300秒内 10 次修改 → 快照
save 60 10000                         # 60秒内 10000 次修改 → 快照
# save ""                              # 禁用 RDB

stop-writes-on-bgsave-error yes       # bgsave 失败时拒绝写入
rdbcompression yes                    # RDB 压缩
rdbchecksum yes                       # RDB 校验
dbfilename dump.rdb
dir /var/lib/redis

# RDB 优缺点:
# + 文件小, 恢复快
# + fork 子进程, 不影响主线程
# - 定期快照, 可能丢失最后一次快照后的数据
# - fork 大内存时可能短暂卡顿

# === AOF 持久化 (追加日志) ===
appendonly yes                        # 启用 AOF
appendfilename "appendonly.aof"
appendfsync everysec                  # 每秒刷盘 (折中方案)
  # always:    每次写入都刷盘 (最安全, 最慢)
  # everysec:  每秒刷盘 (推荐, 最多丢 1 秒)
  # no:        由 OS 决定 (最快, 可能丢更多)
no-appendfsync-on-rewrite no          # rewrite 时不 fsync
auto-aof-rewrite-percentage 100       # AOF 大小增长 100% 时重写
auto-aof-rewrite-min-size 64mb        # AOF 重写最小大小
aof-load-truncated yes                # 加载截断的 AOF

# AOF 优缺点:
# + 数据安全性高 (everysec 最多丢 1 秒)
# + 日志格式可读
# - 文件大, 恢复慢
# - 写入有额外开销

# === 混合持久化 (RDB + AOF, 推荐) ===
aof-use-rdb-preamble yes              # AOF 重写时前半段用 RDB 格式

# === 安全 ===
requirepass YourRedisPassword2026     # 密码 (生产必须设置)
rename-command FLUSHALL ""            # 禁用危险命令
rename-command FLUSHDB ""
rename-command CONFIG "CONFIG_a1b2c3" # 重命名 (加密码后缀)

# === 慢查询 ===
slowlog-log-slower-than 10000         # 慢查询阈值 (微秒, 10ms)
slowlog-max-len 128                   # 慢查询日志最大条数

# === 客户端 ===
maxclients 10000                      # 最大客户端数

# === 惰性释放 (大 KEY 删除优化) ===
lazyfree-lazy-eviction yes            # 异步淘汰
lazyfree-lazy-expire yes              # 异步过期
lazyfree-lazy-server-del yes          # 异步 DEL
lazyfree-lazy-user-del yes            # 用户 DEL 异步

# === IO 多线程 (6.0+, 仅网络 IO) ===
io-threads 4                          # IO 线程数 (CPU 核数 / 2)
io-threads-do-reads yes               # IO 线程读

# === 复制 (从库配置) ===
# replicaof 192.168.1.10 6379          # 主库地址
# masterauth YourRedisPassword2026     # 主库密码
# replica-read-only yes                # 从库只读
# replica-priority 100                 # 优先级 (0=不参与选主)
```

### 4.4 Redis Cluster vs Sentinel

```
方案一: Sentinel (主从 + 哨兵自动故障转移)
┌────────┐     ┌────────┐     ┌────────┐
│Master  │────►│Replica │     │Replica │
│(读写)   │     │(只读)   │     │(只读)   │
└───┬────┘     └────────┘     └────────┘
    │
┌───▼────────────────────────────────────┐
│           Sentinel 集群 (3个)            │
│  监控 Master 健康 → 故障时自动选新主     │
│  客户端连 Sentinel 获取 Master 地址      │
└────────────────────────────────────────┘

方案二: Cluster (分片 + 自动故障转移)
┌─────────────────────────────────────────┐
│              Redis Cluster               │
│                                         │
│  Slot 0-5460      Slot 5461-10922      Slot 10923-16383
│  ┌──────────┐     ┌──────────┐     ┌──────────┐
│  │ Master A │     │ Master B │     │ Master C │
│  │ (读写)    │     │ (读写)    │     │ (读写)    │
│  └────┬─────┘     └────┬─────┘     └────┬─────┘
│       │                  │                  │
│  ┌────▼─────┐     ┌────▼─────┐     ┌────▼─────┐
│  │Replica A │     │Replica B │     │Replica C │
│  │(只读+备)  │     │(只读+备)  │     │(只读+备)  │
│  └──────────┘     └──────────┘     └──────────┘
│
│  16384 个 hash slot 分布在 3 个主节点
│  key → CRC16(key) % 16384 → 落到某个 slot
│  每个 slot 对应一个主节点
│  主节点挂了 → 对应 replica 接管
└─────────────────────────────────────────┘

对比:
  Sentinel: 适合数据量小、全量在单机的场景
  Cluster:  适合数据量大、需要水平扩展的场景
```

```bash
# === Redis Cluster 搭建 (6 节点: 3主3从) ===

# 每个节点配置: /etc/redis/redis-7000.conf
port 7000
cluster-enabled yes
cluster-config-file nodes-7000.conf
cluster-node-timeout 5000
appendonly yes
daemonize yes
bind 0.0.0.0
requirepass YourRedisPassword2026
masterauth YourRedisPassword2026

# 启动 6 个节点 (7000-7005)
for port in 7000 7001 7002 7003 7004 7005; do
    sed "s/7000/$port/g" /etc/redis/redis-7000.conf > /etc/redis/redis-$port.conf
    redis-server /etc/redis/redis-$port.conf
done

# 创建集群
redis-cli --cluster create \
    127.0.0.1:7000 127.0.0.1:7001 127.0.0.1:7002 \
    127.0.0.1:7003 127.0.0.1:7004 127.0.0.1:7005 \
    --cluster-replicas 1 \
    -a YourRedisPassword2026

# 验证
redis-cli -p 7000 -a YourRedisPassword2026 cluster info
redis-cli -p 7000 -a YourRedisPassword2026 cluster nodes
```

## 五、数据库对比总结

### 5.1 ACID 事务实现对比

| 维度 | MySQL (InnoDB) | PostgreSQL | Redis |
|:---|:---|:---|:---|
| 隔离级别 | RR (默认) / RC / RU / Serializable | RC (默认) / RR / Serializable | 单线程无并发问题 |
| 原子性 | Undo Log 回滚 | Undo Log (事务日志) | 单命令原子 |
| 持久性 | Redo Log (WAL) | WAL | RDB / AOF |
| MVCC | Undo Log 多版本 | 行内多版本 (xmin/xmax) | 无 |
| 锁粒度 | 行锁 / 间隙锁 | 行锁 / 表锁 | 单线程无锁 |
| 死锁检测 | 自动检测+回滚 | 自动检测+回滚 | 无 |

### 5.2 复制对比

| 维度 | MySQL | PostgreSQL | Redis |
|:---|:---|:---|:---|
| 复制方式 | binlog (逻辑/混合) | 流复制 (物理) | RDB/AOF (物理) |
| GTID | ✅ | ✅ (LSN) | ❌ |
| 同步复制 | 半同步 | ✅ (可配置级别) | ✅ (WAIT) |
| 级联复制 | ✅ | ✅ | ✅ |
| 自动故障转移 | MHA / Orchestrator | Patroni / repmgr | Sentinel / Cluster |
| 逻辑复制 | ✅ (binlog) | ✅ (Publication/Subscription) | ❌ |

### 5.3 备份对比

| 维度 | MySQL | PostgreSQL | Redis |
|:---|:---|:---|:---|
| 逻辑备份 | mysqldump | pg_dump | ❌ (无原生) |
| 物理备份 | XtraBackup | pgBackRest / pg_basebackup | cp RDB 文件 |
| 增量备份 | XtraBackup | pgBackRest | ❌ |
| 点恢复 | binlog 回放 | WAL 归档 | ❌ (AOF 不支持) |
| 在线备份 | ✅ (--single-transaction) | ✅ (pg_dump / pg_basebackup) | ✅ (BGSAVE) |
