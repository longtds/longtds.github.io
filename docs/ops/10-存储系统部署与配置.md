## 一、存储类型与选型

### 1.1 三大存储类型

```
┌─────────────────────────────────────────────────────────┐
│                      存储类型                             │
│                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐    │
│  │  块存储      │  │  文件存储    │  │  对象存储    │    │
│  │  Block       │  │  File        │  │  Object      │    │
│  │              │  │              │  │              │    │
│  │ 裸磁盘/卷     │  │ 目录/文件     │  │ Bucket/Key   │    │
│  │ 无文件系统    │  │ 有文件系统    │  │ 无文件系统    │    │
│  │ 块设备 /dev/ │  │ 挂载点 /mnt/ │  │ HTTP S3 API  │    │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘    │
│         │                │                │             │
│    ┌────▼────┐      ┌────▼────┐      ┌────▼────┐      │
│    │数据库    │      │应用共享  │      │备份/归档 │      │
│    │虚拟机盘  │      │NFS 挂载  │      │图片/视频 │      │
│    │K8s PV   │      │NAS       │      │大数据    │      │
│    └─────────┘      └─────────┘      └─────────┘      │
└─────────────────────────────────────────────────────────┘

区别:
  块存储:  最底层, 提供裸块设备, 客户端自己格式化文件系统
           性能最好, 延迟最低, 适合数据库/虚拟机
  文件存储: 带文件系统, 支持目录/权限/锁, 可多客户端共享
           易用, 适合应用共享/用户目录
  对象存储: 扁平命名空间 (Bucket+Key), HTTP API 访问
           无限扩展, 适合海量小文件/备份/CDN 源站
```

### 1.2 主流开源方案对比

| 维度 | Ceph | MinIO | GlusterFS | LVM+iSCSI | Longhorn | SeaweedFS |
|:---|:---|:---|:---|:---|:---|:---|
| 类型 | 块+文件+对象 | 对象 | 文件 | 块 | 块 | 对象+文件 |
| 架构 | 统一存储 (RADOS) | 分布式 | 无中心 | 本地+网络 | K8s 原生 | 主从+Filer |
| 协议 | RBD/CephFS/S3 | S3 | POSIX/FUSE | iSCSI | iSCSI/NFS | S3/FUSE |
| 性能 | 中高 | 高 | 中 | ⭐ 最高 | 中 | 高 |
| 扩展性 | PB 级 | PB 级 | PB 级 | TB 级 | TB 级 | PB 级 |
| 数据冗余 | 副本/EC | 副本/EC | 副本 | 无 (单机) | 副本 | 副本/EC |
| 一致性 | 强一致 | 强一致 | 最终一致 | 强一致 | 强一致 | 强一致 |
| 运维复杂度 | ⭐ 高 | 低 | 中 | 低 | 低 | 中 |
| K8s 集成 | Rook | Operator | ❌ | CSI | ⭐ 原生 | CSI |
| 适用场景 | 统一存储平台 | S3 对象存储 | 文件共享 | 数据库本地 | K8s 持久卷 | 海量小文件 |

### 1.3 选型建议

| 场景 | 推荐 | 原因 |
|:---|:---|:---|
| K8s 持久卷 (CSI) | **Ceph (Rook)** / **Longhorn** | 云原生，动态供给 |
| 数据库存储 | **LVM+iSCSI** / **Ceph RBD** | 块设备，低延迟 |
| S3 兼容对象存储 | **MinIO** | 部署简单，性能好，兼容性强 |
| 统一存储平台 | **Ceph** | 一套集群提供块+文件+对象 |
| 文件共享 (多机挂载) | **NFS** / **CephFS** | 简单易用 |
| 海量小文件 | **SeaweedFS** | 专为小文件优化 |
| 备份/归档 | **MinIO** | S3 API，生命周期管理 |
| 边缘/轻量 | **SeaweedFS** / **MinIO** | 单二进制，资源占用低 |

## 二、LVM + iSCSI（块存储）

### 2.1 LVM 原理

```
LVM (Logical Volume Manager) 逻辑卷管理:

  物理磁盘               物理卷              卷组              逻辑卷
  ┌────────┐           ┌────────┐         ┌──────────────────────┐
  │ /dev/  │  pvcreate │  PV    │  vgcreate│  VG: vg_data         │
  │  sdb   │──────────►│        │─────────►│  ┌────────────────┐  │
  └────────┘           │        │          │  │ LV: lv_db       │  │
  ┌────────┐           │        │          │  │ 100GB (xfs)     │  │
  │ /dev/  │  pvcreate │        │          │  └────────────────┘  │
  │  sdc   │──────────►│        │          │  ┌────────────────┐  │
  └────────┘           └────────┘          │  │ LV: lv_web      │  │
                                           │  │ 50GB (ext4)     │  │
                                           │  └────────────────┘  │
                                           │  剩余空间可动态分配    │
                                           └──────────────────────┘

优势:
  - 容量可动态调整 (lvextend/lvreduce)
  - 跨磁盘 (多个 PV 合并为一个 VG)
  - 快照 (LVM Snapshot)
  - 条带化 (striping, 性能提升)
  - 镜像 (mirroring, 冗余)
```

### 2.2 LVM 配置

```bash
# === 创建 LVM ===
# 1. 创建物理卷 (PV)
pvcreate /dev/sdb /dev/sdc
pvdisplay

# 2. 创建卷组 (VG)
vgcreate vg_data /dev/sdb /dev/sdc
vgdisplay

# 3. 创建逻辑卷 (LV)
lvcreate -L 100G -n lv_db vg_data          # 指定大小
lvcreate -l 100%FREE -n lv_web vg_data      # 使用剩余全部空间
lvdisplay

# 4. 格式化
mkfs.xfs /dev/vg_data/lv_db
mkfs.ext4 /dev/vg_data/lv_web

# 5. 挂载
mkdir -p /data/db /data/web
mount /dev/vg_data/lv_db /data/db
mount /dev/vg_data/lv_web /data/web

# /etc/fstab
/dev/vg_data/lv_db  /data/db   xfs  defaults  0 0
/dev/vg_data/lv_web /data/web  ext4 defaults  0 0

# === 动态扩容 ===
# 扩展 LV
lvextend -L +50G /dev/vg_data/lv_db         # 增加 50G
lvextend -l +100%FREE /dev/vg_data/lv_db     # 用完剩余空间

# 扩展文件系统
xfs_growfs /data/db                          # xfs
resize2fs /dev/vg_data/lv_web                # ext4

# === LVM 快照 (备份用) ===
# 创建快照 (大小为源 LV 的 10-20%)
lvcreate -L 10G -s -n lv_db_snap /dev/vg_data/lv_db

# 挂载快照
mount -o ro /dev/vg_data/lv_db_snap /mnt/snap

# 从快照备份
tar -czf /backup/db_snap.tar.gz -C /mnt/snap .

# 删除快照
umount /mnt/snap
lvremove -f /dev/vg_data/lv_db_snap

# === LVM 条带化 (性能提升) ===
# 跨多个磁盘条带化 (类似 RAID 0)
lvcreate -i 2 -I 64k -L 100G -n lv_fast vg_data
# -i 2: 2 个条带 (跨 2 个 PV)
# -I 64k: 条带大小 64KB

# === VG 扩展 (加新磁盘) ===
pvcreate /dev/sdd
vgextend vg_data /dev/sdd
```

### 2.3 iSCSI 配置（网络块存储）

```bash
# === iSCSI Target (服务端, 提供磁盘) ===
dnf install -y targetcli

# 创建块设备后端
targetcli
/> backstores/block create name=disk0 dev=/dev/vg_data/lv_db
/> backstores/block create name=disk1 dev=/dev/sdc

# 创建 iSCSI target
/> iscsi/ create iqn.2026-07.com.example:storage.target

# 绑定 LUN (逻辑单元)
/> iscsi/iqn.2026-07.com.example:storage.target/tpg1/luns create /backstores/block/disk0
/> iscsi/iqn.2026-07.com.example:storage.target/tpg1/luns create /backstores/block/disk1

# 创建 ACL (允许哪些客户端连接)
/> iscsi/iqn.2026-07.com.example:storage.target/tpg1/acls create iqn.2026-07.com.example:client1

# 设置门户 (监听地址)
/> iscsi/iqn.2026-07.com.example:storage.target/tpg1/portals create 0.0.0.0 3260

# 保存
/> saveconfig
/> exit

# 启动
systemctl enable --now target

# 防火墙
firewall-cmd --permanent --add-port=3260/tcp
firewall-cmd --reload

# === iSCSI Initiator (客户端, 使用磁盘) ===
dnf install -y iscsi-initiator-utils

# 配置 Initiator 名称
cat > /etc/iscsi/initiatorname.iscsi << 'EOF'
InitiatorName=iqn.2026-07.com.example:client1
EOF

# 发现 Target
iscsiadm -m discovery -t st -p 192.168.1.10
# 输出: 192.168.1.10:3260,1 iqn.2026-07.com.example:storage.target

# 登录 (连接)
iscsiadm -m node -T iqn.2026-07.com.example:storage.target \
    -p 192.168.1.10 -l

# 设置开机自动登录
iscsiadm -m node -T iqn.2026-07.com.example:storage.target \
    -p 192.168.1.10 --op update -n node.startup -v automatic

systemctl enable --now iscsid

# 验证: 应该看到新的块设备
lsblk
# sdb 8:16 0 100G 0 disk  ← iSCSI 磁盘

# 格式化并挂载
mkfs.xfs /dev/sdb
mount /dev/sdb /mnt/iscsi-disk

# 查看会话
iscsiadm -m session
```

## 三、Ceph（统一存储）

### 3.1 架构原理

```
Ceph 统一存储架构:

                        ┌──────────────────┐
                        │   Ceph Client     │
                        └────────┬─────────┘
                                 │ librados
                    ┌────────────┼────────────┐
                    │            │            │
              ┌─────▼─────┐ ┌───▼────┐ ┌────▼─────┐
              │ RBD        │ │CephFS  │ │ RGW      │
              │ (块设备)    │ │(文件)  │ │ (对象S3) │
              └─────┬─────┘ └───┬────┘ └────┬─────┘
                    │            │            │
                    └────────────┼────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │       RADOS             │
                    │  (Reliable Autonomic    │
                    │   Distributed Object    │
                    │   Store)                │
                    │                         │
                    │  ┌─────┐ ┌─────┐ ┌─────┐│
                    │  │OSD 1│ │OSD 2│ │OSD 3││  ← 数据存储节点
                    │  └─────┘ └─────┘ └─────┘│
                    └────────────┬────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │     Monitor (MON)        │
                    │  集群地图/状态/认证        │
                    │  (3 或 5 个, 奇数)        │
                    └─────────────────────────┘

CRUSH 算法 (核心):
  传统: 中心化元数据服务器 → 瓶颈
  Ceph: CRUSH 算法直接计算数据位置 → 无中心

  写入流程:
    1. 客户端计算 PG (Placement Group) → object → PG ID
    2. CRUSH(PG ID) → 得到 OSD 列表 [OSD-1, OSD-3, OSD-5]
    3. 客户端直接写入主 OSD (OSD-1)
    4. OSD-1 同步到副本 OSD-3, OSD-5
    5. 全部确认后返回成功

数据分布:
  Object → PG → OSD
  ┌──────────┐    ┌─────────┐    ┌─────┐
  │ Object A │───►│ PG 1.3  │───►│OSD 1│ (主)
  │ Object B │───►│ PG 2.7  │───►│OSD 3│ (主)
  │ Object C │───►│ PG 1.3  │───►│OSD 5│ (主)
  └──────────┘    └─────────┘    └─────┘
                  每个 PG 映射到
                  3 个 OSD (副本)
```

**Ceph 三种存储接口：**

| 接口 | 全称 | 协议 | 场景 |
|:---|:---|:---|:---|
| **RBD** | RADOS Block Device | librbd / iSCSI | 虚拟机磁盘, K8s PV, 数据库 |
| **CephFS** | Ceph File System | POSIX / NFS | 文件共享, NFS 替代 |
| **RGW** | RADOS Gateway | S3 / Swift | 对象存储, 备份, CDN |

### 3.2 部署（cephadm）

```bash

# 至少 3 节点, 每节点:
#   - 系统盘 + 数据盘 (裸盘, 未格式化)
#   - Python 3
#   - Docker / Podman
#   - NTP 时间同步
#   - 节点间 SSH 免密

# === 在第一个节点 (monitor) 上操作 ===

# 安装 cephadm
dnf install -y epel-release
dnf install -y python3 docker
curl --silent --remote-name \
    https://raw.githubusercontent.com/ceph/ceph/quincy/src/cephadm/cephadm
chmod +x cephadm
mv cephadm /usr/local/bin/

# 添加 Ceph 仓库
cephadm add-repo --release quincy
dnf install -y cephadm ceph-common

# 引导集群 (第一个节点成为 Monitor)
cephadm bootstrap \
    --mon-ip 192.168.1.10 \
    --initial-dashboard-admin-password CephAdmin2026

# 输出会包含:
# - Ceph Dashboard URL: https://192.168.1.10:8443
# - 用户名: admin
# - 告知如何添加其他节点
```

### 3.3 Ceph RBD（块存储）

```bash
# === 创建 RBD 池 ===
ceph osd pool create rbd_pool 128
rbd pool init rbd_pool

# 设置副本数 (默认 3)
ceph osd pool set rbd_pool size 3
ceph osd pool set rbd_pool min_size 2

# === 创建块设备镜像 ===
rbd create rbd_pool/vm-disk-01 --size 100G
rbd create rbd_pool/db-data --size 500G --image-feature layering

# 查看镜像
rbd ls rbd_pool
rbd info rbd_pool/vm-disk-01

# === 客户端映射 (Linux) ===
# 安装客户端
dnf install -y ceph-common

# 复制配置和密钥 (从 Ceph 节点)
scp root@192.168.1.10:/etc/ceph/ceph.conf /etc/ceph/
scp root@192.168.1.10:/etc/ceph/ceph.client.admin.keyring /etc/ceph/

# 禁用不需要的 image features (兼容性)
rbd feature disable rbd_pool/vm-disk-01 object-map fast-diff deep-flatten

# 映射 RBD 到本地块设备
rbd map rbd_pool/vm-disk-01
# /dev/rbd0

# 格式化
mkfs.xfs /dev/rbd0

# 挂载
mount /dev/rbd0 /mnt/rbd-disk

# 开机自动映射
cat >> /etc/ceph/rbdmap << 'EOF'
rbd_pool/vm-disk-01 id=admin,keyring=/etc/ceph/ceph.client.admin.keyring
EOF
systemctl enable rbdmap

# /etc/fstab
/dev/rbd0  /mnt/rbd-disk  xfs  defaults,_netdev  0 0

# === RBD 快照 ===
rbd snap create rbd_pool/vm-disk-01@snap-20260709
rbd snap ls rbd_pool/vm-disk-01

# 从快照恢复 (克隆)
rbd snap protect rbd_pool/vm-disk-01@snap-20260709
rbd clone rbd_pool/vm-disk-01@snap-20260709 rbd_pool/vm-disk-01-restore

# === RBD 扩容 ===
rbd resize rbd_pool/vm-disk-01 --size 200G
xfs_growfs /mnt/rbd-disk
```

### 3.4 CephFS（文件存储）

```bash
# === 创建 CephFS ===
# 创建元数据池和数据池
ceph osd pool create cephfs_metadata 64
ceph osd pool create cephfs_data 128

# 创建文件系统
ceph fs new myfs cephfs_metadata cephfs_data

# 查看
ceph fs ls
ceph fs status myfs

# === 部署 MDS (Metadata Server) ===
ceph orch apply mds myfs --placement="3 node-1 node-2 node-3"

# === 客户端挂载 ===
# 方式一: 内核挂载 (推荐)
mkdir -p /mnt/cephfs

# 获取 admin 密钥
ceph auth get-key client.admin | tee /etc/ceph/admin.secret

mount -t ceph 192.168.1.10:6789:/ /mnt/cephfs -o name=admin,secretfile=/etc/ceph/admin.secret

# /etc/fstab
192.168.1.10:6789:/  /mnt/cephfs  ceph  name=admin,secretfile=/etc/ceph/admin.secret,_netdev,noatime  0 0

# 方式二: FUSE 挂载
dnf install -y ceph-fuse
ceph-fuse /mnt/cephfs

# === 创建独立用户 (最小权限) ===
ceph fs subvolumegroup create myfs share
ceph fs subvolume create myfs share/app-data --size 100000000000  # 100GB

# 获取挂载信息
ceph fs subvolume getpath myfs share/app-data
# /volumes/share/app-data/xxxx

# 创建客户端
ceph auth get-or-create client.app-user \
    mon 'allow r' \
    mds 'allow rws path=/volumes/share/app-data' \
    osd 'allow rw pool=cephfs_data' \
    -o /etc/ceph/ceph.client.app-user.keyring
```

### 3.6 生产调优

```bash
# === PG 数量调优 ===
# PG (Placement Group) 数量影响数据分布
# 公式: PG数 = (OSD数 × 100) / 副本数 / 池数
# 例: 30 OSD, 3 副本, 2 池 → PG = (30 × 100) / 3 / 2 = 500 → 取 512

ceph osd pool set rbd_pool pg_num 512
ceph osd pool set rbd_pool pgp_num 512

# === 纠删码 (Erasure Coding, 省磁盘) ===
# 副本模式: 3 副本 → 100% 额外开销
# 纠删码: 4+2 → 50% 额外开销, 可丢 2 块盘

ceph osd pool create ec_pool 128 erasure
ceph osd erasure-code-profile set my-profile k=4 m=2
ceph osd pool set ec_pool erasure_code_profile my-profile
```

## 四、MinIO（对象存储）

### 4.1 架构原理

```
MinIO 分布式架构:

  ┌──────────────────────────────────────────────┐
  │              MinIO 集群                       │
  │                                              │
  │  Server 1            Server 2                │
  │  ┌─────────────┐     ┌─────────────┐        │
  │  │ /data1 (Set1)│     │ /data1 (Set1)│        │
  │  │ /data2 (Set2)│     │ /data2 (Set2)│        │
  │  └─────────────┘     └─────────────┘        │
  │                                              │
  │  Server 3            Server 4                │
  │  ┌─────────────┐     ┌─────────────┐        │
  │  │ /data1 (Set1)│     │ /data1 (Set1)│        │
  │  │ /data2 (Set2)│     │ /data2 (Set2)│        │
  │  └─────────────┘     └─────────────┘        │
  │                                              │
  │  每个 Set 包含 4 个节点 (最小 4 节点)          │
  │  数据写入时: EC 4+2 或 副本模式                 │
  │  一个节点挂了, 数据仍可读                       │
  └──────────────────────────────────────────────┘

MinIO 特点:
  - S3 兼容 (API 完全兼容 AWS S3)
  - 单二进制 (Go 编写, 无依赖)
  - 纠删码 (Erasure Coding) 默认, 数据安全
  - 服务端加密 (SSE-S3/SSE-KMS/SSE-C)
  - 版本控制 (Versioning)
  - 生命周期管理 (Lifecycle)
  - Bucket 通知 (Webhook/Kafka/MQTT)
  - 对象锁定 (Object Lock, WORM)
  - 监控内置 Prometheus 指标

纠删码原理:
  原始数据 → 分成 N 份 → 生成 M 份校验 → 共 N+M 份
  任意丢失 M 份 → 仍可恢复
  默认: EC:4 (每 Set 4 节点, 2 数据 + 2 校验)
  可配: EC:8 (8 节点, 4 数据 + 4 校验, 50% 利用率)

  vs 副本模式:
  - 3 副本: 100% 额外开销, 可丢 2 节点
  - EC 4+2: 50% 额外开销, 可丢 2 节点 (相同容错, 更省盘)
```

## 五、GlusterFS（文件存储）

### 5.1 架构原理

```
GlusterFS 无中心架构:

  ┌──────────────────────────────────────────────┐
  │              GlusterFS 集群                    │
  │                                              │
  │  ┌──────────┐    ┌──────────┐               │
  │  │ Server 1 │    │ Server 2 │               │
  │  │ /brick1  │    │ /brick1  │               │
  │  │          │    │          │               │
  │  │ 无中心   │◄──►│ 无中心   │  ← 互相通信    │
  │  │ 弹性哈希 │    │ 弹性哈希 │               │
  │  └──────────┘    └──────────┘               │
  │       ▲               ▲                     │
  │       │    ┌──────────┘                     │
  │       │   │                                 │
  │  ┌────┴───▼────────┐                       │
  │  │  Client (FUSE)   │  ← 客户端挂载          │
  │  │  Volume: gv0     │                       │
  │  └──────────────────┘                       │
  └──────────────────────────────────────────────┘

特点:
  - 无元数据服务器 (弹性哈希定位数据)
  - 每个节点都能接收客户端请求
  - 任意节点宕机不影响整体
  - 添加节点即可扩展

Volume 类型:
  ┌────────────┐
  │ Distribute │  分布式 (默认): 文件分布在所有 brick
  │ (哈希分布)  │  无冗余, 容量=所有 brick 之和
  └────────────┘
  ┌────────────┐
  │ Replicate  │  复制 (镜像): 文件在多个 brick 各存一份
  │ (副本)      │  冗余, 容量=单个 brick
  └────────────┘
  ┌────────────┐
  │ Stripe     │  条带: 文件分块分布在多个 brick
  │ (条带)      │  性能提升, 无冗余
  └────────────┘
  ┌────────────┐
  │ Disperse   │  纠删码: 类似 RAID 5/6
  │ (纠删码)    │  省空间 + 冗余
  └────────────┘
  ┌────────────────────────┐
  │ Distributed Replicate  │  分布式+复制 (最常用)
  │ (分布+副本)             │  先分布再复制, 兼顾扩展和冗余
  └────────────────────────┘
```

### 5.2 部署

```bash
# === 安装 (所有节点) ===
# Rocky / RHEL
dnf install -y centos-release-glusterfs
# 或:
dnf install -y https://download.gluster.org/pub/gluster/glusterfs/LATEST/Rocky/glusterfs-rocky-release.repo
dnf install -y glusterfs-server

# Ubuntu
apt install -y glusterfs-server

# 启动
systemctl enable --now glusterd

# === 配置信任池 ===
# 在 node-1 上:
gluster peer probe 192.168.1.11
gluster peer probe 192.168.1.12

# 查看
gluster peer status
# Number of Peers: 2
# Hostname: 192.168.1.11
# State: Peer in Cluster (Connected)
# Hostname: 192.168.1.12
# State: Peer in Cluster (Connected)

# === 创建 Brick (所有节点) ===
# Brick = 一个目录 + 一个磁盘
mkfs.xfs /dev/sdb
mkdir -p /data/brick1
mount /dev/sdb /data/brick1
echo "/dev/sdb /data/brick1 xfs defaults 0 0" >> /etc/fstab

# === 创建 Volume ===
# 分布式复制卷 (推荐: 兼顾扩展和冗余)
# 4 个 brick, 2 副本 → 逻辑 2 个 brick 的容量
gluster volume create gv0 \
    replica 2 \
    192.168.1.10:/data/brick1/gv0 \
    192.168.1.11:/data/brick1/gv0 \
    192.168.1.12:/data/brick1/gv0 \
    192.168.1.13:/data/brick1/gv0

# 启动 volume
gluster volume start gv0

# 查看
gluster volume info gv0
gluster volume status gv0

# === 客户端挂载 ===
# FUSE 挂载
dnf install -y glusterfs-fuse
mount -t glusterfs 192.168.1.10:/gv0 /mnt/glusterfs

# /etc/fstab
192.168.1.10:/gv0  /mnt/glusterfs  glusterfs  defaults,_netdev,backupvolfile-server=192.168.1.11  0 0

# === Volume 管理 ===
# 扩容 (添加 brick)
gluster volume add-brick gv0 \
    192.168.1.14:/data/brick1/gv0 \
    192.168.1.15:/data/brick1/gv0

# 重新平衡
gluster volume rebalance gv0 start
gluster volume rebalance gv0 status

# 缩容 (移除 brick)
gluster volume remove-brick gv0 \
    192.168.1.14:/data/brick1/gv0 \
    192.168.1.15:/data/brick1/gv0 start
gluster volume remove-brick gv0 \
    192.168.1.14:/data/brick1/gv0 \
    192.168.1.15:/data/brick1/gv0 commit
```

## 六、NFS（文件存储，基础）

### 6.1 NFS 架构

```
NFS (Network File System):

  ┌──────────┐          ┌──────────┐
  │  Client  │          │  Server  │
  │          │  RPC     │          │
  │ mount    │◄────────►│ rpcbind  │
  │ /mnt/nfs │  NFS     │ nfsd     │
  │          │◄────────►│ mountd   │
  │          │          │ /export  │
  └──────────┘          └──────────┘

NFS 版本:
  v3: 无状态, TCP/UDP, 广泛兼容
  v4: 有状态, 仅 TCP, 防火墙友好 (单端口 2049)
  v4.1: pNFS (并行 NFS, 性能提升)
  v4.2: 服务端复制, 稀疏文件

NFS vs CephFS vs GlusterFS:
  NFS:     简单, 单机, 无冗余 (需 DRBD/Pacemaker 做高可用)
  CephFS:  分布式, 强一致, 冗余, 运维复杂
  GlusterFS: 分布式, 无中心, 冗余, 中等复杂度
```

### 6.2 NFS 部署

```bash
# === 服务端 ===
dnf install -y nfs-utils

# 创建共享
mkdir -p /srv/nfs/share
chmod 755 /srv/nfs/share

# /etc/exports
cat > /etc/exports << 'EOF'
# 格式: 目录  客户端(选项)
/srv/nfs/share  192.168.1.0/24(rw,sync,root_squash,no_subtree_check,sec=sys)
/srv/nfs/iso    *(ro,sync,all_squash,sec=sys)
EOF

# 启动
systemctl enable --now nfs-server
exportfs -ra

# 防火墙
firewall-cmd --permanent --add-service=nfs
firewall-cmd --permanent --add-service=rpc-bind
firewall-cmd --permanent --add-service=mountd
firewall-cmd --reload

# === 客户端 ===
dnf install -y nfs-utils

# 查看服务端导出
showmount -e 192.168.1.10

# 挂载
mount -t nfs -o vers=4.2,rsize=1048576,wsize=1048576,hard,timeo=600,retrans=2,noatime \
    192.168.1.10:/srv/nfs/share /mnt/nfs

# /etc/fstab
192.168.1.10:/srv/nfs/share  /mnt/nfs  nfs  vers=4.2,hard,rsize=1048576,wsize=1048576,noatime,_netdev  0 0

# === NFS 高可用 (Pacemaker + DRBD) ===
# 两台 NFS 服务器 + DRBD 复制 + Pacemaker 故障转移
# 主节点写数据 → DRBD 实时复制到备节点
# 主节点挂了 → Pacemaker 提升备节点 → 虚拟 IP 飘移
# 客户端通过虚拟 IP 访问, 无感知故障转移
```

## 七、对比总结

### 7.1 按存储类型推荐

| 类型 | 轻量方案 | 生产方案 | 云原生方案 |
|:---|:---|:---|:---|
| 块存储 | LVM + iSCSI | Ceph RBD | Longhorn (K8s) |
| 文件存储 | NFS | CephFS / GlusterFS | CephFS (Rook) |
| 对象存储 | MinIO (单节点) | MinIO (分布式) | MinIO (Operator) |
| 统一存储 | — | Ceph | Ceph (Rook) |

### 7.2 数据冗余方式对比

| 方式 | 额外开销 | 容错 | 写放大 | 适用 |
|:---|:---|:---|:---:|:---|
| 3 副本 | 200% | 可丢 2 | 低 | 性能优先 |
| 2 副本 | 100% | 可丢 1 | 低 | 容量优先 |
| EC 4+2 | 50% | 可丢 2 | 高 | 冷数据/备份 |
| EC 8+4 | 50% | 可丢 4 | 高 | 大容量归档 |
| EC 6+3 | 50% | 可丢 3 | 中 | 平衡 |

### 7.3 性能对比（参考值）

| 方案 | 顺序写 | 随机写 | 延迟 | 并发扩展 |
|:---|:---:|:---:|:---:|:---|
| 本地 LVM | ⭐ 1000+ MB/s | ⭐ 50k IOPS | ⭐ 0.1ms | ❌ |
| iSCSI (LVM) | 200-500 MB/s | 5-20k IOPS | 0.5-2ms | ❌ |
| Ceph RBD | 100-300 MB/s | 5-15k IOPS | 1-5ms | ✅ |
| CephFS | 100-300 MB/s | 5-10k IOPS | 1-5ms | ✅ |
| GlusterFS | 50-200 MB/s | 3-8k IOPS | 2-10ms | ✅ |
| MinIO | 200-800 MB/s | — | 1-5ms | ✅ |
| NFS | 100-300 MB/s | 3-10k IOPS | 0.5-3ms | ❌ |
