## 一、全文检索选型

### 1.1 为什么不用数据库做全文搜索

```sql
-- MySQL LIKE 搜索的问题:
SELECT * FROM articles WHERE content LIKE '%云计算%';

-- 问题:
-- 1. 全表扫描 (不走索引, 性能差)
-- 2. 无法分词 (搜"云"匹配不到"云计算")
-- 3. 无相关性排序 (结果按主键排, 不是按相关度)
-- 4. 无中文分词 (搜"服务器"匹配"服务"和"器"但不是连在一起)
-- 5. 无高亮、纠错、同义词、聚合

-- 专用全文检索引擎:
-- 1. 倒排索引 (O(1) 查找, 极快)
-- 2. 分词 (中文: IK/Jieba, 英文: Standard)
-- 3. 相关性打分 (TF-IDF / BM25)
-- 4. 高亮、纠错、聚合、同义词
```

### 1.3 选型建议

| 场景 | 推荐 | 原因 |
|:---|:---|:---|
| 日志分析 (ELK) | **Elasticsearch** | 生态最完善，聚合能力强 |
| 大规模搜索 (亿级+) | **Elasticsearch** | 分布式成熟，PB 级 |
| 开源合规要求 | **OpenSearch** | Apache 2.0 许可证 |
| 中小型应用搜索 | **Meilisearch** | 零配置，<1ms 延迟，Rust 轻量 |

| 极致性能 + 低资源 | **Manticore / Typesense** | C++ 无 JVM 开销 |

## 二、倒排索引原理

### 2.1 正排索引 vs 倒排索引

```
正排索引 (数据库的 B+Tree):
  文档 ID → 文档内容

  Doc 1: "云计算架构设计指南"
  Doc 2: "云原生应用开发实践"
  Doc 3: "微服务架构与云原生"

  搜索 "架构" → 遍历所有文档 → 慢

倒排索引 (搜索引擎的核心):
  词项 (Term) → 文档 ID 列表 (Posting List)

  ┌──────────┬─────────────────────┐
  │ Term     │ Posting List        │
  ├──────────┼─────────────────────┤
  │ 云计算    │ [1]                 │
  │ 云原生    │ [2, 3]              │
  │ 架构     │ [1, 3]              │
  │ 设计     │ [1]                 │
  │ 指南     │ [1]                 │
  │ 应用     │ [2]                 │
  │ 开发     │ [2]                 │
  │ 实践     │ [2]                 │
  │ 微服务   │ [3]                 │
  │ 与       │ [3]                 │
  └──────────┴─────────────────────┘

  搜索 "架构" → 直接查 "架构" 的 Posting List → [1, 3] → O(1)
  搜索 "云原生 架构" → [2,3] ∩ [1,3] = [3] → Doc 3
```

### 2.2 倒排索引结构

```
倒排索引完整结构:

  ┌─────────────────────────────────────────────────┐
  │              倒排索引 (Inverted Index)            │
  │                                                 │
  │  ┌─────────────────────────────────────────┐    │
  │  │           Term Dictionary (词典)          │    │
  │  │  词项 → 词项 ID 映射 (FST/Hash)           │    │
  │  │  "架构" → T1                             │    │
  │  │  "云原生" → T2                            │    │
  │  └──────────────────┬──────────────────────┘    │
  │                     │                            │
  │  ┌──────────────────▼──────────────────────┐    │
  │  │          Term Index (词项索引)            │    │
  │  │  FST (有限状态转换器) → 快速定位词项        │    │
  │  │  常驻内存, 极快查找                        │    │
  │  └──────────────────┬──────────────────────┘    │
  │                     │                            │
  │  ┌──────────────────▼──────────────────────┐    │
  │  │          Posting List (倒排表)            │    │
  │  │                                          │    │
  │  │  T1 "架构":                              │    │
  │  │  ┌──────────────────────────────────┐   │    │
  │  │  │ DocID │ TF │ Position │ Offset   │   │    │
  │  │  │   1   │  1 │   [3]    │ [6,8]   │   │    │
  │  │  │   3   │  1 │   [1]    │ [0,2]   │   │    │
  │  │  └──────────────────────────────────┘   │    │
  │  │                                          │    │
  │  │  T2 "云原生":                            │    │
  │  │  ┌──────────────────────────────────┐   │    │
  │  │  │ DocID │ TF │ Position │ Offset   │   │    │
  │  │  │   2   │  1 │   [0]    │ [0,3]   │   │    │
  │  │  │   3   │  1 │   [3]    │ [7,10]  │   │    │
  │  │  └──────────────────────────────────┘   │    │
  │  └──────────────────────────────────────────┘    │
  │                                                 │
  │  Posting List 编码:                              │
  │    FOR  (Frame of Reference): 增量编码 + 位压缩  │
  │    Roaring Bitmap: 大数据集位图                   │
  │    RBM: 稀疏数据优化                              │
  └─────────────────────────────────────────────────┘

  TF (Term Frequency): 词在文档中出现次数
  Position: 词在文档中的位置 (用于短语查询)
  Offset: 词在文档中的字符偏移 (用于高亮)
```

### 2.3 相关性打分

```
BM25 算法 (Elasticsearch 默认):

  score(D, Q) = Σ IDF(qi) × TF(qi, D) × (k1 + 1)
                ────────────────────────────────
                TF(qi, D) + k1 × (1 - b + b × |D| / avgdl)

  其中:
    D  = 文档
    Q  = 查询 (q1, q2, ..., qn)
    IDF(qi) = log(1 + (N - n(qi) + 0.5) / (n(qi) + 0.5))
      N = 文档总数
      n(qi) = 包含 qi 的文档数
    TF(qi, D) = qi 在 D 中出现次数
    |D| = 文档长度
    avgdl = 平均文档长度
    k1 = 1.2 (词频饱和参数)
    b = 0.75 (长度归一化参数)

  含义:
  - IDF: 词越罕见, 权重越高 (搜"的"无意义, 搜"Kubernetes"权重高)
  - TF: 词在文档中出现越多, 权重越高 (但有饱和, 不会无限增长)
  - 文档长度: 短文档匹配权重更高 (避免长文档刷分)
```

## 三、Elasticsearch

### 3.1 架构原理

```
┌─────────────────────────────────────────────────────────┐
│                Elasticsearch 集群                        │
│                                                         │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ Node-1   │  │ Node-2   │  │ Node-3   │              │
│  │ Master   │  │ Master   │  │ Master   │              │
│  │ Data     │  │ Data     │  │ Data     │              │
│  │eligible  │  │eligible  │  │eligible  │              │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘              │
│       │             │             │                     │
│       └────── Cluster State ──────┘                     │
│                                                         │
│  Index: articles (3 主分片, 每分片 1 副本)               │
│                                                         │
│  ┌──────────────────────────────────────────┐           │
│  │  Shard 0     Shard 1     Shard 2        │           │
│  │  (Primary)   (Primary)   (Primary)      │           │
│  │  Node-1      Node-2      Node-3         │           │
│  │                                          │           │
│  │  Shard 0     Shard 1     Shard 2        │           │
│  │  (Replica)   (Replica)   (Replica)      │           │
│  │  Node-2      Node-3      Node-1         │           │
│  └──────────────────────────────────────────┘           │
│                                                         │
│  每个 Shard = 一个 Lucene Index                         │
│  Lucene Index = 多个 Segment (不可变) + 算一个分片        │
└─────────────────────────────────────────────────────────┘

节点角色:
  Master  管理集群元数据 (索引创建/删除, 分片分配)
  Data    存储数据, 执行 CRUD/搜索/聚合
  Coordinating 接收请求, 路由到分片, 汇聚结果
  Ingest  预处理管道 (写入前转换数据)
  ML      机器学习任务

Segment:
  - Lucene 的最小存储单元, 不可变 (写入后不能修改)
  - 新文档先写入内存 buffer → refresh (1s) → 新 Segment
  - 多个 Segment 定期 merge (合并减少文件数)
  - 删除 = 标记删除位 (实际在 merge 时清理)
  - refresh_interval 控制近实时 (NRT) 的延迟
```

### 3.2 安装部署

```bash
# === Rocky / RHEL 9 ===
# 导入 GPG key
rpm --import https://artifacts.elastic.co/GPG-KEY-elasticsearch

# 仓库
cat > /etc/yum.repos.d/elasticsearch.repo << 'EOF'
[elasticsearch-8.x]
name=Elasticsearch repository for 8.x packages
baseurl=https://artifacts.elastic.co/packages/8.x/yum
gpgcheck=1
gpgkey=https://artifacts.elastic.co/GPG-KEY-elasticsearch
enabled=1
autorefresh=1
type=rpm-md
EOF

dnf install -y elasticsearch kibana

# === Ubuntu 22.04 ===
wget -qO - https://artifacts.elastic.co/GPG-KEY-elasticsearch | gpg --dearmor -o /usr/share/keyrings/elasticsearch-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/elasticsearch-keyring.gpg] https://artifacts.elastic.co/packages/8.x/apt stable main" > /etc/apt/sources.list.d/elastic-8.x.list
apt update && apt install -y elasticsearch kibana

# === JVM 调整 (默认堆太大了, 小机器需要调) ===
# /etc/elasticsearch/jvm.options.d/heap.options
-Xms4g
-Xmx4g
# 规则: 堆大小 = 物理内存的 50%, 不超过 31GB (指针压缩)

# === 系统配置 ===
cat > /etc/sysctl.d/99-elasticsearch.conf << 'EOF'
vm.max_map_count=262144
fs.file-max=65536
EOF
sysctl -p /etc/sysctl.d/99-elasticsearch.conf

# 关闭 swap
swapoff -a
# 永久关闭: 注释 /etc/fstab 中的 swap 行
```

### 3.5 索引操作

```bash
# === 创建索引 ===
curl -u elastic:password -X PUT "localhost:9200/articles-202607" -H 'Content-Type: application/json' -d '
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1,
    "refresh_interval": "1s"
  },
  "mappings": {
    "properties": {
      "title":   { "type": "text", "analyzer": "ik_max_word", "search_analyzer": "ik_smart" },
      "content": { "type": "text", "analyzer": "ik_max_word", "search_analyzer": "ik_smart" },
      "author":  { "type": "keyword" },
      "tags":    { "type": "keyword" },
      "date":    { "type": "date", "format": "yyyy-MM-dd HH:mm:ss" }
    }

}'

# === 写入文档 ===

{

  "content": "本文探讨云原生架构的核心组件，包括容器编排、服务网格和不可变基础设施。",

  "tags": ["云原生", "Kubernetes", "架构"],

}'

# 批量写入 (性能高)
curl -X POST "localhost:9200/_bulk" -H 'Content-Type: application/x-ndjson' -d '
{"index": {"_index": "articles-202607"}}
{"title": "K8s Operator 开发指南", "content": "Operator 扩展 Kubernetes API...", "author": "李四", "tags": ["K8s", "Operator"]}
{"index": {"_index": "articles-202607"}}
{"title": "微服务网关选型", "content": "对比 Envoy、Nginx、Traefik 三种网关...", "author": "王五", "tags": ["微服务", "网关"]}
'

# === 搜索 ===
# 基本搜索 (全文搜索)
curl "localhost:9200/articles-202607/_search?q=云原生&pretty"

# DSL 搜索 (结构化查询)
curl -X POST "localhost:9200/articles/_search" -H 'Content-Type: application/json' -d '
{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "云原生" } },
        { "match": { "content": "架构" } }
      ],
      "filter": [
        { "term":  { "author": "张三" } },
        { "terms": { "tags": ["Kubernetes"] } },
        { "range": { "date": { "gte": "2026-01-01" } } }
      ]
    }
  },
  "highlight": {
    "fields": {
      "title":   { "pre_tags": ["<em>"], "post_tags": ["</em>"] },
      "content": { "pre_tags": ["<em>"], "post_tags": ["</em>"] }
    }
  },
  "sort": [
    { "_score": "desc" },
    { "date": "desc" }
  ],
  "from": 0,
  "size": 20
}'

# 聚合 (类似 GROUP BY)
curl -X POST "localhost:9200/articles/_search" -H 'Content-Type: application/json' -d '
{
  "size": 0,
  "aggs": {
    "by_author": {
      "terms": { "field": "author", "size": 10 },
      "aggs": {
        "avg_views": { "avg": { "field": "view_count" } }
      }
    }
  }
}'

# === 索引管理 ===
# 查看索引
curl "localhost:9200/_cat/indices?v"

# 索引别名 (零停机切换索引)
curl -X POST "localhost:9200/_aliases" -H 'Content-Type: application/json' -d '
{
  "actions": [
    { "remove": { "index": "articles-202606", "alias": "articles" } },
    { "add":    { "index": "articles-202607", "alias": "articles" } }
  ]
}'

# 段合并 (减少 Segment 数量, 提升搜索性能)
curl -X POST "localhost:9200/articles-202607/_forcemerge?max_num_segments=1"

# 关闭/打开索引 (关闭后不占资源但不能搜索)
curl -X POST "localhost:9200/old-index/_close"
curl -X POST "localhost:9200/old-index/_open"
```

## 四、OpenSearch

### 4.1 与 Elasticsearch 的关系

```
OpenSearch 是 AWS 在 2021 年从 Elasticsearch 7.10 fork 的开源分支

分叉原因:
  Elastic 公司从 7.11 开始将许可证从 Apache 2.0 改为 SSPL (非 OSI 开源)
  AWS 认为这违背了开源精神, fork 出 OpenSearch

兼容性:
  - API 高度兼容 (90%+ 查询语法相同)
  - Kibana → OpenSearch Dashboards
  - IK 分词器等插件有 OpenSearch 版本
  - 可直接使用 ES 客户端 (部分高级特性可能不同)

差异:
  - 许可证: OpenSearch = Apache 2.0 (真正开源)
  - 安全: OpenSearch 默认启用安全插件 (无需付费)
  - 发展路线: 逐渐分叉, 新版本特性不完全同步
```

### 5.1 特点
