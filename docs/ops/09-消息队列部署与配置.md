## 一、消息队列选型

### 1.1 为什么需要消息队列

```
没有消息队列 (同步调用):
┌──────┐     ┌──────┐     ┌──────┐     ┌──────┐
│ 用户  │────►│ API  │────►│订单   │────►│库存   │
└──────┘     └──────┘     └──────┘     └──────┘
                            │
                       ┌────▼────┐
                       │支付     │  ← 串行调用, 慢
                       └────┬────┘
                       ┌────▼────┐
                       │通知     │  ← 任一环节失败, 整体失败
                       └─────────┘

有消息队列 (异步解耦):
┌──────┐     ┌──────┐     ┌────────┐
│ 用户  │────►│ API  │────►│MQ      │  ← 快速响应
└──────┘     └──────┘     └───┬────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
         ┌────▼────┐     ┌────▼────┐     ┌────▼────┐
         │订单消费  │     │库存消费  │     │通知消费  │  ← 并行, 互不阻塞
         └─────────┘     └─────────┘     └─────────┘

消息队列的核心价值:
  1. 解耦: 生产者不需要知道消费者是谁
  2. 异步: 非关键路径异步处理, 快速响应用户
  3. 削峰: 高峰期请求先入队, 消费者按自身速率处理
  4. 可靠: 消息持久化, 消费者宕机不丢数据
  5. 广播: 一条消息多个消费者各取所需
```

### 1.2 主流消息队列对比

| 维度 | Kafka | RabbitMQ | RocketMQ | Redis Streams | NATS |
|:---|:---|:---|:---|:---|:---|
| 开发语言 | Scala/Java | Erlang | Java | C | Go |
| 定位 | 日志/流处理 | 通用消息中间件 | 金融级消息 | 轻量级消息 | 云原生消息 |
| 吞吐量 | ⭐ 百万级/秒 | 万级/秒 | 十万级/秒 | 万级/秒 | 十万级/秒 |
| 延迟 | 毫秒级 | ⭐ 微秒级 | 毫秒级 | 毫秒级 | ⭐ 微秒级 |
| 消息顺序 | Partition 内有序 | 队列内有序 | 队列内有序 | 消费组内有序 | ❌ |
| 消息可靠性 | 高 (ISR+ACK) | ⭐ 极高 (确认机制) | ⭐ 极高 (同步刷盘) | 中 | 中 |
| 持久化 | 磁盘 (段文件) | 磁盘 (可选) | 磁盘 (CommitLog) | RDB+AOF | 内存/JetStream |
| 消费模型 | Pull | Push | Push/Pull | 消费组 | Push/Pull |
| 延迟消息 | ❌ (需外部) | ✅ (插件/TTL+DLX) | ✅ (原生) | ❌ | ✅ (JetStream) |
| 死信队列 | ❌ | ✅ | ✅ | ❌ | ✅ (JetStream) |
| 事务消息 | ✅ (2PC) | ❌ | ⭐ 原生事务 | ✅ (MULTI) | ❌ |
| 消息回溯 | ✅ (offset) | ❌ | ✅ (时间戳) | ✅ (ID) | ✅ (JetStream) |
| 集群 | 强 (ZK/KRaft) | 集群镜像 | NameServer+Broker | Redis Cluster | ⭐ Gossip 自治 |
| 协议 | 自有 | AMQP/MQTT/STOMP | 自有 (兼容) | RESP | NATS/MQTT/WS |
| 运维复杂度 | 高 | 中 | 中高 | 低 | ⭐ 低 |
| 适用场景 | 大数据/日志/流 | 企业应用/路由 | 金融/电商订单 | 已有 Redis/轻量 | 微服务/IoT/边缘 |

### 1.3 选型建议

| 场景 | 推荐 | 原因 |
|:---|:---|:---|
| 日志采集 / 大数据流处理 | **Kafka** | 吞吐量最高，生态完善 |
| 企业应用 / 复杂路由 / 任务队列 | **RabbitMQ** | 路由灵活，延迟低，可靠性强 |
| 电商订单 / 金融交易 | **RocketMQ** | 事务消息，延迟消息，高可靠 |
| 已有 Redis / 轻量消息 | **Redis Streams** | 零额外组件，够用即可 |
| 微服务 / 云原生 / IoT | **NATS** | 轻量极快，运维简单 |
| 事件溯源 / CQRS | **Kafka** | 消息回溯，流处理 |

## 二、Kafka

### 2.1 核心概念

```
┌─────────────────────────────────────────────────────────┐
│                    Kafka 集群                             │
│                                                         │
│  Topic: order-events                                    │
│  ┌──────────────────────────────────────────┐           │
│  │  Partition 0   Partition 1   Partition 2  │           │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐│           │
│  │  │msg-0     │  │msg-1     │  │msg-2     ││           │
│  │  │msg-3     │  │msg-4     │  │msg-5     ││           │
│  │  │msg-6     │  │msg-7     │  │msg-8     ││           │
│  │  │offset:0,1│  │offset:0,1│  │offset:0,1││           │
│  │  │  2,3...  │  │  2,3...  │  │  2,3...  ││           │
│  │  └─────┬────┘  └─────┬────┘  └─────┬────┘│           │
│  └────────┼─────────────┼─────────────┼─────┘           │
│           │             │             │                  │
│     ┌─────▼─────┐ ┌────▼─────┐ ┌────▼─────┐             │
│     │ Replica   │ │ Replica  │ │ Replica  │             │
│     │ Leader    │ │ Leader   │ │ Leader   │             │
│     │ (Broker1)│ │(Broker2) │ │(Broker3) │             │
│     └─────┬─────┘ └──────────┘ └──────────┘             │
│       ┌───▼───┐                                         │
│       │Follower│  (Broker2 上的副本, 跟随 Leader)         │
│       └───────┘                                         │
│                                                         │
│  Consumer Group A                                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │Consumer 1│  │Consumer 2│  │Consumer 3│              │
│  │← Part 0  │  │← Part 1  │  │← Part 2  │  ← 每个消费者一个分区
│  └──────────┘  └──────────┘  └──────────┘              │
└─────────────────────────────────────────────────────────┘

核心概念:
  Topic        逻辑消息分类 (如 order-events)
  Partition    分区, Topic 的物理分片, 并行度单位
  Offset       分区内消息的序号 (从 0 递增)
  Replica      副本, 每个 Partition 有多个副本 (Leader + Follower)
  ISR          In-Sync Replicas, 与 Leader 保持同步的副本集
  Broker       Kafka 集群中的一个节点
  Producer     消息生产者
  Consumer     消息消费者
  Consumer Group 消费者组, 同组内分区被均分消费
  ZooKeeper/KRaft 集群元数据管理 (KRaft 是 Kafka 3.3+ 内置方案)
```

### 2.2 架构原理

```
Kafka 高吞吐量设计:

1. 顺序写磁盘 (比随机写快 6000 倍)
   消息追加到 Partition 末尾 → 顺序 IO → 磁盘也能很快

2. Page Cache (零拷贝)
   Producer → 写 Page Cache → OS 异步刷盘
   Consumer → 从 Page Cache 读 → sendfile() 零拷贝 → 网卡
   数据不经过用户空间

3. 批量发送
   Producer 积攒消息 → 批量发送 → 减少网络 RTT

4. 分区并行
   Topic 分多个 Partition → 多 Producer/Consumer 并行

5. 压缩
   Producer 压缩 (snappy/lz4/zstd) → Broker 存压缩态 → Consumer 解压

数据存储结构:
  /var/lib/kafka/data/
  ├── order-events-0/                ← Partition 0
  │   ├── 00000000000000000000.log   ← Segment 日志文件 (1GB)
  │   ├── 00000000000000000000.index ← 偏移量索引
  │   ├── 00000000000000000000.timeindex ← 时间戳索引
  │   └── 00000000000000123456.log   ← 下一个 Segment
  ├── order-events-1/                ← Partition 1
  └── order-events-2/                ← Partition 2

消息查找 (用 Offset 定位消息):
  1. 二分查找 .index 文件 → 找到最近的索引项
  2. 顺序扫描 .log 文件 → 定位到具体消息
  3. 时间复杂度 O(log N)
```

### 2.3 安装部署

```bash
# === 方式一: 直接安装 (KRaft 模式, 无需 ZooKeeper) ===
# Kafka 3.3+ 支持 KRaft 模式, 内置元数据管理

# 下载
wget https://downloads.apache.org/kafka/3.7.1/kafka_2.13-3.7.1.tgz
tar xzf kafka_2.13-3.7.1.tgz
mv kafka_2.13-3.7.1 /opt/kafka

# 配置: /opt/kafka/config/kraft/server.properties
cat > /opt/kafka/config/kraft/server.properties << 'EOF'
# === 基本配置 ===
process.roles=broker,controller        # KRaft: 同时充当 broker 和 controller
node.id=1                              # 节点 ID (集群内唯一)
controller.quorum.voters=1@192.168.1.10:9093,2@192.168.1.11:9093,3@192.168.1.12:9093

# === 监听 ===
listeners=PLAINTEXT://:9092,CONTROLLER://:9093
advertised.listeners=PLAINTEXT://192.168.1.10:9092
listener.security.protocol.map=CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT
controller.listener.names=CONTROLLER
inter.broker.listener.name=PLAINTEXT

# === 存储 ===
log.dirs=/var/lib/kafka/data            # 数据目录
log.segment.bytes=1073741824            # Segment 大小 1GB
log.retention.hours=168                 # 数据保留 7 天
log.retention.check.interval.ms=300000  # 清理检查间隔 5 分钟
log.cleanup.policy=delete               # delete | compact
log.segment.ms=604800000                # Segment 滚动周期 7 天

# === 性能 ===
num.network.threads=8                   # 网络线程
num.io.threads=16                       # IO 线程
socket.send.buffer.bytes=102400
socket.receive.buffer.bytes=102400
socket.request.max.bytes=104857600
queued.max.requests=500

# === Topic 默认 ===
num.partitions=3                        # 默认分区数
default.replication.factor=3            # 默认副本数
min.insync.replicas=2                   # 最小同步副本 (acks=all 时)

# === 复制 ===
replica.lag.time.max.ms=30000           # Follower 最大滞后时间
replica.fetch.max.bytes=1048576

# === 事务 ===
transaction.state.log.replication.factor=3
transaction.state.log.min.isr=2
EOF

# 格式化存储 (KRaft)
KAFKA_CLUSTER_ID="$(/opt/kafka/bin/kafka-storage.sh random-uuid)"
/opt/kafka/bin/kafka-storage.sh format \
    -t "$KAFKA_CLUSTER_ID" \
    -c /opt/kafka/config/kraft/server.properties

# systemd 服务
cat > /etc/systemd/system/kafka.service << 'EOF'
[Unit]
Description=Apache Kafka (KRaft)
After=network.target

[Service]
Type=simple
Environment="JAVA_HOME=/usr/lib/jvm/java-17"
Environment="KAFKA_HEAP_OPTS=-Xmx4G -Xms4G"
Environment="KAFKA_JVM_PERFORMANCE_OPTS=-XX:+UseG1GC -XX:MaxGCPauseMillis=20"
ExecStart=/opt/kafka/bin/kafka-server-start.sh /opt/kafka/config/kraft/server.properties
ExecStop=/opt/kafka/bin/kafka-server-stop.sh
Restart=always
User=kafka
LimitNOFILE=100000

[Install]
WantedBy=multi-user.target
EOF

useradd -r -s /sbin/nologin kafka
mkdir -p /var/lib/kafka/data
chown -R kafka:kafka /var/lib/kafka /opt/kafka
systemctl daemon-reload
systemctl enable --now kafka
```

### 2.4 Topic 与消费组管理

```bash
# === Topic 管理 ===
# 创建 Topic
/opt/kafka/bin/kafka-topics.sh --bootstrap-server localhost:9092 \
    --create --topic order-events \
    --partitions 6 \
    --replication-factor 3 \
    --config retention.ms=604800000 \
    --config compression.type=lz4 \
    --config min.insync.replicas=2

# 查看 Topic 列表
/opt/kafka/bin/kafka-topics.sh --bootstrap-server localhost:9092 --list

# 查看 Topic 详情
/opt/kafka/bin/kafka-topics.sh --bootstrap-server localhost:9092 \
    --describe --topic order-events

# 增加分区 (只能增加, 不能减少)
/opt/kafka/bin/kafka-topics.sh --bootstrap-server localhost:9092 \
    --alter --topic order-events --partitions 12

# 修改配置
/opt/kafka/bin/kafka-configs.sh --bootstrap-server localhost:9092 \
    --alter --entity-type topics --entity-name order-events \
    --add-config retention.ms=2592000000

# 删除 Topic
/opt/kafka/bin/kafka-topics.sh --bootstrap-server localhost:9092 \
    --delete --topic order-events

# === 生产消息 ===
# 命令行生产者 (测试用)
/opt/kafka/bin/kafka-console-producer.sh \
    --bootstrap-server localhost:9092 \
    --topic order-events \
    --property "parse.key=true" \
    --property "key.separator=:"
# > order-1001:{"product":"laptop","qty":1,"price":9999}

# === 消费消息 ===
# 命令行消费者
/opt/kafka/bin/kafka-console-consumer.sh \
    --bootstrap-server localhost:9092 \
    --topic order-events \
    --from-beginning \
    --group order-consumer-group \
    --property "print.key=true"

# === 消费组管理 ===
# 查看所有消费组
/opt/kafka/bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 --list

# 查看消费组详情 (Lag 监控)
/opt/kafka/bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
    --describe --group order-consumer-group
# TOPIC           PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG  CONSUMER-ID
# order-events    0          1000            1005            5    consumer-1
# order-events    1          998             1000            2    consumer-2

# 重置 Offset (从头消费)
/opt/kafka/bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
    --group order-consumer-group \
    --topic order-events \
    --reset-offsets --to-earliest --execute

# 重置到指定时间
/opt/kafka/bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
    --group order-consumer-group \
    --topic order-events \
    --reset-offsets --to-datetime 2026-07-09T00:00:00.000 --execute

# === ACL 安全 (可选) ===
# 创建用户
/opt/kafka/bin/kafka-configs.sh --bootstrap-server localhost:9092 \
    --alter --add-config 'SCRAM-SHA-256=[password=KafkaPass2026]' \
    --entity-type users --entity-name admin
```

### 2.5 生产者/消费者代码示例

```python
# pip install confluent-kafka

from confluent_kafka import Producer, Consumer, KafkaError
import json

# === 生产者 ===
producer = Producer({
    'bootstrap.servers': '192.168.1.10:9092,192.168.1.11:9092',
    'client.id': 'order-service',
    'acks': 'all',                    # all=ISR 全确认 (最安全)
    'enable.idempotence': True,       # 幂等生产 (防重复)
    'compression.type': 'lz4',        # 压缩
    'linger.ms': 10,                  # 批量等待 10ms
    'batch.size': 32768,              # 批量大小 32KB
    'max.in.flight.requests.per.connection': 5,
    'retries': 2147483647,            # 无限重试
})

def delivery_report(err, msg):
    if err:
        print(f'消息发送失败: {err}')
    else:
        print(f'消息发送成功: {msg.topic()}[{msg.partition()}]@{msg.offset()}')

# 发送消息
for i in range(100):
    key = f'order-{i}'
    value = json.dumps({'order_id': i, 'product': 'laptop', 'qty': 1})
    producer.produce('order-events', key=key.encode(), value=value.encode(), callback=delivery_report)
    producer.poll(0)  # 触发回调

producer.flush()  # 等待所有消息发送完成

# === 消费者 ===
consumer = Consumer({
    'bootstrap.servers': '192.168.1.10:9092,192.168.1.11:9092',
    'group.id': 'order-consumer-group',
    'auto.offset.reset': 'latest',     # earliest | latest | none
    'enable.auto.commit': False,       # 手动提交 (安全)
    'session.timeout.ms': 10000,
    'max.poll.interval.ms': 300000,
    'fetch.min.bytes': 1024,           # 最小拉取量
    'fetch.max.wait.ms': 500,          # 最大等待时间
})

consumer.subscribe(['order-events'])

try:
    while True:
        msg = consumer.poll(1.0)       # 拉取消息 (超时 1 秒)
        if msg is None:
            continue
        if msg.error():
            if msg.error().code() == KafkaError._PARTITION_EOF:
                continue
            print(f'消费错误: {msg.error()}')
            continue

        # 处理消息
        key = msg.key().decode() if msg.key() else None
        value = json.loads(msg.value().decode())
        print(f'收到: {key} → {value} (partition={msg.partition()}, offset={msg.offset()})')

        # 手动提交 offset
        consumer.commit(msg)
except KeyboardInterrupt:
    pass
finally:
    consumer.close()
```

### 2.6 生产配置要点

```bash
# 生产者关键参数:
acks=all                        # 所有 ISR 副本确认 (不丢数据)
enable.idempotence=true         # 幂等 (防网络重试导致重复)
compression.type=lz4            # 压缩 (减少网络+存储)
linger.ms=10                    # 批量发送等待 (吞吐 vs 延迟)
batch.size=32768                # 批量大小
max.in.flight.requests.per.connection=5  # 并发请求数 (幂等时 ≤5)

# 消费者关键参数:
enable.auto.commit=false        # 手动提交 (防消息丢失)
auto.offset.reset=latest        # 无 offset 时从最新开始
max.poll.records=500            # 单次拉取最大记录
max.poll.interval.ms=300000     # 处理超时 (超时被踢出组)
session.timeout.ms=10000        # 心跳超时

# Broker 关键参数:
min.insync.replicas=2           # 最小同步副本 (acks=all 时至少 2 个)
unclean.leader.election.enable=false  # 禁止非 ISR 副本成为 Leader (防丢数据)
log.message.format.version=3.0  # 消息格式版本
num.replica.fetchers=4          # 副本拉取线程数
```

## 三、RabbitMQ

### 3.1 核心概念

```
┌─────────────────────────────────────────────────────────┐
│                   RabbitMQ 架构                          │
│                                                         │
│  ┌──────────┐                                           │
│  │ Producer │                                           │
│  └────┬─────┘                                           │
│       │ 发布消息 (Publish)                                │
│  ┌────▼──────────────────────────────────────────┐      │
│  │                 Exchange (交换机)                │      │
│  │                                                 │      │
│  │  ┌─────────┐ ┌──────────┐ ┌─────────┐         │      │
│  │  │direct   │ │topic     │ │fanout   │         │      │
│  │  │精确匹配  │ │模式匹配   │ │广播      │         │      │
│  │  │routing  │ │routing   │ │所有队列  │         │      │
│  │  │key      │ │key (通配)│ │都收到    │         │      │
│  │  └────┬────┘ └────┬─────┘ └────┬────┘         │      │
│  └───────┼───────────┼────────────┼──────────────┘      │
│          │           │            │                      │
│     Binding        Binding      Binding                  │
│          │           │            │                      │
│  ┌───────▼───┐  ┌───▼────┐  ┌───▼────┐                  │
│  │ Queue A   │  │Queue B │  │Queue C │                  │
│  │           │  │        │  │        │                  │
│  │ msg-1     │  │ msg-1  │  │ msg-1  │                  │
│  │ msg-2     │  │ msg-2  │  │ msg-2  │                  │
│  └───────┬───┘  └───┬────┘  └───┬────┘                  │
│          │          │            │                      │
│  ┌───────▼───┐  ┌───▼────┐  ┌───▼────┐                  │
│  │ Consumer  │  │Consumer│  │Consumer│                   │
│  │ A         │  │ B      │  │ C      │                   │
│  └───────────┘  └────────┘  └────────┘                   │
└─────────────────────────────────────────────────────────┘

核心概念:
  Producer      消息生产者
  Exchange      交换机, 接收消息, 根据路由规则分发到队列
  Queue         队列, 存储消息
  Binding       绑定, Exchange 和 Queue 之间的路由规则
  Routing Key   路由键, 生产者发送时指定, Exchange 根据它路由
  Consumer      消息消费者
  vhost         虚拟主机, 逻辑隔离 (多租户)

Exchange 类型:
  direct   精确匹配 routing key
  topic    通配符匹配 (* 匹配一个词, # 匹配多个词)
  fanout   广播到所有绑定的队列 (忽略 routing key)
  headers  根据消息头匹配 (不常用)
```

### 3.6 死信队列与延迟消息

```
死信队列 (Dead Letter Queue):
  消息变成死信的条件:
    1. 消息被 NACK 且 requeue=false
    2. 消息 TTL 过期
    3. 队列超过最大长度

  流程:
  ┌──────────┐  NACK(requeue=false)  ┌──────────┐
  │ 正常队列  │ ─────────────────────► │ 死信交换机 │
  │          │                        │ (DLX)    │
  └──────────┘                        └────┬─────┘
                                           │
                                     ┌─────▼─────┐
                                     │死信队列    │
                                     │(人工处理)  │
                                     └───────────┘

延迟消息 (TTL + DLX):
  ┌──────────┐  TTL 30s 过期   ┌──────────┐
  │ 延迟队列  │ ──────────────► │ 死信交换机 │ ──► 目标队列
  │ (不消费)  │                 │ (DLX)    │
  └──────────┘                 └──────────┘

  消息进入延迟队列 → 不被消费 → TTL 过期 → 转到死信交换机 → 到目标队列被消费
```

```python
# 延迟消息实现
# 1. 声明死信交换机 + 目标队列
channel.exchange_declare(exchange='order.dlx', exchange_type='direct', durable=True)
channel.queue_declare(queue='order.timeout.queue', durable=True)
channel.queue_bind(exchange='order.dlx', queue='order.timeout.queue', routing_key='order.timeout')

# 2. 声明延迟队列 (绑定 DLX + TTL)
channel.queue_declare(
    queue='order.delay.30s',
    durable=True,
    arguments={
        'x-dead-letter-exchange': 'order.dlx',
        'x-dead-letter-routing-key': 'order.timeout',
        'x-message-ttl': 30000,       # 30 秒 TTL
    },
)

# 3. 发消息到延迟队列
channel.basic_publish(exchange='', routing_key='order.delay.30s', body=b'{"order_id":1001}')

# 4. 30 秒后消息自动转到 order.timeout.queue 被消费
```

## 四、RocketMQ

### 4.1 核心概念与架构

```
┌──────────────────────────────────────────────────────────┐
│                    RocketMQ 架构                          │
│                                                          │
│  ┌──────────┐                                            │
│  │ Producer │                                            │
│  └────┬─────┘                                            │
│       │                                                  │
│  ┌────▼────────────┐    ┌──────────────────┐            │
│  │ NameServer 集群  │    │   Broker 集群     │            │
│  │ (元数据, 无状态)  │◄──►│                  │            │
│  │                  │    │  ┌────────────┐  │            │
│  │  NS-1   NS-2    │    │  │ Master-1   │  │            │
│  │                  │    │  │ (读写)      │  │            │
│  └──────────────────┘    │  └─────┬──────┘  │            │
│                          │        │ 同步     │            │
│                          │  ┌─────▼──────┐  │            │
│                          │  │ Slave-1    │  │            │
│                          │  │ (只读)      │  │            │
│                          │  └────────────┘  │            │
│                          └──────────────────┘            │
│       │                                                  │
│  ┌────▼─────┐                                            │
│  │ Consumer │                                            │
│  └──────────┘                                            │
└──────────────────────────────────────────────────────────┘

存储结构 (所有 Topic 共用一个 CommitLog):
  ${ROCKETMQ_HOME}/store/
  ├── commitlog/                       ← 所有消息顺序写入
  │   └── 00000000000000000000         ← 1GB 文件
  │   └── 00000000001073741824
  ├── consumequeue/                    ← 逻辑消费队列 (索引)
  │   └── TopicA/
  │       ├── 0/                       ← Queue 0
  │       │   └── 00000000000000000000 ← 20 bytes × N
  │       └── 1/
  └── index/                           ← 索引文件 (按 key 查找)

  特点:
  - 所有 Topic 的消息混写在 CommitLog 中 (顺序写, 极快)
  - ConsumeQueue 是逻辑队列, 存消息在 CommitLog 中的偏移量
  - 消费者从 ConsumeQueue 读取偏移量, 再到 CommitLog 取消息

  vs Kafka:
  - Kafka: 每个 Partition 独立文件 (多文件顺序写)
  - RocketMQ: 所有消息一个 CommitLog (单文件顺序写, 更极致)
```

### 4.4 事务消息原理

```
RocketMQ 事务消息 (两阶段提交):

  Producer                    Broker
     │                          │
     │ 1. 发送半消息 (half)      │
     │─────────────────────────►│  ← 半消息对消费者不可见
     │                          │
     │ 2. 执行本地事务           │
     │   (写数据库)              │
     │                          │
     │ 3. 提交/回滚              │
     │─────────────────────────►│
     │   Commit → 消息可见       │
     │   Rollback → 删除消息     │
     │                          │
     │ 4. (超时) 回查事务状态     │
     │◄─────────────────────────│  ← Broker 定期回查
     │                          │
     │ 5. 返回 Commit/Rollback   │
     │─────────────────────────►│
```

```java
// Java 事务消息示例
TransactionMQProducer producer = new TransactionMQProducer("tx-group");
producer.setNamesrvAddr("192.168.1.10:9876");

// 事务回查监听器
producer.setTransactionListener(new TransactionListener() {
    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        // 执行本地事务 (写数据库)
        try {
            orderService.createOrder(msg);
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }

    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // Broker 回查: 本地事务是否成功?
        boolean exists = orderService.isOrderExists(msg.getKeys());
        return exists ? LocalTransactionState.COMMIT_MESSAGE
                      : LocalTransactionState.ROLLBACK_MESSAGE;
    }
});

producer.start();

// 发送事务消息
Message msg = new Message("order-topic", "tag",
    "order-1001", "{\"product\":\"laptop\"}".getBytes());
producer.sendMessageInTransaction(msg, null);
```

## 五、Redis Streams

### 5.1 适用场景

```
Redis Streams 是 Redis 5.0+ 内置的消息队列功能

优势:
  - 已有 Redis, 无需额外组件
  - 延迟低 (内存操作)
  - 支持消费组
  - 支持消息回溯 (通过 ID)
  - 支持持久化 (RDB+AOF)

劣势:
  - 吞吐量不如 Kafka
  - 消息容量受 Redis 内存限制
  - 不适合大数据流场景

适合: 中小规模应用、已有 Redis 环境、对延迟敏感的场景
```

### 5.2 基本操作

```bash
# === 生产消息 ===
# XADD key [NOMKSTREAM] [MAXLEN|MINID [=|~] threshold] field value [field value ...]
redis-cli XADD orders '*' order_id 1001 product laptop qty 1
# * 表示自动生成 ID (时间戳-序号)
# 返回: 1717891200000-0

# 指定最大长度 (防止内存爆炸)
redis-cli XADD orders MAXLEN ~ 10000 '*' order_id 1002 product mouse qty 2
# ~ 表示近似裁剪 (性能更好)

# === 消费消息 ===
# 按范围读取
redis-cli XRANGE orders - +              # 所有消息
redis-cli XRANGE orders - + COUNT 10     # 前 10 条
redis-cli XRANGE orders 1717891200000-0 +  # 从指定 ID 开始

# 反向读取
redis-cli XREVRANGE orders + - COUNT 5   # 最新 5 条

# === 消费组 ===
# 创建消费组
redis-cli XGROUP CREATE orders order-group 0    # 0=从头消费
redis-cli XGROUP CREATE orders order-group $    # $=只消费新消息

# 消费组读取
redis-cli XREADGROUP GROUP order-group consumer-1 COUNT 10 BLOCK 5000 STREAMS orders >
# > 表示读取未投递的消息
# BLOCK 5000: 阻塞等待 5 秒

# 确认消息 (ACK)
redis-cli XACK orders order-group 1717891200000-0

# 查看待确认消息 (未 ACK)
redis-cli XPENDING orders order-group

# 消息回溯: 重新消费
redis-cli XCLAIM orders order-group consumer-2 0 1717891200000-0
# 将消息转移给 consumer-2

# === 消息长度 ===
redis-cli XLEN orders

# === 裁剪 ===
redis-cli XTRIM orders MAXLEN 1000    # 只保留 1000 条
```

### 5.3 Python 示例

```python
# pip install redis
import redis
import json
import time

r = redis.Redis(host='127.0.0.1', port=6379, password='***', decode_responses=True)

# === 生产者 ===
for i in range(100):
    msg = json.dumps({'order_id': i, 'product': f'item-{i}'})
    msg_id = r.xadd('orders', {'data': msg}, maxlen=10000, approximate=True)
    print(f'发送: {msg_id} → {msg}')

# === 消费者 (消费组) ===
# 确保消费组存在
try:
    r.xgroup_create('orders', 'order-group', id='0', mkstream=True)
except redis.ResponseError as e:
    if 'BUSYGROUP' not in str(e):
        raise

while True:
    # 读取消息
    messages = r.xreadgroup(
        groupname='order-group',
        consumername='consumer-1',
        streams={'orders': '>'},     # > = 新消息
        count=10,
        block=5000,                  # 阻塞 5 秒
    )

    for stream, msg_list in messages:
        for msg_id, fields in msg_list:
            data = json.loads(fields['data'])
            print(f'消费: {msg_id} → {data}')

            # 处理消息...
            process_order(data)

            # ACK
            r.xack('orders', 'order-group', msg_id)
```

## 六、NATS

### 6.1 核心概念

```
NATS 是一个云原生消息系统, Go 编写, 极轻量

核心特点:
  - 单二进制 (无依赖, <20MB)
  - 极高性能 (百万级消息/秒)
  - 微秒级延迟
  - Gossip 自治集群 (无需 ZooKeeper/etcd)
  - 支持 NATS/MQTT/WebSocket 协议

NATS Subject (主题) 命名:
  order.created          ← 精确
  order.*                ← 通配一级
  order.>                ← 通配多级
  *.created              ← 任意一级 + created

  vs RabbitMQ:
  - RabbitMQ: Exchange + Queue + Binding (显式绑定)
  - NATS: Subject 直接匹配 (无 Exchange 概念, 更简单)
```

### 6.3 JetStream（持久化流）

```bash
# 创建 Stream
nats stream add ORDERS \
    --subjects 'order.*' \
    --storage file \
    --retention limits \
    --max-msgs 100000 \
    --max-age 168h

# 查看流
nats stream info ORDERS

# 创建消费者
nats consumer add ORDERS PROCESSOR \
    --filter 'order.created' \
    --ack explicit \
    --max-deliver 3 \
    --pull

# 查看消费者
nats consumer info ORDERS PROCESSOR

# 查看消息
nats stream view ORDERS --from 0
```

```python
# pip install nats-py
import nats
import json

async def main():
    nc = await nats.connect("nats://admin:NATSPass2026@127.0.0.1:4222")
    js = nc.jetstream()

    # 发布
    await js.add_stream(name="ORDERS", subjects=["order.*"])
    ack = await js.publish("order.created", json.dumps({"id": 1001}).encode())
    print(f"发布: seq={ack.seq}")

    # 订阅
    async def handler(msg):
        data = json.loads(msg.data)
        print(f"消费: {data}")
        await msg.ack()

    await js.subscribe("order.created", handler=handler, durable="PROCESSOR")

nats.run(main())
```

## 七、对比总结

### 7.1 消息可靠性保障对比

| 机制 | Kafka | RabbitMQ | RocketMQ | Redis Streams | NATS JetStream |
|:---|:---|:---|:---|:---|:---|
| 生产确认 | ACK (acks=all) | Publisher Confirm | 同步刷盘 | XADD 返回 ID | ACK |
| 消费确认 | Offset Commit | ACK | ACK | XACK | ACK |
| 消息持久化 | 磁盘 (段文件) | 磁盘 (可选) | 磁盘 (CommitLog) | RDB+AOF | 磁盘 (文件) |
| 消息重复 | 幂等生产 | ❌ (需业务去重) | ✅ (幂等) | ❌ | ✅ |
| 死信队列 | ❌ | ✅ (原生) | ✅ (原生) | PEL 转移 | ✅ |
| 延迟消息 | ❌ | ✅ (TTL+DLX) | ✅ (原生) | ❌ | ✅ |
| 事务消息 | ✅ (2PC) | ❌ | ⭐ (半消息+回查) | ✅ (MULTI) | ❌ |
| 消息回溯 | ✅ (Offset) | ❌ | ✅ (时间戳) | ✅ (ID) | ✅ (Seq) |

### 7.2 运维复杂度对比

| 维度 | Kafka | RabbitMQ | RocketMQ | Redis Streams | NATS |
|:---|:---|:---|:---|:---|:---|
| 依赖组件 | 无 (KRaft) / ZK | Erlang | NameServer | Redis | 无 |
| 部署难度 | 中 | 低 | 中 | ⭐ 最低 | ⭐ 最低 |
| 内存占用 | 高 (JVM) | 中 (Erlang VM) | 高 (JVM) | 低 | ⭐ 最低 |
| 集群搭建 | 中 | 中 | 中 | 低 | ⭐ 低 |
| 监控方案 | JMX+Prometheus | Management UI | Dashboard | INFO | Prometheus |
| 故障恢复 | 自动 (ISR) | 镜像队列 | Master-Slave | Cluster | ⭐ 自动 |
