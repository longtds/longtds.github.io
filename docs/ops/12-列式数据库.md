## 一、列式数据库选型

### 1.1 行存 vs 列存

```
行式存储 (MySQL/PostgreSQL):
  数据按行存储, 一行的所有字段连续存放

  物理布局:
  ┌─────────────────────────────────────────────┐
  │ Row 1: [id=1, name="Alice",  age=25, city="北京"] │
  │ Row 2: [id=2, name="Bob",    age=30, city="上海"] │
  │ Row 3: [id=3, name="Carol",  age=28, city="广州"] │
  └─────────────────────────────────────────────┘

  SELECT name, age FROM users WHERE age > 25;
  → 读取所有行 (包括不需要的 id, city) → 大量无效 IO

列式存储 (ClickHouse/Doris):
  数据按列存储, 每一列单独存放

  物理布局:
  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
  │ id 列     │ │ name 列   │ │ age 列    │ │ city 列   │
  │          │ │          │ │          │ │          │
  │ 1        │ │ "Alice"  │ │ 25       │ │ "北京"   │
  │ 2        │ │ "Bob"    │ │ 30       │ │ "上海"   │
  │ 3        │ │ "Carol"  │ │ 28       │ │ "广州"   │
  └──────────┘ └──────────┘ └──────────┘ └──────────┘

  SELECT name, age FROM users WHERE age > 25;
  → 只读取 name 列 + age 列 → IO 减少 50%+
```

### 1.2 列存的核心优势

```
1. 减少IO (只读需要的列)
   分析查询通常只涉及部分列 (如 SELECT count(*), avg(price) FROM orders GROUP BY region)
   100 列的表, 查询 5 列 → IO 减少 95%

2. 高压缩比 (同列数据类型一致)
   行存: "Alice", 25, "北京", "Bob", 30, "上海" → 混合类型, 难压缩
   列存: "Alice","Bob","Carol" → 同类型, 字典编码+LZ4 → 5-10x 压缩

   压缩编码:
   ┌────────────┬──────────────────────────────────┐
   │ 编码方式    │ 适用场景                          │
   ├────────────┼──────────────────────────────────┤
   │ 字典编码    │ 低基数列 (性别/状态/枚举)          │
   │ Run-Length  │ 重复值多的列 (排序后的列)          │
   │ Delta       │ 递增序列 (时间戳/自增ID)           │
   │ LZ4         │ 通用压缩 (速度快)                  │
   │ ZSTD        │ 通用压缩 (压缩比高)                │
   │ Gorilla     │ 浮点数 (监控指标)                  │
   │ Bit-Packed  │ 小整数                            │
   └────────────┴──────────────────────────────────┘

3. 向量化执行 (SIMD)
   一次处理一批数据 (如 1024 个值), 利用 CPU SIMD 指令
   行存: 逐行处理 (分支预测失败, CPU 流水线中断)
   列存: 批量处理 (无分支, SIMD, CPU 缓存友好)

4. 预聚合 (Materialized View / Rollup)
   预先计算常用聚合, 查询时直接读结果

行存仍然适合:
   - OLTP (点查/单行读写)
   - 高频小事务
   - 需要行级更新/删除
```

## 二、ClickHouse

### 2.1 架构原理

```
┌─────────────────────────────────────────────────────────┐
│                  ClickHouse 集群                         │
│                                                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │  Node-1      │  │  Node-2      │  │  Node-3      │  │
│  │              │  │              │  │              │  │
│  │ ┌──────────┐ │  │ ┌──────────┐ │  │ ┌──────────┐ │  │
│  │ │ Shard 1  │ │  │ │ Shard 2  │ │  │ │ Shard 3  │ │  │
│  │ │ Replica  │ │  │ │ Replica  │ │  │ │ Replica  │ │  │
│  │ │  (主)    │ │  │ │  (主)    │ │  │ │  (主)    │ │  │
│  │ └────┬─────┘ │  │ └────┬─────┘ │  │ └────┬─────┘ │  │
│  │      │       │  │      │       │  │      │       │  │
│  │ ┌────▼─────┐ │  │ ┌────▼─────┐ │  │ ┌────▼─────┐ │  │
│  │ │ Shard 2  │ │  │ │ Shard 3  │ │  │ │ Shard 1  │ │  │
│  │ │ Replica  │ │  │ │ Replica  │ │  │ │ Replica  │ │  │
│  │ │  (备)    │ │  │ │  (备)    │ │  │ │  (备)    │ │  │
│  │ └──────────┘ │  │ └──────────┘ │  │ └──────────┘ │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
│                                                         │
│  分片 (Shard): 水平切分数据, 分布在不同节点               │
│  副本 (Replica): 同一分片的数据复制, 保证高可用            │
│                                                         │
│  分布式表 (Distributed): 跨 Shard 聚合查询的虚拟表        │
│  本地表 (MergeTree): 实际存储数据的物理表                 │
└─────────────────────────────────────────────────────────┘

存储引擎: MergeTree 家族
┌──────────────────────────────────────────────────┐
│  MergeTree (基础引擎)                              │
│  │                                                │
│  ├── 数据按 Partition 分区                          │
│  │     Partition → 多个 Part (数据块)              │
│  │                                                │
│  ├── Part 内部按 OrderBy 排序                       │
│  │     排序后数据有序 → 范围查询极快                 │
│  │                                                │
│  ├── 每个 Part = 多个 Column 文件                   │
│  │     每列一个文件 (.bin) + 索引 (.idx)            │
│  │     + 标记文件 (.mrk) → 偏移量                   │
│  │                                                │
│  ├── 后台合并 (Merge)                               │
│  │     多个小 Part → 合并为大 Part                   │
│  │     减少文件数量, 提升查询性能                    │
│  │                                                │
│  └── 数据压缩                                       │
│        每列独立压缩 (LZ4/ZSTD)                      │
│        压缩比通常 5-10x                             │
└──────────────────────────────────────────────────┘

MergeTree 变体:
  MergeTree           基础版 (最常用)
  ReplacingMergeTree  自动去重 (按 OrderBy 去重)
  SummingMergeTree    自动求和 (按维度聚合)
  AggregatingMergeTree 自动聚合 (预聚合)
  CollapsingMergeTree 带标记的合并 (sign=1/-1)
  VersionedCollapsingMergeTree 带版本号的合并
```

### 2.2 数据存储详解

```
ClickHouse 存储结构:

  /var/lib/clickhouse/data/
  └── default/
      └── logs/                              ← 表名
          ├── 202607_1_5_2/                  ← Partition_SrcPart_DstPart_Level
          │   ├── date.bin                   ← date 列数据 (压缩)
          │   ├── date.idx                   ← date 列索引 (min/max)
          │   ├── date.mrk2                  ← date 列标记 (偏移量)
          │   ├── level.bin                  ← level 列数据
          │   ├── level.idx
          │   ├── level.mrk2
          │   ├── message.bin                ← message 列数据
          │   ├── message.idx
          │   ├── message.mrk2
          │   ├── checksums.txt              ← 校验和
          │   ├── columns.txt                ← 列信息
          │   ├── count.txt                  ← 行数
          │   └── primary.idx               ← 主键索引 (稀疏索引)
          ├── 202608_6_6_0/
          └── format_version.txt

查询流程:
  1. 根据 Partition 过滤 → 只扫描匹配的 Part
  2. 主键索引 (稀疏, 每隔 8192 行一个条目) → 定位数据范围
  3. 标记文件 (.mrk2) → 定位列数据在 .bin 中的偏移
  4. 解压 .bin → 读取列数据
  5. 向量化执行 (SIMD)

主键索引 (稀疏索引):
  ┌──────────────────────────────────┐
  │ primary.idx (每 8192 行一个条目)   │
  │                                  │
  │ Entry 0: date=2026-07-01, id=1   │ ──→ Mark 0 → .bin offset 0
  │ Entry 1: date=2026-07-03, id=8193│ ──→ Mark 1 → .bin offset 4096
  │ Entry 2: date=2026-07-05, id=16385│──→ Mark 2 → .bin offset 8192
  │ ...                              │
  └──────────────────────────────────┘

  稀疏索引: 索引小 (常驻内存), 适合范围查询
  二分查找 → 定位到 Mark → 读取对应数据块
```

### 2.5 建表与查询

```sql
-- === 基本建表 (MergeTree) ===
CREATE TABLE logs.access_log
(
    event_time     DateTime,
    user_id        UInt64,
    session_id     String,
    url            String,
    method         LowCardinality(String),    -- 低基数字符串 (字典编码)
    status_code    UInt16,
    response_time  UInt32,
    bytes_sent     UInt64,
    referer        String,
    user_agent     String,
    country        LowCardinality(String),
    device         LowCardinality(String)
)
ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(event_time)           -- 按天分区
ORDER BY (event_time, user_id)                -- 排序键 (稀疏索引)
SETTINGS index_granularity = 8192,             -- 索引粒度
         compression = 'lz4',                  -- 压缩算法
         ttl_only_drop_parts = 1;

-- TTL (自动过期)
ALTER TABLE logs.access_log
    MODIFY TTL event_time + INTERVAL 90 DAY;   -- 90 天后自动删除

-- === ReplacingMergeTree (去重) ===
CREATE TABLE events.user_events
(
    event_id    String,
    user_id     UInt64,
    event_type  LowCardinality(String),
    event_time  DateTime,
    properties  String,
    sign        Int8                          -- 1=正常, -1=撤销
)
ENGINE = ReplacingMergeTree(event_time)        -- 按 event_time 去重 (保留最新)
PARTITION BY toYYYYMM(event_time)
ORDER BY (event_id, user_id);

-- === 分布式表 ===
-- 本地表 (每个节点上创建)
CREATE TABLE logs.access_log_local ON CLUSTER my_cluster
(
    event_time  DateTime,
    user_id     UInt64,
    url         String,
    status_code UInt16
)
ENGINE = ReplicatedMergeTree(
    '/clickhouse/tables/{shard}/logs/access_log_local',
    '{replica}'
)
PARTITION BY toYYYYMMDD(event_time)
ORDER BY (event_time, user_id);

-- 分布式表 (查询入口, 自动路由到各 Shard)
CREATE TABLE logs.access_log_distributed ON CLUSTER my_cluster
AS logs.access_log_local
ENGINE = Distributed(
    my_cluster,             -- 集群名
    logs,                   -- 库名
    access_log_local,       -- 本地表名
    rand()                  -- 分片规则 (rand()=随机, 也可按字段哈希)
);

-- 写入分布式表 → 自动分发到各 Shard
-- 查询分布式表 → 自动聚合各 Shard 结果

-- === 物化视图 (预聚合) ===
-- 源表: 每条访问日志
-- 目标: 每小时按 URL 聚合

CREATE MATERIALIZED VIEW logs.access_hourly_mv
ENGINE = SummingMergeTree()
PARTITION BY toYYYYMM(hour)
ORDER BY (hour, url)
AS
SELECT
    toStartOfHour(event_time) AS hour,
    url,
    count()                    AS pv,
    uniq(user_id)              AS uv,
    sum(response_time)         AS total_response_time,
    countIf(status_code >= 400) AS error_count
FROM logs.access_log
GROUP BY hour, url;

-- 查询物化视图 (极快, 已经预聚合)
SELECT hour, url, pv, uv
FROM logs.access_hourly_mv
WHERE hour >= '2026-07-09 00:00:00'
ORDER BY pv DESC
LIMIT 20;

-- === 批量写入 (极快) ===
-- CSV 文件导入
clickhouse-client --query="INSERT INTO logs.access_log FORMAT CSV" < access_log.csv

-- JSONEachRow
clickhouse-client --query="INSERT INTO logs.access_log FORMAT JSONEachRow" < data.json

-- 压缩导入 (网络传输少)

-- === 查询示例 ===
```

### 3.1 架构原理

```
┌─────────────────────────────────────────────────────────┐
│                   Apache Doris 架构                      │
│                                                         │
│  ┌──────────────────────────────────────────┐           │
│  │              FE (Frontend)                │           │
│  │                                          │           │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐  │           │
│  │  │ FE-1    │  │ FE-2    │  │ FE-3    │  │           │
│  │  │ Master  │  │ Follower│  │ Observer│  │           │
│  │  │ (读写)   │  │ (读)    │  │ (读)    │  │           │
│  │  └────┬────┘  └────┬────┘  └────┬────┘  │           │
│  │       │            │            │       │           │
│  │       └────── BDBJE ────────────┘       │           │
│  │       (元数据同步, 基于 BerkeleyDB)       │           │
│  │                                          │           │
│  │  职责:                                   │           │
│  │  - SQL 解析/优化/执行计划                  │           │
│  │  - 元数据管理 (表/分区/副本)               │           │
│  │  - 查询调度 (分配给 BE)                    │           │
│  │  - MySQL 协议 (兼容 MySQL 客户端)         │           │
│  └──────────────────────────────────────────┘           │
│                      │                                  │
│                      │ 分发查询                          │
│                      ▼                                  │
│  ┌──────────────────────────────────────────┐           │
│  │              BE (Backend)                 │           │
│  │                                          │           │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐  │           │
│  │  │ BE-1    │  │ BE-2    │  │ BE-3    │  │           │
│  │  │         │  │         │  │         │  │           │
│  │  │ Tablet 1│  │ Tablet 2│  │ Tablet 3│  │           │
│  │  │ Tablet 2│  │ Tablet 3│  │ Tablet 1│  │           │
│  │  │ (副本)  │  │ (副本)  │  │ (副本)  │  │           │
│  │  └─────────┘  └─────────┘  └─────────┘  │           │
│  │                                          │           │
│  │  职责:                                   │           │
│  │  - 数据存储 (列式, Tablet 为单位)         │           │
│  │  - 查询执行 (MPP 并行)                    │           │
│  │  - 数据导入 (Stream Load/Broker Load)     │           │
│  └──────────────────────────────────────────┘           │
│                                                         │
│  Tablet: 数据分片的最小单位                               │
│  一个 Partition 包含多个 Tablet, 每个 Tablet 有多个副本    │
└─────────────────────────────────────────────────────────┘

MPP (Massively Parallel Processing) 执行:
  查询 → FE 生成执行计划 → 分发给多个 BE → 并行执行 → 汇聚结果
  vs ClickHouse 的 Scatter-Gather: 每个节点独立计算, 汇聚节点合并

Doris 数据模型:
┌──────────────────────────────────────────────────┐
│  Duplicate (明细模型, 默认)                         │
│  不做任何聚合, 存储所有原始行                        │
│  适合: 明细查询, 日志存储                           │
│                                                  │
│  Aggregate (聚合模型)                              │
│  按 Key 维度自动聚合 (SUM/MIN/MAX/REPLACE)         │
│  适合: 预聚合, 报表                                 │
│                                                  │
│  Unique (唯一主键模型)                              │
│  按 Key 去重, 新数据覆盖旧数据                       │
│  适合: 维表, 需要更新的数据                         │
│                                                  │
│  Unique (Merge-on-Write, 推荐)                     │
│  唯一模型的改进版, 写入时标记删除, 查询时不需合并     │
│  查询性能接近 Duplicate 模型                        │
└──────────────────────────────────────────────────┘
```

## 四、StarRocks

### 4.1 与 Doris 的关系

```
StarRocks 是 Apache Doris (原百度 Palo) 的商业分支

分叉时间: 2020 年
分叉原因: 团队对 Doris 的发展方向有分歧, StarRocks 专注
          多表 JOIN 性能和实时数仓场景

主要改进 (vs Doris):
  1. CBO (Cost-Based Optimizer) 优化器 — 更智能的 JOIN 重排
  2. Global Runtime Filter — 大表 JOIN 加速

  6. Data Cache — 本地缓存远端数据 (查询 Hive/Iceberg)
  7. 存算分离架构 (Shared-Data) — 计算和存储独立扩展

兼容性:
  - MySQL 协议 (同 Doris)
  - API 高度兼容 Doris
  - 部分语法有差异 (StarRocks 独有的函数和特性)

选择:
  - 简单报表/轻量 OLAP → Doris (Apache 2.0 开源)
  - 多表 JOIN/复杂分析/实时数仓 → StarRocks
```

### 4.3 建表与查询

```sql
-- === 建表 (同 Doris 语法, 增加了主键模型) ===

-- 主键模型 (Primary Key, StarRocks 推荐)
-- 支持实时更新, 查询性能接近明细模型
CREATE TABLE events.user_events
(
    event_id    VARCHAR(64),
    user_id     BIGINT,
    event_type  VARCHAR(32),
    event_time  DATETIME NOT NULL,
    properties  JSON,
    amount      DOUBLE
)
ENGINE = OLAP
PRIMARY KEY(event_id)                         -- 主键模型
PARTITION BY date_trunc('day', event_time)
DISTRIBUTED BY HASH(event_id) BUCKETS 10
PROPERTIES (
    "replication_num" = "2",
    "enable_persistent_index" = "true"        -- 持久化主键索引
);

-- === 物化视图 (StarRocks 特色, 透明加速) ===
-- 异步物化视图 (基于定时刷新)
CREATE MATERIALIZED VIEW mv_hourly_stats
DISTRIBUTED BY HASH(country) BUCKETS 5
REFRESH ASYNC EVERY (INTERVAL 1 HOUR)         -- 每小时刷新
AS
SELECT
    date_trunc('hour', event_time) AS hour,
    country,
    device,
    count(*) AS pv,
    count(DISTINCT user_id) AS uv,
    avg(response_time) AS avg_rt
FROM logs.access_log
GROUP BY hour, country, device;

-- 查询时自动改写 (用户不需要知道物化视图存在)
SELECT country, sum(pv) FROM mv_hourly_stats  -- 手动查
WHERE hour >= '2026-07-09 00:00:00'
GROUP BY country;

-- 或自动改写 (查询原始表, 优化器自动选择物化视图)
SELECT country, count(*)
FROM logs.access_log
WHERE event_time >= '2026-07-09 00:00:00'

-- Runtime Filter 在运行时过滤

EXPLAIN ANALYZE
SELECT
    u.username,
    u.country,
    e.event_type,
    count(*) AS cnt,
    sum(e.amount) AS total_amount
FROM events.user_events e
JOIN dim.users u ON e.user_id = u.user_id
WHERE e.event_time >= '2026-07-01'
```

## 五、Apache Druid

### 5.1 架构原理

```
┌─────────────────────────────────────────────────────────┐
│                   Apache Druid 架构                      │
│                                                         │
│  ┌──────────────────────────────────────────┐           │
│  │              Master 节点                   │           │
│  │  ┌──────────────┐  ┌──────────────┐      │           │
│  │  │ Coordinator  │  │ Overlord     │      │           │
│  │  │ (数据管理)    │  │ (任务管理)    │      │           │
│  │  │ 管理数据可用性 │  │ 管理数据摄入  │      │           │
│  │  └──────────────┘  └──────────────┘      │           │
│  └──────────────────────────────────────────┘           │
│                      │                                  │
│       ┌──────────────┼──────────────┐                  │
│       │              │              │                  │
│  ┌────▼────┐   ┌────▼────┐   ┌────▼────┐              │
│  │ Query   │   │ Query   │   │ Query   │              │
│  │ Server  │   │ Server  │   │ Server  │              │
│  │ (Broker)│   │ (Broker)│   │ (Broker)│              │
│  │ 查询路由 │   │         │   │         │              │
│  └────┬────┘   └────┬────┘   └────┬────┘              │
│       │              │              │                  │
│  ┌────▼──────────────▼──────────────▼────┐            │
│  │           Historical 节点              │            │
│  │  ┌──────────┐  ┌──────────┐          │            │
│  │  │ Segment 1│  │ Segment 2│  ...     │            │
│  │  │ (历史数据)│  │          │          │            │
│  │  └──────────┘  └──────────┘          │            │
│  │  存储已构建好的 Segment (不可变)        │            │
│  └──────────────────────────────────────┘            │
│                      │                                  │
│  ┌───────────────────▼──────────────────┐            │
│  │           Middle Manager 节点          │            │
│  │  ┌──────────┐  ┌──────────┐          │            │
│  │  │ Peon     │  │ Peon     │  ...     │            │
│  │  │ (实时摄入)│  │ (实时摄入)│          │            │
│  │  └──────────┘  └──────────┘          │            │
│  │  实时消费 Kafka → 生成 Segment        │            │
│  └──────────────────────────────────────┘            │
│                                                         │
│  Deep Storage (S3/HDFS): Segment 永久存储               │
│  ZooKeeper: 集群协调                                    │
└─────────────────────────────────────────────────────────┘

Druid 核心概念:
  DataSource    相当于表
  Segment       数据段 (按时间分块, 不可变)
  Timestamp     必须有时间列 (Druid 是时序优先的)
  Granularity   时间粒度 (segmentGranularity / queryGranularity)

Druid 数据模型:
  时间戳列:   __time (必须)
  维度列:     字符串, 用于过滤和分组 (filter_by / group_by)
  指标列:     数值, 用于聚合 (sum / count / max / min)

  vs ClickHouse: 不需要预定义聚合, 查询时动态聚合
  vs Druid: 摄入时必须指定聚合方式, Segment 不可变

实时摄入流程:
  Kafka → MiddleManager (Peon) → 内存聚合 → Segment
  → handoff 到 Historical → 永久存储到 Deep Storage
```

## 六、对比总结

### 6.1 写入方式对比

| 方式 | ClickHouse | Doris | StarRocks | Druid |
|:---|:---|:---|:---|:---|
| HTTP 导入 | ✅ HTTP | ⭐ Stream Load | ⭐ Stream Load | ✅ |
| Kafka 实时 | Kafka 引擎 | Routine Load | Routine Load | ⭐ Supervisor |
| HDFS/S3 批量 | ✅ S3 表函数 | Broker Load | Broker Load | Hadoop Indexer |
| 文件导入 | ✅ (极快) | ✅ | ✅ | ❌ |
| INSERT SELECT | ✅ | ✅ | ✅ | ❌ |
| JDBC 导入 | ✅ | ✅ | ✅ | ❌ |

### 6.2 性能对比

| 场景 | ClickHouse | Doris | StarRocks | Druid |
|:---|:---|:---|:---|:---|
| 单表聚合 | ⭐⭐ 最强 | 强 | 强 | 强 |
| 多表 JOIN | 弱 | 强 | ⭐⭐ 最强 | 弱 |
| 实时写入吞吐 | ⭐ 极高 | 高 | 高 | 中 |
| 查询延迟 | ⭐ <100ms | <1s | <1s | ⭐ <100ms |
| 高并发查询 | 中 | 高 | 高 | ⭐ 极高 |
| 数据压缩比 | ⭐ 5-10x | 3-5x | 3-5x | 3-5x |
| 资源占用 | ⭐ 低 (C++) | 中 (JVM) | 中 (JVM) | 高 (JVM) |

### 6.3 运维对比

| 维度 | ClickHouse | Doris | StarRocks | Druid |
|:---|:---|:---|:---|:---|
| 部署复杂度 | 中 | ⭐ 低 | ⭐ 低 | 高 |
| 依赖组件 | ZK (可选) | 无 | 无 | ZK+MySQL+S3 |
| 集群扩缩容 | 中 | ⭐ 简单 | ⭐ 简单 | 中 |
| 数据均衡 | 手动 | ⭐ 自动 | ⭐ 自动 | 自动 |
| 监控方案 | system 表 + Exporter | 内置 + Exporter | 内置 + Exporter | Web UI + Exporter |
| MySQL 协议 | ✅ (9004) | ⭐ ✅ (9030) | ⭐ ✅ (9030) | ❌ |
| BI 工具兼容 | 通过 MySQL 端口 | ⭐ 原生 | ⭐ 原生 | SQL API |

### 6.4 选型决策树

```
需要列式 OLAP?
├── 单表聚合为主 (日志/监控/行为分析)
│   ├── 数据量 PB 级 → ClickHouse
│   └── 数据量 TB 级 → ClickHouse / Doris
│
├── 多表 JOIN 为主 (数仓/BI)
│   ├── 需要极致 JOIN 性能 → StarRocks
│   └── 轻量数仓 → Doris
│
├── 实时数仓 (Flink + OLAP)
│   ├── 多表 JOIN → StarRocks
│   └── 简单聚合 → Doris / ClickHouse
│
├── 高并发实时看板 (亚秒级)
│   ├── 预聚合 + 高并发 → Druid / Pinot
│   └── 单表聚合 → ClickHouse
│
└── 已有 MySQL 生态
    └── Doris / StarRocks (MySQL 协议)
```
