# AIOps 发展与展望

> LLM 把 AIOps 从"自动化"推向了"智能化"，运维即将进入对话式时代。

## 一、LLM 重塑运维（最大变革）

### 1.1 经典 ML AIOps vs LLM AIOps

| 维度 | 经典 ML | LLM 加持 |
|:---|:---|:---|
| 解释能力 | 弱（黑盒） | 强（自然语言） |
| 知识利用 | 有限（标注数据） | 全网知识 + RAG |
| 上手成本 | 高（数据科学家） | 低（提示工程） |
| 适合场景 | 量化异常 | 复杂场景、跨域推理 |
| 局限 | 通用性差 | 实时性、可靠性 |

### 1.2 主流 LLM 运维工具

| 工具 | 厂商 | 能力 |
|:---|:---|:---|
| **HolmesGPT** | Robusta | K8s 故障诊断 + 根因推理 |
| **k8sgpt** | 社区 | K8s 资源问题分析 |
| **Robusta** | 商业 | Prometheus 告警 + LLM 解释 |
| **PagerDuty AI** | PagerDuty | 告警上下文摘要 |
| **Datadog Bits AI** | Datadog | 全栈智能助手 |
| **New Relic AI** | New Relic | 自然语言查指标 |

### 1.3 LLM AIOps 三种模式

```
模式 1: 告警增强（最易落地）
        告警 → LLM 解释 → 推荐 Runbook
        
模式 2: 故障诊断（高价值）
        告警 → LLM Agent 调用工具（kubectl/SQL）→ 输出根因
        
模式 3: 自然语言运维（终极形态）
        SRE 说："最近一小时 P99 突增的服务有哪些？"
        LLM → 查 Prometheus → 关联日志 → 给出列表 + 分析
```

## 二、可观测性现代化

### 2.1 OpenTelemetry 一统江湖

```
旧:  Prometheus (指标) + ELK (日志) + Jaeger (追踪) → 3 套 SDK
新:  OpenTelemetry SDK → 统一采集 → 任意后端
```

### 2.2 新一代后端栈

```
Mimir / VictoriaMetrics  (指标长期存储)
Loki / ClickHouse         (日志)
Tempo / Jaeger            (追踪)
        ↓
        统一查询: LogQL / TraceQL / PromQL
        ↓
        Grafana 统一可视化
```

### 2.3 持续 Profiling

eBPF + Parca/Pyroscope 实现"始终在线"的性能分析：

| 维度 | 传统 | 持续 Profiling |
|:---|:---|:---|
| 触发 | 出问题才采 | 7×24 采样 |
| 开销 | 高（10%+） | 低（< 1%） |
| 数据 | 时间点快照 | 持续时间序列 |

## 四、自动化修复

### 4.1 自愈层次

```
L0:  手动响应                     ← 起点
L1:  人工 Runbook 半自动
L2:  规则触发自动执行（IFTTT）
L3:  ML 触发自动执行
L4:  LLM Agent 自主决策 + 执行     ← 目标
```

### 4.2 LLM Agent 闭环

```
告警 → 上下文收集 (kubectl/SQL/日志)
     → LLM 推理 (根因 + 修复方案)
     → 安全检查 (生产环境约束)
     → 执行（人确认 or 自动）
     → 验证 + 复盘
```

风险点：
- LLM 幻觉 → 必须验证才执行
- 误操作扩散 → 灰度 + 回滚
- 合规审计 → 全链路日志

## 六、RAG 运维知识库

```
文档源（Confluence/GitLab Wiki/Runbook）
   ↓ 切分 + 向量化
向量库（Qdrant/Milvus）
   ↓ 检索增强
LLM 回答
   ↓ 引用来源
SRE 可追溯
```

**落地建议**：
- 知识源去重、定期清理
- 关键文档加结构化标签
- 评估检索准确率（Hit Rate / MRR）
- 设置"不知道"边界（避免幻觉）
