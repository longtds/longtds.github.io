# Kubernetes 发展与展望

> K8s 正从"通用容器编排"扩展到"AI / VM / Wasm / 边缘"全场景平台。

## 一、当前热点方向

### 1.1 K8s for AI（最热）

| 项目 | 用途 |
|:---|:---|
| **KubeRay** | Ray 分布式训练 |
| **Kueue** | K8s 原生批调度 |
| **Volcano** | Gang Scheduling、bin packing |
| **NVIDIA GPU Operator** | GPU 驱动 + Device Plugin 自动化 |
| **KServe / Seldon** | 模型推理服务 |
| **vLLM Production Stack** | LLM 推理服务 |
| **llm-d** | 分布式 LLM 推理（Red Hat 主推） |

### 1.2 多集群管理

```
单集群          ← K8s 1.0-1.20 主要场景
↓
多集群          ← 容灾、混合云
↓
集群即资源      ← Cluster API、Karmada
↓
Fleet 管理      ← Rancher Fleet、ArgoCD App-of-Apps
```

| 项目 | 模式 |
|:---|:---|
| Cluster API (CAPI) | 用 K8s API 管理 K8s 集群 |
| Karmada | 华为主导，工作负载跨集群分发 |
| Rancher Fleet | GitOps 多集群 |
| KubeFed v2 | (社区降温) |

### 1.3 Gateway API

替代 Ingress 的新一代 API：

| 维度 | Ingress | Gateway API |
|:---|:---|:---|
| 模型 | 单一资源 | GatewayClass + Gateway + Route 三层 |
| 协议 | HTTP/HTTPS | HTTP/TCP/UDP/gRPC/TLS |
| 角色分离 | ❌ | ✅ 平台/应用/运维分离 |
| 扩展性 | annotations 散乱 | 标准 + 厂商扩展点 |

K8s 1.31 进入 v1 GA。

## 三、K8s + VM 融合（KubeVirt）

让 K8s 同时跑容器和 VM：

```yaml
# 一个 VM 看起来像 Pod
apiVersion: kubevirt.io/v1
kind: VirtualMachine
```

**应用场景**：
- 把传统 VM 工作负载迁移到 K8s 平台
- 私有云从 OpenStack 切到 K8s
- AI 训练用 VM（驱动隔离），推理用容器

代表方案：Red Hat OpenShift Virtualization、Harvester (SUSE)。

## 四、边缘 K8s

| 方案 | 节点要求 | 特点 |
|:---|:---|:---|
| k3s | 512MB+ | Rancher，单二进制 |
| k0s | 1GB+ | 简洁、Mirantis |
| KubeEdge | 1GB+ | 华为，边缘自治 |
| OpenYurt | 1GB+ | 阿里，节点自治 |
| MicroK8s | 1GB+ | Canonical |

**边缘场景**：5G MEC、车联网、工厂、零售门店、CDN。

## 五、Serverless on K8s

| 方案 | 模式 |
|:---|:---|
| Knative | 标准 CNCF，scale-to-zero |
| OpenFaaS | 函数即服务 |
| Kubeless | (已停止) |
| KEDA | 事件驱动自动扩缩 |

K8s 1.30+ 配合 KEDA 可以接 200+ 事件源。

## 六、Wasm on K8s

```
方案 1：Krustlet (废弃)
方案 2：containerd shim + runwasi  ✅ 主流
方案 3：SpinKube (Fermyon)        ✅ 新兴
```

**Pod 跑 Wasm 模块**：
```yaml
spec:
  runtimeClassName: wasmtime
  containers:
    - name: my-wasm-app
      image: registry.io/my-wasm:v1
```

## 七、安全演进

| 趋势 | 项目 |
|:---|:---|
| 零信任 | SPIFFE/SPIRE + Cilium mTLS |
| 运行时安全 | Falco + Tetragon (eBPF) |
| 镜像签名 | Sigstore (cosign + policy-controller) |
| 策略引擎 | OPA Gatekeeper / Kyverno |
| 机密容器 | Confidential Containers (CoCo) |

## 八、未来 3-5 年展望

| 方向 | 预测 |
|:---|:---|
| **AI 调度** | 90% 的 GPU 集群用 K8s 调度 |
| **多集群** | Cluster API 成为多云事实标准 |
| **Gateway API** | 完全替代 Ingress |
| **KubeVirt 普及** | 大部分私有云改用 K8s + KubeVirt |
| **边缘 K8s** | 5G/IoT 场景标配 |
| **AI 控制平面** | LLM 辅助运维（kubectl ai 等） |
| **极简控制面** | etcd 可选，单二进制部署普及 |

## 九、对运维的影响

| 变化 | 运维新要求 |
|:---|:---|
| AI 工作负载 | Volcano/Kueue 批调度、GPU MIG |
| Gateway API | 替换 Ingress，重写规则 |
| eBPF 网络 | Cilium 调试技能 |
| KubeVirt | 同时具备 K8s + VM 运维 |
| 多集群 | GitOps + CAPI |
| LLM 运维助手 | k8sgpt / HolmesGPT 等工具 |

## 十、学习路径（持续路径）

1. **基础认证**：CKAD → CKA → CKS
2. **进阶专项**：
   - 网络：Cilium 深度
   - 存储：Rook/Ceph 或 Longhorn
   - 安全：Falco + OPA
   - 调度：Volcano + Kueue
3. **生态扩展**：Helm/ArgoCD/Prometheus/OTel
4. **AI 平台**：KServe/KubeRay/llm-d
5. **多集群**：CAPI/Karmada

> 💡 K8s 不是终点，而是云原生时代的"x86"——所有上层应用都建立在它的 API 上。
