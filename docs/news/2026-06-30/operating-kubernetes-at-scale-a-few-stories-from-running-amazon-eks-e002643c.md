## 摘要

Amazon EKS 在 30 多个 AWS 区域运行数十万个 Kubernetes 集群。大规模运营 Kubernetes 的帖子：运行 Amazon EKS 的一些故事首先出现在 The New Stack 上。

## 正文

Amazon EKS 在 30 多个 AWS 区域运行数十万个 Kubernetes 集群。如此规模的运营教会了我们一些东西，这些东西决定了我们构建服务的方式，并且我们认为这对任何大规模运行 Kubernetes 的人来说都是有用的：大多数可用性问题并不是源于组件故障。它们来自对问题作出反应的组件，其方式使问题变得更糟。缓存会过时并提供错误的答案。重新启动保持集群活动的进程的运行状况检查。

弹性控制平面与脆弱控制平面的区别不在于故障数量。问题在于故障是否仍然是故障还是成为故障。这篇文章讲述了我们如何以不断增长的规模将 EKS 管理的 Kubernetes 控制平面保持在该线的右侧：我们所做的根本性改变及其原因，以及大规模运营教会了我们如何构建能够容忍故障而不是传播故障的系统。这些就是我们要求最严格的客户自信地在 EKS 上运行其关键任务工作负载的原因。

### 人工智能和分析工作负载如何重塑“规模”的含义

Kubernetes 是为特定的工作节奏而构建的。吊舱以可预测的速度来来去去，控制人员有几秒钟或几分钟的时间进行协调。该系统的设计反映了这种节奏：强大的数据一致性、有序的监视以及将正确性放在首位的共识复制存储。它完美地实现了它的设计目的，而且仍然如此。

>

“弹性控制平面与脆弱控制平面的区别不在于故障的数量。而在于故障是否会一直是故障还是会变成中断。”

但[工作负载的发展速度比任何人预期的都要快](https://thenewstack.io/why-ai-workloads-are-fueling-a-move-back-to-postgres/)。基础模型训练只需几分钟即可在数千个 GPU 节点上运行扩展训练作业。实时推理服务从温暖基线扩展到数千个副本，然后在一小时内回落。 Apache Spark 分析管道从零爆发到数以万计的执行器 Pod，遍历数据集，然后消失。新兴的代理人工智能工作负载增加了另一个维度：自主代理可以在几秒或更短的时间内旋转、扇出、执行任务和拆卸。这些工作负载有一个与传统微服务不同的共同特征：它们在压缩的时间窗口内生成大量的状态转换，并且非常不能容忍延迟。这种状态变化的速度促使我们重新发明一些机制，以支持以前不可能实现的规模，并为上游做出我们所能做的贡献。

### EKS 如何重新构想 

Kubernetes 存储基础

每个 Kubernetes 集群都依赖 etcd 作为其事实来源。每个应用程序、每个服务端点、每个调度决策都存储在那里。如果 etcd 丢失数据，集群就会忘记它所知道的一切。保护该状态是托管 Kubernetes 服务最重要的工作。

为一个集群操作 etcd 是很好理解的。为数以百万计的车队运营它是一个完全不同的问题。硬件故障、网络故障和磁盘性能下降，因此必须在没有人参与的情况下处理这些事件。而 etcd 最需要的操作，例如更换故障成员或在区域事件后恢复，正是在压力下行事的人可能会犯下导致永久数据丢失的错误的操作。

从一开始，我们就构建了一个操作员代理，它与每个 etcd 实例一起运行并自动化其整个生命周期。代理人有两份工作。首先，备份和恢复：它获取时间点快照并将其持久存储在集群外部。如果一次丢失太多实例并且幸存者无法形成多数，代理会自动检测情况并从最新快照进行重建。其次，成员管理：当实例发生故障时，代理会删除已终止的成员并按照保护法定人数并防止脑裂的顺序添加其替代成员。

最近的一个更根本的变化是用一个专门构建的日志取代了 etcd 的共识机制 Raft，该日志提供独立于 etcd 的持久、有序存储。在传统的 etcd 中，大多数成员必须在提交之前就每次写入达成一致。如果三个中的两个不健康，集群将变得不可用。通过将持久性转移给日志，etcd 同行之间不再协商法定人数。一旦日志确认持久性，就会立即写入提交，并且整个 etcd 仲裁丢失失败类别都消失了。由于日志处理持久性，etcd 不再需要 fsync 写入本地磁盘，因此其数据存储已移动到内存文件系统。原来受磁盘限制的系统变成了受计算限制的系统，并且存储延迟完全从关键路径中消除。要更深入地了解此架构，请阅读“[底层：Amazon EKS 超大规模集群](https://aws.amazon.com/blogs/containers/under-the-hood-amazon-eks-ultra-scale-clusters/)”。

>

“原来受磁盘限制的系统变成了受计算限制的系统，并且存储延迟完全从关键路径中消除了。”

对于超大规模集群，我们更进一步，将 etcd 划分为特定于资源的分片。每个分区都独立运行，具有自己的存储预算和吞吐量。主要价值是故障隔离。在整体部署中，如果由于行为不当的控制器创建对象的速度快于垃圾收集删除对象的速度而导致事件键空间超出其配额，则会阻止对所有内容的写入，包括节点租约。

突然间，健康的节点变得不健康，因为它们的租约续订被拒绝。使用分区的 etcd，事件分区达到其配额，但租约分区继续正常运行。节点保持健康。调度程序继续运行。

### 解锁什么取代etcd的共识机制

取消仲裁要求使我们能够做出长期以来想要的更改：在与 API 服务器相同的主机上运行 etcd。在传统布局中，每次读取和写入都会跨越不同机器之间的网络。每次旅行本身都很快，但如果每秒数千次，旅行时间就会增加。通过搭配，API 服务器通过环回接口与其本地 etcd 进行通信，并且 Pod 调度和控制器协调速度明显加快。对于作业控制器队列深度是绑定约束的工作负载，减少每个 API 调用的毫秒数意味着在队列开始增长之前每秒会处理更多作业。

![EKS Kubernetes 架构演变图](https://cdn.thenewstack.io/media/2026/06/0ebb7b12-image2-1024x779.png)这就是运营商代理获得回报的地方。当 etcd 在同一主机上运行时，每当更换控制平面主机时，etcd 成员就会来来去去，这种情况经常发生。只有当会员管理完全安全且自动时，这才有效，而这正是代理已经在做的事情。我们不必从头开始构建主机托管；我们将其构建在从第一天起就一直安全管理 etcd 会员资格的基础设施之上。

Collocation还给我们上了一个值得传承的教训：便捷的路径需要故障转移，以防万一发生故障。本地 etcd 是快速路径，但如果它受到损害，API 服务器会故障转移到另一个 etcd 成员，该成员正在主动为同一日志中的其他 API 服务器提供服务。当您针对常见情况进行优化时，请在优化不可用时谨慎进行设计。

### 修复整个堆栈的瓶颈

在极端规模下，您必须解决[整个 Kubernetes 堆栈中的瓶颈](https://thenewstack.io/ai-workloads-kubernetes-infrastructure-drift/)，其中大多数都不是传统意义上的错误。它们的设计选择在 Kubernetes 最初的目标规模上是正确的，只有当数量变大时才会崩溃。我们不是在内部解决这些问题，而是在上游修复它们，以便整个社区受益。

一个例子涉及监视缓存，这是一个内存层，它将状态更改从 etcd 分发到每个监视更新的控制器。当控制器启动时，它通过称为 WatchList 的机制请求当前状态的完整快照，并且现有实现在响应构建期间持有共享读锁。

对于数十万个对象，该工作运行的时间足够长，足以让需要独占访问的编写器挨饿，因此缓存的资源版本无法前进。一致的读取会看到陈旧的缓存并故障转移到 etcd，而响应构建会在锁定下通过数十万个分配进行搅动。我们认为这是手表缓存锁定模型的限制，并正在与社区合作重构底层数据结构和接口以消除争用。同样的形状也出现在其他地方。在 Horizo​​ntal Pod Autoscaler 中，保护缩放状态的单个互斥体在高 HPA 计数时成为序列化点，工作人员几乎所有时间都被阻塞，而不是做有用的工作。重新设计的数据存储 (PR #[139142](https://github.com/kubernetes/kubernetes/pull/139142)) 恢复了并行性，并将协调吞吐量提高了几个数量级。在调度程序中，我们发现了一个瓶颈（问题#[138426](https://github.com/kubernetes/kubernetes/issues/138426)）：每个调度周期都会通过扫描集群中的每个节点来重建一组正在使用的持久卷，即使对于根本不使用存储的 Pod 也是如此。该修复会延迟计算该信息，并且仅针对实际需要的 Pod 计算，从而大规模恢复吞吐量。

其中每一个都是从实际生产工作负载达到悬崖开始的，我们正在上游修复，以便每个 Kubernetes 用户都能得到改进。

### 从工程到保障：EKS 预置控制平面

上述工程使 EKS 控制平面更具弹性和性能。但客户有一个不同的问题：他们可以观察到控制平面今天保持不变，但他们无法像保留计算或 GPU 容量一样保留其容量。

规划千节点训练运行的团队可以提前几周保护实例，但没有用于协调它们的编排层的等效机制。 [EKS 预置控制平面](https://docs.aws.amazon.com/eks/latest/userguide/eks-provisioned-control-plane.html) 填补了这一空白。它将控制平面的性能公开为您明确调整的维度，并得到您期望其他基础设施的相同承诺的支持。

您可以选择映射到具体的、可衡量的功能的扩展层：API 请求并发性、Pod 调度率和集群数据库大小。尺码范围从 XL 到 8XL。在高端，Kubernetes 1.34 上的 8XL 提供 16,000 个并发 API 请求席位、每秒 400 个 Pod 的调度速率和 16 GB 的集群数据库存储，所有这些都由以一分钟间隔测量的 99.99% 可用性 SLA 提供支持。层级不是静态的。您可以在 GPU 训练运行或大型销售活动之前加强工作，在安静时期退一步，或者随着平台的成熟而永久成长。配置通过任何集群上的控制台、CLI、eksctl、CloudFormation 或 Terraform 进行，无需重新创建或停机。

对于 AI 工作负载，编排容量与 GPU 容量一起规划，在计算上线时可用。对于每分钟提交数百个作业的分析平台，控制平面在突发到来之前就已做好准备。对于需要跨阶段、生产和灾难恢复环境一致性的组织来说，同一层可以保证各处的性能特征一致。

### 将相同的基础带到边缘

![AWS Outposts 上的 Amazon EKS 架构图](https://cdn.thenewstack.io/media/2026/06/b641c1d2-image1-1024x753.png)

某些工作负载无法迁移到云，无论是由于数据主权要求、延迟限制还是与区域的不可靠连接。在这些断开连接的环境中运行 Kubernetes 带来了独特的挑战：etcd 必须在只有几台机器的硬件上保持持久性，集群必须在不到达云的情况下进行自我修复，并且可观测性必须能够在持续几天的网络分区中幸存下来。

通过实例存储 Outposts 上 EKS 本地集群的更新架构，我们将边缘集群引入与云中 EKS 集群相同的管理平面和软件堆栈。

控制平面位于 Outpost 上 EKS 管理的帐户中，而不是位于客户的帐户中，因此客户永远不会自己管理控制平面实例、etcd 备份或日志代理，并且他们不会意外破坏保持集群活动的事物。相同的机器映像、容器映像和操作员代理在两个位置运行，并通过配置选择特定于边缘的行为。

由于它们是相同的堆栈，因此新的 Kubernetes 和 EKS 平台版本与其云版本同步发布，并且 EKS 附加组件、Pod 身份和访问条目等功能的工作方式与它们在区域中的工作方式相同。最困难的部分是在只有几台机器的硬件上保持 etcd 的健康，这些机器可能会一次与云断开数天。我们通过扩展相同的代理来解决这个问题。它使数据的备用副本持续保持最新状态，并在机器发生故障时立即对其进行升级，因此集群可以在无需人工参与且无需连接到云的情况下自行修复。

[可观测性](https://thenewstack.io/taking-your-observability-strategy-to-the-next-level/) 也能在断开连接后继续存在：指标代理继续收集并写入本地磁盘，当空间不足时首先删除最不重要的数据，因此最重要的信号是最后一个。当链接返回时，缓冲的数据将以其原始时间戳刷新回来。

所有这一切之所以有效，是因为该系统从一开始就被设计为无需任何人登录即可运行。同样的设计使得在整个车队中安全地部署更改成为可能。

### 车队规模安全运营

这些更改中的每一项都被部署到由数十万个集群组成的正在运行的集群中。日志迁移和配置需要单独转换每个集群。每次迁移都遵循严格的顺序：验证前置条件、创建时间点快照、执行切换、验证后置条件。如果任何步骤失败，系统会自动回滚。

部署按单元、区域、区域进行，并通过自动监控比较更新和未更新集群之间的延迟、错误率和吞吐量。任何统计上显着的偏差都会触发自动停止。

使这一切成为可能的原因在于，EKS 的构建无需人工干预即可在单个集群级别运行。通过零操作员访问，该架构可防止 AWS 人员通过技术途径访问托管控制平面中的客户内容。一个设计为无需人工访问即可工作的系统从一开始就必须是可观察的、可恢复的和可自动化的，并且同样的规则可以实现极端规模的操作。

三个操作经验决定了我们如何开展这项工作。首先，健康的领导者与工作的领导者不同。控制平面的控制器以主动-被动配置运行，早期，我们将不健康的备用数据库视为集群操作已停止。他们没有；重要的是领导者是否存在。但更惨痛的教训是：领导者可以悄悄地停止进步，同时仍然续签租约并通过每一项健康检查。捕捉到这一点的信号正在观察控制器的工作队列深度。如果队列已满，而领导者看起来很健康，则系统会以任何活性探测器无法捕捉到的方式落后。

>

“领导者可以悄悄地停止进展，同时仍然续订租约并通过每一次健康检查。捕捉到这一点的信号是观察控制器的工作队列深度。”

其次，维护订单与维护本身同样重要。 etcd 碎片整理会阻塞，并且暂停时间会随着数据库大小的增加而增加。当它击中领导者时，每个写入都会停止。我们教代理在碎片整理之前将领导力转移到健康的节点，因此破坏性工作总是落在追随者身上，同时写入保持流动。

第三，活力并非就绪。进程在预热缓存时可能处于活动状态但尚未准备好，并且仅基于活动状态的路由会将请求发送到无法处理它们的实例。同样，在正常排空期间准备就绪的情况永远不应该触发重新启动。我们将两个信号严格分开：一个决定恢复；另一个决定恢复。另一个决定路由。

这些作品从外部是看不到的，这就是重点。最大的集群给我们带来了教训，使每个集群都变得更快。最危险的迁移产生了保护每次升级的安全机制。我们为 Kubernetes 可以处理流回项目每个用户的边缘的工作负载贡献了上游修复。

>

“这些作品从外面是看不到的，这就是重点。”

当您部署在 EKS 上并且您的 Pod 在几秒钟内启动时，即使在爆发期间，即使幕后出现问题，这种速度也并非偶然。它是多年大规模运营的积累结果，小问题很快就会变成大问题，并在问题发生之前设计系统来遏制它们。要探索本文中引用的架构，请参阅 [EKS 预置控制平面](https://docs.aws.amazon.com/eks/latest/userguide/eks-provisioned-control-plane.html) 和 [AWS Outposts 上的本地 Amazon EKS 集群](https://docs.aws.amazon.com/eks/latest/userguide/eks-outposts-local-cluster-overview.html)。
