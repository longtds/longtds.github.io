## 正文

在足够大的 GPU 集群上，总会出现一些问题。这只是生活的一个事实。标准修复方法是回滚到最后一个检查点并重新计算此后的所有内容，这既缓慢又昂贵。 [https://clockwork.io/](https://clockwork.io/) 希望让这个修复变得过时——并且它愿意对此做出保证。

Clockwork 防止训练运行失败的能力的基础是 TorchPass，[Clockwork 的容错产品](https://thenewstack.io/clockworks-fleetiq-aims-to-fix-ais-costly-network-bottleneck/)，该产品于 3 月份全面上市。当 GPU 或整个节点发生故障时，TorchPass 可以将训练作业的内存状态（包括模型权重、梯度和优化器状态）转移到健康的备用 GPU（或正在运行较低优先级作业的 GPU）上，并保持作业运行，通常只需几分钟即可恢复。

周三，该公司推出了 YOCO 担保，即“You Only Compute Once”的缩写。该公司保证支持的训练运行中 90% 的故障都将得到解决，不会丢失进度、不会回滚检查点，也不会重新计算。如果 Clockwork 在给定的合同年度内达不到要求，客户将获得 25% 的积分用于下次续约或扩展。

>

“人工智能团队需要完成他们的模型，而不是启动他们的节点。” ——Suresh Vasudevan，Clockwork 首席执行官。

Clockwork 首席执行官 [Suresh Vasudevan](https://thenewstack.io/keeping-gpus-ticking-like-clockwork/) 表示：“人工智能团队需要完成他们的模型，而不是启动节点。业界一直在衡量节点的正常运行时间，并将其称为可靠性。YOCO 让我们对唯一重要的事情负责——完成你的模型。”

### 实时迁移而不是回滚

Clockwork 的首席商务官兼联合创始人 Dan Cheng 向 The New Stack 介绍了当前的现状：“如果你定期检查点并且出现故障，你必须返回到之前的检查点，这可能是一小时、两小时前。你必须重新计算。”

当然，这些重新计算的工作都不是免费的。团队必须付出昂贵的 GPU 时间来重做他们已经完成的工作。TorchPass 通过在作业仍在运行时移动作业来解决这些回滚问题。 “在非常高的层面上，我喜欢将其视为 vMotion，但这是针对 GPU 的，所以有点像 gMotion，”Zheng 说，他指的是 VMware 在物理主机之间[转移正在运行的虚拟机](https://blogs.vmware.com/cloud-foundation/2019/07/09/the-vmotion-process-under-the-hood/) 的术语，无需停机。

但当然，这种替代必须来自某个地方。 “如果其中一个 GPU 发生故障，而您可以访问备用节点，它可以是备用节点，也可以是正在运行低优先级作业的节点，”Zheng 说。

团队可以将空闲节点留作专用备用节点，或者为了避免为空闲 GPU 付费，让 TorchPass 从优先级较低的工作中提取一个节点，关闭该作业，将该节点放入训练集群中，并重建 GPU 之间的连接，以便在下一步而不是最后一个检查点恢复运行。

TorchPass 也不必等待出现问题。当警告信号已经存在时，它可以在失败之前转移工作。 “如果温度超过某个阈值，你就知道 GPU 将在某个时候出现故障，”Zheng 说。 “当我仍然可以访问 GPU 内存时，为什么不采取一些措施呢？”

### 如何启用 Torchpass

TorchPass 有两种模式，主要区别在于每种模式必须移动多少状态以及因此恢复的速度。

快速选项是模型感知的，正如郑所描述的那样，只需要几行额外的代码。这使得 TorchPass 能够准确地知道要抓取什么，因此它可以移动更少的数据并在数十秒内恢复。

另一个是团队所描述的“模型透明”。要使用它，培训团队无需更改培训代码中的任何内容。 “我们能够拍摄系统级快照，”郑说。这是使用 TorchPass 的一种更简单的方法，但作为交换，它会移动更多数据并且需要几分钟才能恢复。

不过，突然的崩溃仍然是一个问题。毕竟，TorchPass 无法对已经死亡的节点进行快照。对于这种情况，Clockwork 表示，它会从作业已经运行的健康数据并行副本（首先是为了并行性而存在的副本）重建丢失的工作线程的状态。“火炬传递”有其局限性，郑承认这一点。 “如果整个网络瘫痪，你就无能为力，”他说。 “就像停电了一样——任何人都无能为力。”

### 失败的运行真正的代价是多少

大规模失败并不是罕见的事件。 Clockwork 的公告中引用了 Meta 的 FAIR 团队的研究，该研究表明 1,024 个 GPU 集群的平均故障时间为 7.9 小时；当 GPU 数量为 16,384 时，时间降至 1.8 小时。 Clockwork 表示，结果是 GPU 集群的有效运行率为理论性能的 30% 到 50%。但该团队认为，硬件并不是瓶颈，而是围绕它的可靠性模型假设故障比这些集群实际看到的要少得多。

Clockwork 估计，在典型的 2,048 GPU H200 部署中，故障驱动的重启每年会浪费超过 600 万美元的计算资源。

### “不适用于 Anthropic 和 Open

AI”

“这个解决方案不适合 Anthropic 和 OpenAI，因为他们拥有工程台实力，”郑说。 “这不是为了谷歌。这实际上是为了其他所有人。”他指的是人工智能原生初创公司、企业以及量化和生物技术公司，他们想要他所说的“前沿人工智能实验室级别的弹性”，而不需要自己构建它。

随着越来越多的工作从训练前转向训练后和强化学习，这个市场正在不断增长，这使得比一年前更多的团队面临着大型培训工作。 Clockwork 表示，其客户已经包括 Neoclouds Nebius、Nscale 和 WhiteFiber，以及 DCAI 以及 Uber 和 Wells Fargo 等企业。

不过，意识仍在迎头赶上。 “与我们交谈的很多人都已经习惯了检查点重启，他们甚至不知道还有其他可用的东西，”郑说。

### 测试结果

Clockwork 表示，最常见的选择就是维持现状。团队经常检查点，并在出现问题时重新计算。另一个是 [TorchFT](https://github.com/meta-pytorch/torchft)，这是 2024 年末发布的开源框架 Meta，它通过在 GPU 发生故障时删除整个副本组并在没有它的情况下继续执行作业直至更换为止来保持作业继续进行。 Clockwork 表示，即使没有出现任何故障，这种方法也会带来每一步的开销，并且该公司将 TorchPass 定位为更轻量级的选项，适用于在数百或数千个 GPU 上运行的更适度的作业，其中故障不会每隔几分钟发生一次。

SemiAnalysis 在 TorchPass 上运行了一些独立基准测试。

SemiAnalysis 的技术人员、ClusterMAX 基准测试的主要作者 Jordan Nanos 表示：“供应商制作幻灯片说明其产品有效，与将其写入合同之间存在很大差异。” “在我们的测试中，与作业完成时间的检查点重启相比，TorchPass 为在 64x H200 集群上运行的 GPT-OSS-120B 训练提供了最快、最高效的容错性能。TorchPass 在这项作业中的表现也优于 TorchFT（在 MFU 和令牌/秒/GPU 方面），同时匹配其恢复时间。YOCO 保证只是反映了我们在测试中看到的情况，并使其具有契约性。”

### 软件层的可靠性

“我喜欢回想早期的 Google 时光，比如 Google 文件系统，”郑说。 “您拥有商品旋转硬盘，但从开发人员的角度来看，您希望写入一次并确保数据持久化。您并不关心是否有人需要在数据中心交换磁盘。”

“我们也需要在软件层面拥有弹性层，”他说，“这样作为开发人员或人工智能研究人员，你就有更高的信心。你只需专注于培训，不必担心基础设施。”

### 可观测性

Clockwork 诞生于一个项目，该项目的使命与公司现在的使命完全不同：同步服务器时钟。但由于这还涉及对这些服务器之间的延迟进行非常精确的测量，因此该团队意识到，仅通过这样做就可以了解有关集群的很多信息。从那时起，该公司就在这方面进行了扩展，值得注意的是，可观测性仍然是 Clock work 工作的核心部分。郑说，TorchPass 和 Clockwork 的监控工具是互补的。 “它与 TorchPass 齐头并进，如果你发现问题，你就可以迁移，”他说。

该公司的许多新工作实际上都在进入可观测性领域。郑说，其车队监控工具现在可以将结构问题追溯到特定的链路或交换机，并且 Clockwork 正在与几家大型云运营商一起试点该功能。他说，尽早发现 GPU 性能下降或链路拥塞，TorchPass 可以在故障出现之前转移工作。
